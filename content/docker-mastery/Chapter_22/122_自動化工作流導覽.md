# 自動化工作流導覽

## 📝 課程概述

本單元開啟自動化（Automation/CI/CD）系列章節，介紹現代容器工作流中如何將程式碼從開發階段自動化地測試、打包並部署到生產環境。講師將重點放在 CI（Continuous Integration）部分，因為這是學員最常感到困惑的領域，後續會有超過 10 個實際範例可供參考。

---

## 核心觀念與實作解析

### 從 CI/CD 到自動化平台

傳統上我們稱之為 CI/CD（Continuous Integration / Continuous Deployment），但現代自動化範疇已經超越這兩個標籤，更適合稱為「自動化平台」。

#### 為什麼 Jenkins 正在衰退？

根據 2022 年開發者調查，Jenkins 的使用率正在下降，主要原因：

- **設定與維護複雜**：需要大量時間撰寫腳本
- **老舊的管理方式**：需要手動備份、維護
- **與程式碼距離太遠**：自動化與程式碼倉庫分離

#### 新趨勢：自動化靠近程式碼

現代最佳實踐是將自動化平台 **整合到程式碼存放處**：

- **GitHub Actions**（講師首選）
- GitLab CI
- Bitbucket Pipelines

這些工具讓自動化設定檔（YAML）直接存在於程式碼倉庫中，與程式碼版本同步管理。

### Pull Request 工作流

如果你來自 Ops 背景，可能對軟體開發的 **Pull Request（PR）工作流** 不熟悉。這是現代軟體開發的標準實踐。

#### PR 工作流的三個主要步驟

```
建立分支 → 建立 Pull Request → 合併到主分支
```

1. **建立分支（Branch）**：從 main/master 分支建立新分支進行修改
2. **建立 Pull Request**：在修改過程中（非完成後）就建立 PR，讓團隊成員可以查看、評論、建議修改
3. **合併（Merge）**：當所有檢查通過、獲得核准後，合併回主分支

> **關鍵概念**：PR 應該在「修改過程中」就建立，而非等到完成後。這讓 DevOps 團隊、安全團隊等可以即時參與審查。

#### 兩個主要自動化觸發時機

| 時機 | 觸發事件 | 典型自動化任務 |
| --- | --- | --- |
| **PR 建立/更新** | `pull_request` | Lint、Test、Build、Security Scan |
| **PR 合併** | `push` to main | Push Image、Deploy to Production |

---

### 自動化要做哪些事？

本系列將涵蓋以下核心自動化任務：

1. **Linting** — 程式碼風格檢查
2. **Testing** — 單元測試、整合測試
3. **Building** — 建置 Container Image
4. **Vulnerability Scanning** — 安全漏洞掃描
5. **Pushing to Registry** — 推送到 Image Registry
6. **Deploying** — 部署到目標環境

> 這些範例都是講師在實際工作中使用的設定，存放在 GitHub 倉庫中供學員直接取用。

## 💡 重點摘要

- **現代自動化趨勢是將 CI/CD 平台整合到程式碼倉庫（如 GitHub Actions），而非使用獨立的 Jenkins 伺服器。**
- **Pull Request 工作流是現代軟體開發標準，有兩個主要自動化觸發時機：PR 建立時與 PR 合併時。**
- **PR 應在修改過程中就建立，讓團隊成員可以即時參與審查與回饋。**
- **自動化範疇包括 Lint、Test、Build、Scan、Push、Deploy，本系列將提供超過 10 個實際範例。**
