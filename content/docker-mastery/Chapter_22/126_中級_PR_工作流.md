# 中級 PR 工作流

## 📝 課程概述

本單元介紹中級程度的自動化工作流，從基礎的線性流程升級為 **平行執行** 的多階段工作流。講師說明如何利用 Docker Multi-stage Build 管理測試環境，以及如何整合 CVE 掃描、SAST 掃描與 Kubernetes 煙霧測試。

---

## 核心觀念與實作解析

### 從線性到平行執行

基礎工作流是線性的：Build → Test → (repeat)。中級工作流則引入 **平行執行**：

```
PR Push ─┬→ Build Image (testing stage) ─┬→ Unit Test
         │                                ├→ Integration Test  
         │                                └→ CVE Scan
         │
         └→ Lint (parallel) ─────────────────────────────→ Gate → Merge → Deploy
```

#### 為什麼要平行執行？

- **節省時間**：Lint 與 Build 可以同時進行
- **更快回饋**：開發者能更快看到所有檢查結果
- **獨立失敗**：單一檢查失敗不會阻擋其他檢查執行

---

### Multi-stage Build 管理測試環境

講師推薦使用 Docker 的 **Multi-stage Build** 來管理測試框架版本。

#### 核心概念

```dockerfile
# Production stage (放在 Dockerfile 上方)
FROM node:18-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
CMD ["node", "server.js"]

# Testing stage (放在 Dockerfile 下方，作為額外層)
FROM production AS testing
RUN npm install --only=development
COPY test ./test
CMD ["npm", "test"]
```

#### Why 這樣設計？

- **版本控制集中**：測試框架版本由 Dockerfile 管理，而非依賴 CI Server 預裝
- **環境一致性**：測試 Image 與生產 Image 使用相同的基礎層
- **安全隔離**：測試相依套件不會進入生產 Image

> **重要觀念**：測試階段是在生產 Image 之上「疊加」額外層。最終生產 Image 只包含 production stage，不包含測試工具。

---

### 測試層級拆分

中級工作流將測試拆分為不同層級：

| 測試類型 | 特點 | 常用工具 |
| --- | --- | --- |
| **Unit Test** | 最快、最基礎 | Jest、pytest |
| **Integration Test** | 需要 DB 或外部服務 | Docker Compose |
| **CVE Scan** | 漏洞掃描 | Trivy、Snyk |
| **SAST Scan** | 程式碼安全分析 | CodeQL、Snyk Code |
| **Smoke Test** | 部署可行性驗證 | Kubernetes 臨時叢集 |

#### Integration Test 實作方式

使用 Docker Compose 在 CI 環境中啟動：

- 應用 Container
- 測試用的資料庫 Container
- 後端 Worker Container

這樣可以在隔離環境中進行完整的整合測試。

---

### CVE 掃描與 SAST 掃描

#### CVE Scan（Container Vulnerability Scan）

掃描 Image 中的已知漏洞：

- **Trivy** — 免費開源
- **Snyk** — 免費層可用

#### SAST Scan（Static Application Security Testing）

掃描原始碼中的安全問題：

- **CodeQL** — GitHub 原生支援
- **Snyk Code** — 與 Snyk 整合

> SAST 掃描比 CVE 掃描更深入，它能發現你「自己寫出的漏洞」，而不只是依賴套件的漏洞。

---

### Kubernetes 煙霧測試（Smoke Test）

這是進階但非常實用的驗證步驟：

1. 使用 CI 環境建立 **臨時 Kubernetes 叢集**（如 kind、k3s）
2. 部署剛建置的 Image
3. 驗證 Pod 能正常啟動並通過健康檢查
4. 測試完成後自動清理

> **目的**：在合併前驗證「這個 Image 真的能在 Kubernetes 上運行」。

---

### Gate（檢查關卡）

在 GitHub 中，Gate 就是 **Checks**：

- 所有平行執行的檢查都必須通過（綠色勾勾）
- 可能還需要人工審核（Code Review）
- 全部通過後才能點擊 Merge 按鈕

#### 合併後的動作

```
Merge → Build Production Image → Push to Registry → Deploy
```

生產 Image 是測試 Image 的子集，移除了測試相關層，只保留生產需要的內容。

---

## 💡 重點摘要

- **中級工作流引入平行執行，讓 Lint、Build、Test、Scan 同時進行，加快回饋速度。**
- **使用 Docker Multi-stage Build 管理測試環境，讓測試框架版本由 Dockerfile 控制，而非依賴 CI Server。**
- **CVE Scan 掃描 Image 漏洞，SAST Scan 掃描原始碼安全問題，兩者互補。**
- **煙霧測試在臨時 Kubernetes 叢集上驗證部署可行性，是合併前的重要關卡。**
