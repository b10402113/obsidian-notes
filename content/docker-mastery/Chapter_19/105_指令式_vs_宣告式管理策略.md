# 指令式 vs 宣告式管理策略

## 📝 課程概述

本單元探討 Kubernetes 的三種管理風格：指令式 CLI 指令、指令式 YAML 檔案、以及宣告式物件配置。理解這些風格的差異與適用場景，是建立正確 Kubernetes 工作流程的關鍵。

---

## 核心觀念與實作解析

### 指令式（Imperative）vs 宣告式（Declarative）

#### 類比：點一杯咖啡

| 方式 | 過程 |
| --- | --- |
| **指令式** | 自己煮咖啡：燒水、磨豆、沖泡...每一步都要自己決定 |
| **宣告式** | 向吧台點咖啡：說「我要一杯咖啡」，吧台決定如何製作 |

#### 軟體開發的對應

| 方式 | 特性 |
| --- | --- |
| **指令式** | 關注「如何做」（how）：每個步驟的順序與細節 |
| **宣告式** | 關注「要什麼」（what）：最終狀態，不關心如何達成 |

---

### Kubernetes 的三種管理風格

#### 1. 指令式 CLI 指令（Imperative Commands）

```bash
kubectl run my-app --image nginx
kubectl create deployment my-app --image nginx
kubectl expose deployment my-app --port 80
kubectl scale deployment my-app --replicas 3
```

**特性**：
- 直接在 CLI 輸入指令
- 每個動作有對應的動詞（create、scale、expose）
- 你必須知道資源的當前狀態

**適用場景**：
- 學習階段
- 個人專案
- 快速測試與除錯
- 臨時操作

**缺點**：
- 難以自動化
- 沒有變更歷史記錄
- 不適合團隊協作

---

#### 2. 指令式 YAML 檔案（Imperative Object Configuration）

```bash
kubectl create -f deployment.yaml
kubectl replace -f deployment.yaml
kubectl delete -f deployment.yaml
```

**特性**：
- 使用 YAML 檔案定義資源
- 每個動作仍需明確指定動詞（create、replace、delete）
- 有檔案記錄，可存入 git

**適用場景**：
- 從指令式過渡到宣告式的中間階段
- 需要明確區分 create/replace/delete 的場景
- 團隊開始建立配置檔案規範

**缺點**：
- 仍需手動判斷該用 create 還是 replace
- 不適合高度自動化

---

#### 3. 宣告式物件配置（Declarative Object Configuration）

```bash
kubectl apply -f deployment.yaml
kubectl apply -f ./configs/          # 整個目錄
kubectl diff -f deployment.yaml      # 預覽變更
```

**特性**：
- 只使用 `kubectl apply` 指令
- Kubernetes 自動判斷需要 create、update 還是 delete
- 與 Swarm stack deploy 概念相似

**適用場景**：
- 生產環境
- CI/CD 自動化
- GitOps 工作流程
- 團隊協作

**優點**：
- 同一指令處理所有操作
- 最適合自動化
- 天然支援 GitOps

---

### 三種風格的比較

| 面向 | 指令式 CLI | 指令式 YAML | 宣告式 YAML |
| --- | --- | --- | --- |
| **指令** | run、create、scale | create -f、replace -f | apply -f |
| **自動化** | ❌ 困難 | ⚠️ 中等 | ✅ 容易 |
| **歷史記錄** | ❌ 無 | ✅ 有（git） | ✅ 有（git） |
| **學習曲線** | 低 | 中 | 高 |
| **生產適用** | ❌ 不建議 | ⚠️ 可行 | ✅ 最佳 |

---

### 最重要的規則：不要混用風格

> **在生產環境中，千萬不要混用這三種風格。**

問題範例：
```bash
# 用指令式建立
kubectl create deployment my-app --image nginx

# 後來改用宣告式管理
kubectl apply -f my-app.yaml
# ❌ 可能導致衝突或非預期行為
```

**正確做法**：
- 從一開始就決定使用哪種風格
- 生產環境建議直接採用宣告式
- 如果已用指令式建立資源，需先清理再重新以宣告式建立

---

### 實務建議：四個步驟

1. **學習指令式 CLI**：理解資源類型與基本操作
2. **盡早學習 apply**：快速熟悉宣告式配置
3. **學習 YAML spec**：使用 `--dry-run -o yaml` 查看各資源的完整結構
4. **存入 git**：每次變更都 commit，建立歷史記錄

---

### GitOps 與宣告式的關係

宣告式配置是 **GitOps** 的基礎：

```
編輯 YAML → git commit → kubectl apply → 編輯 YAML → ...
```

好處：
- git log 就是完整的變更歷史
- 可以回溯任何時間點的配置
- 團隊可以 code review YAML 變更
- 自動化工具可以監控 git repo 觸發部署

---

## 💡 重點摘要

- **指令式關注「如何做」，宣告式關注「要什麼」—— Kubernetes 支援兩者。**
- **三種管理風格：指令式 CLI → 指令式 YAML → 宣告式 YAML，自動化程度遞增。**
- **生產環境最重要的規則：不要混用管理風格，從一開始就選定一種。**
- **宣告式配置使用 `kubectl apply`，是 GitOps 與自動化的基礎。**
- **將 YAML 存入 git 並每次變更都 commit，建立完整的變更歷史。**
