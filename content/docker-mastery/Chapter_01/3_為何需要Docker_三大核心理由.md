# 為何需要 Docker？三大核心理由

## 📝 課程概述

理解 Docker 的基本操作後，本單元回到更根本的問題：**為什麼我們需要 Container？** 老師從軟體發展的歷史脈絡出發，歸納出三大核心問題：隔離（Isolation）、環境差異（Environments）與速度（Speed），說明 Docker 如何解決這些長期困擾開發與維運團隊的痛點。

---

## 核心觀念與實作解析

### 問題一：隔離（Isolation）

#### 早期伺服器時代的困境

回到 1990 年代到 2000 年代初期，伺服器管理員面臨的典型場景：

```
┌─────────────────────────────────────────┐
│           大型實體伺服器                  │
│                                          │
│   App A  │  App B  │  App C  │  App D   │
│  Python  │  PHP    │  Ruby   │  Java    │
│  v2.7    │  v5.4   │  v1.9   │  v6      │
│                                          │
│       共用相同的檔案系統與相依套件          │
└─────────────────────────────────────────┘
```

**問題所在：**
- 一台伺服器上執行多個應用程式，但**沒有隔離機制**
- 不同 App 可能需要不同版本的 Python、Apache 或其他依賴
- 修改一個 App 的設定，可能導致另一個 App 故障
- 伺服器變得**脆弱且難以維護**

#### 虛擬化時代的過度擴張

VM（Virtual Machine）普及後，我們開始用 VM 來隔離應用程式：

```
┌──────────────────────────────────────────────────┐
│                   實體 Host                        │
│                                                   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │
│  │   VM 1  │ │   VM 2  │ │   VM 3  │ │   VM 4  │ │
│  │  App A  │ │  App B  │ │  App C  │ │  App D  │ │
│  │Guest OS │ │Guest OS │ │Guest OS │ │Guest OS │ │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ │
└──────────────────────────────────────────────────┘
```

**新的問題：**
- VM 數量暴增，單一管理員可能要管理數十甚至數百台 VM
- 每台 VM 都需要獨立的 Guest OS，資源浪費嚴重
- Cloud 讓建立 VM 變得更容易，但也讓 VM 數量失控

#### Container 如何解決？

Container 提供了**類似 VM 的隔離**，但不需要多個 OS：

```
┌──────────────────────────────────────────────────┐
│                   實體 Host                        │
│               單一 Linux Kernel                    │
│                                                   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐ │
│  │Cont. 1  │ │Cont. 2  │ │Cont. 3  │ │Cont. 4  │ │
│  │  App A  │ │  App B  │ │  App C  │ │  App D  │ │
│  │獨立 FS  │ │獨立 FS  │ │獨立 FS  │ │獨立 FS  │ │
│  │獨立 IP  │ │獨立 IP  │ │獨立 IP  │ │獨立 IP  │ │
│  └─────────┘ └─────────┘ └─────────┘ └─────────┘ │
└──────────────────────────────────────────────────┘
```

**Container 帶來的好處：**
- 每個 App 有獨立的檔案系統、IP、程序空間
- 可在同一台主機上執行**相同 App 的不同版本**
- 減少 OS 數量，降低管理負擔
- 不再需要 VM 作為隔離層

> Container 的隔離來自 Linux Namespace，讓每個 Container 看起來像獨立的系統。

### 問題二：環境差異（Environments）

#### "Works on My Machine" 現象

開發者常遇到的經典問題：應用程式在自己的電腦上正常運作，但部署到其他環境就出問題。Jeff Atwood（Stack Overflow 共同創辦人）曾戲稱這為 "Works on My Machine"。

#### Docker 稱之為「Matrix from Hell」

```
                    環境類型
              ┌─────────┬─────────┬─────────┬─────────┐
              │   Dev   │   QA    │  Stage  │   Prod  │
         ┌────┼─────────┼─────────┼─────────┼─────────┤
         │App1│ ConfigA │ ConfigB │ ConfigC │ ConfigD │
    應    ├────┼─────────┼─────────┼─────────┼─────────┤
    用    │App2│ ConfigE │ ConfigF │ ConfigG │ ConfigH │
    程    ├────┼─────────┼─────────┼─────────┼─────────┤
    式    │App3│ ConfigI │ ConfigJ │ ConfigK │ ConfigL │
         └────┴─────────┴─────────┴─────────┴─────────┘
              
              每個組合都需要不同的設定與依賴！
```

**問題本質：**
- 每個環境可能有不同的 OS、不同的依賴版本
- 開發者用 Mac，QA 用 Windows，生產環境用 Linux
- 文件必須說明每種環境的安裝方式

#### Container Image 作為「契約」

Container Image 與 Container 本身成為了執行環境與應用程式之間的**抽象契約**。

**貨櫃的比喻：**
- 貨櫃運輸革命讓全球物流標準化
- 運送者不需要知道貨櫃裡裝什麼
- 託運者不需要知道貨櫃如何被運送
- **Container Image 就是軟體世界的貨櫃**

**OCI 標準的保證：**

> 無論 Container 在哪裡執行，它都會以相同的方式運作，使用完全相同的依賴項目。環境間的差異只剩下資料庫連線設定、URL、Port 等設定值。

### 問題三：速度（Speed）

這裡的「速度」不是指 CPU 效能，而是**業務執行的速度**——從想法到交付給客戶的時間。

#### IT 基礎架構的演進史

老師用一張圖展示數十年來的演進：

```
Mainframe → PC（分散式） → VM（虛擬化） → Cloud（雲端） → Container
   │           │            │              │              │
   └───────────┴────────────┴──────────────┴──────────────┘
                          追求更快的變更速度
```

| 時期 | 驅動因素 |
|------|---------|
| **Mainframe → PC** | 不同部門需要以不同速度變更，不再等待 Mainframe 統一更新 |
| **Bare Metal → VM** | 不想等待硬體採購與資料中心部署，提高資源利用率 |
| **Data Center → Cloud** | 需要更快速取得運算資源，從數週縮短到數分鐘 |
| **VM → Container** | VM 數量爆炸、OS 過多，需要更輕量的隔離方式 |

> Serverless 本質上也是執行 Container，只是由雲供應商管理。

#### Container 加速軟體生命週期

Container 讓整個軟體生命週期加速：

- **開發更快**：本機環境與生產環境一致
- **建置更快**：Image 建立標準化、可重複
- **測試更快**：測試環境與生產環境使用相同 Image
- **部署更快**：隨時可以從 Registry 取得 Image 執行

---

## 💡 重點摘要

- **Container 解決隔離問題：在同一個 OS 上提供 VM 等級的隔離，可執行相同 App 的不同版本且互不干擾。**
- **Container 解決環境差異問題：Image 作為應用程式與環境之間的契約，確保「Build once, Run anywhere」。**
- **Container 解決速度問題：加速開發、建置、測試、部署的整個軟體生命週期。**
- **「Matrix from Hell」是指應用程式與環境的組合爆炸，Container Image 標準化了解決方案。**
- **IT 基礎架構的演進史就是不斷追求更快變更速度的過程，Container 是目前的最新階段。**
