# 平行化作業（Bonus）

## 📝 課程概述

本單元是加碼內容，示範如何將前述所有測試步驟整合成一個完整的平行化 Workflow。透過 GitHub Actions 的 Job 依賴機制，實現「建置一次，多項測試平行執行，全部通過後才推送」的進階工作流。

---

## 核心觀念與實作解析

### 從單一 Job 到多 Job 平行

前面所有範例都是單一 Job 內的步驟。要實現平行執行，需要將工作拆分成多個 Job。

#### Job 依賴關係

```yaml
jobs:
  build:
    # 建置測試用 Image

  test-unit:
    needs: build
    # 單元測試

  test-integration:
    needs: build
    # 整合測試

  test-cve:
    needs: build
    # CVE 掃描

  push:
    needs: [test-unit, test-integration, test-cve]
    # 推送到生產 Registry
```

#### 執行順序

```
          ┌→ test-unit ─────┐
          │                  │
build ───┼→ test-integration┼──→ push
          │                  │
          └→ test-cve ───────┘
```

---

### Build Job：建置並推送測試 Image

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    outputs:
      image-tag: ${{ steps.meta.outputs.tags }}
    steps:
      - name: Build and push to GHCR
        uses: docker/build-push-action@v4
        with:
          push: true
          tags: ghcr.io/${{ github.repository }}:${{ github.run_id }}
```

#### Job Outputs

透過 `outputs` 將 Image 標籤傳遞給下游 Job：

```yaml
outputs:
  image-tag: ghcr.io/${{ github.repository }}:${{ github.run_id }}
```

---

### Test Jobs：平行執行的測試

每個測試 Job 都使用 `needs: build` 確保 Image 已就緒：

```yaml
test-unit:
  needs: build
  runs-on: ubuntu-latest
  steps:
    - name: Run unit tests
      run: docker run ${{ needs.build.outputs.image-tag }} npm test

test-cve:
  needs: build
  runs-on: ubuntu-latest
  steps:
    - name: Run Trivy
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: ${{ needs.build.outputs.image-tag }}
```

> **平行效益**：假設每個測試需要 2 分鐘，三個測試平行執行只需 2 分鐘，而非序列執行的 6 分鐘。

---

### Push Job：最終關卡

```yaml
push:
  needs: [test-unit, test-integration, test-cve]
  runs-on: ubuntu-latest
  steps:
    - name: Push to Docker Hub
      uses: docker/build-push-action@v4
      with:
        push: true
        tags: user/image:latest
```

只有當 `needs` 列出的所有 Job 都成功時，`push` Job 才會執行。任何一個測試失敗，整個 Workflow 都會中斷，不會推送有問題的 Image。

---

### GitHub Checks 視覺化

在 GitHub PR 頁面中，平行 Job 會以圖形化方式顯示：

```
✓ build
├─ ✓ test-unit
├─ ✓ test-integration
└─ ✓ test-cve
   ✓ push
```

這讓審查者一目了然地看到所有檢查狀態。

---

## 💡 重點摘要

- **將工作拆分成多個 Job 並透過 `needs` 建立依賴關係，實現平行執行。**
- **Build Job 負責建置測試 Image 並透過 outputs 傳遞標籤給下游 Job。**
- **所有測試 Job 平行執行，大幅縮短整體等待時間。**
- **Push Job 依賴所有測試 Job，確保只有通過全部檢查的 Image 才會推送。**
