# 單元測試與整合測試

## 📝 課程概述

本單元介紹如何在 GitHub Actions 中執行單元測試（Unit Test）與整合測試（Integration Test）。講師說明如何利用 Docker Compose 在 CI 環境中啟動相依服務（如資料庫），以及如何正確處理測試的 Exit Code。

---

## 核心觀念與實作解析

### 單元測試

單元測試是最簡單的測試形式，通常只需要一兩個指令即可執行。

#### 實作方式

```yaml
- name: Build test image
  uses: docker/build-push-action@v4
  with:
    context: .
    load: true
    tags: user/image:test

- name: Run unit tests
  run: docker run --rm user/image:test npm test
```

#### 關鍵參數

- `load: true`：將建置好的 Image 載入到本地 Docker，而非推送到 Registry
- 這樣後續的 `docker run` 指令就能使用這個 Image

> **流程**：建置測試用 Image → 在 Container 中執行測試指令 → 測試成功才進入下一步（建置生產 Image）

---

### 整合測試

整合測試需要外部相依服務（如資料庫、Worker、API）。

#### 實作方式：使用 Docker Compose

```yaml
- name: Run integration tests
  run: |
    export IMAGE_ID=user/image:${{ github.run_id }}
    docker-compose -f docker-compose.test.yml up --abort-on-container-exit --exit-code-from sut
```

#### docker-compose.test.yml 範例

```yaml
services:
  app:
    image: ${IMAGE_ID}
    depends_on:
      postgres:
        condition: service_healthy

  postgres:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5

  sut:
    image: ${IMAGE_ID}
    command: curl -f http://app:3000/health
    depends_on:
      - app
```

#### 關鍵概念

| 參數 | 說明 |
| --- | --- |
| `condition: service_healthy` | 等待 Postgres 健康檢查通過才啟動 app |
| `--abort-on-container-exit` | 當任一 Container 退出時，停止所有服務 |
| `--exit-code-from sut` | 以 `sut` 服務的 Exit Code 作為整體結果 |

> **Why 用 sut？** Docker Compose 同時啟動多個 Container，需要明確指定「以哪個服務的 Exit Code 為準」。`sut`（System Under Test）是我們真正想測試的指令。

---

### 為什麼選擇 Docker Compose？

雖然 GitHub Actions 有內建的 **Services** 功能，但講師偏好 Docker Compose：

- **跨平台相容**：同樣的 YAML 可以在 GitHub Actions、GitLab CI、Jenkins 中使用
- **本地可重現**：開發者可以在本地執行相同的測試環境
- **更彈性**：支援複雜的服務依賴關係與健康檢查

### Kubernetes 煙霧測試（進階）

這是講師最喜歡的測試方式——在每次 PR 都做一次「部署驗證」。

#### 實作步驟

1. **建立臨時 Kubernetes 叢集**（使用 k3d，約 30 秒完成）

```yaml
- name: Create k3d cluster
  uses: nolar/setup-k3d-k3s@v1
```

2. **推送 Image 到 GitHub Container Registry（測試用）**

```yaml
- name: Push to GHCR
  uses: docker/build-push-action@v4
  with:
    context: .
    push: true
    tags: ghcr.io/${{ github.repository }}:${{ github.run_id }}
```

3. **部署並測試**

```yaml
- name: Deploy and test
  run: |
    kubectl create secret docker-registry regcred ...
    envsubst < deployment.yaml | kubectl apply -f -
    kubectl rollout status deployment/myapp
    kubectl exec deployment/myapp -- curl -f localhost:3000/health
```

> **envsubst 工具**：可以將環境變數注入到 YAML 檔案中，例如將 `${IMAGE_TAG}` 替換成實際的 Image 標籤。

---

### 雙 Registry 策略

範例中使用了兩個 Registry：

| Registry | 用途 |
| --- | --- |
| **GitHub Container Registry (ghcr.io)** | 測試階段的臨時 Image |
| **Docker Hub** | 生產階段的最終 Image |

> **Why 分開？** 測試 Image 只需要暫時存在，使用 GHCR 避免污染 Docker Hub；合併後才推送正式版本到 Docker Hub。

---

## 💡 重點摘要

- **單元測試透過 `docker run` 在 Container 內執行測試指令，使用 `load: true` 將 Image 載入本地 Docker。**
- **整合測試使用 Docker Compose 啟動相依服務，以 `--exit-code-from` 指定 Exit Code 來源。**
- **Kubernetes 煙霧測試在臨時叢集上驗證部署可行性，使用 k3d 可在 30 秒內建立叢集。**
- **雙 Registry 策略：測試用 GHCR，生產用 Docker Hub，避免污染正式 Registry。**
