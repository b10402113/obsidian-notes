# 部署抽象化工具

## 📝 課程概述

本單元探討 Kubernetes 生態中「如何部署 YAML」的各種工具與方法論。隨著團隊規模擴大、權限管理變得複雜，單純的 `kubectl apply` 已不足以應付需求。講師介紹了 Helm、Kustomize、Compose on Kubernetes 等超過 60 種部署工具，幫助學員建立選擇的判斷依據。

---

## 核心觀念與實作解析

### 為什麼需要部署工具？

當你從單人開發進展到多團隊協作時，會面臨以下挑戰：

- **權限管理**：不同團隊需要不同層級的存取權限
- **審批流程**：部署需要經過核准
- **範本化需求**：類似的應用需要一致的部署範本
- **版本控制**：YAML 本身也需要版本管理

Kubernetes 本身是非意見式的（unopinionated），這意味著它不會告訴你「應該怎麼做」，於是各種工具應運而生，提供「最佳實踐」與「意見」。

---

### Helm：Kubernetes 的套件管理器

**Helm** 是 Kubernetes 部署工具中的事實標準（de facto standard），被稱為「Kubernetes 的 apt/yum」。

#### 核心概念

- **Chart**：一組範本化的 YAML 檔案
- **Release**：Chart 的一次安裝實例
- **Repository**：存放 Chart 的倉庫

#### 運作方式

Helm 使用 **Go Template** 語法，讓你可以：

```yaml
# 範本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Values.appName }}
spec:
  replicas: {{ .Values.replicaCount }}
```

透過 `values.yaml` 注入變數，Helm 會生成最終的 Kubernetes YAML 並部署。

#### Helm 3.0 的改進

- 移除了 Tiller（舊版的伺服器端元件），大幅簡化安全模型
- 更輕量、更安全

> **講師觀察**：幾乎所有 Kubernetes 發行版都支援或內建 Helm。在大規模環境中，你幾乎無法避免使用它。

---

### Kustomize：Kubernetes 原生的配置管理

**Kustomize** 是 Kubernetes 內建的配置管理工具，從 `kubectl` 直接支援。

#### 與 Helm 的差異

| 特性 | Helm | Kustomize |
| --- | --- | --- |
| 範本引擎 | Go Template | 無範本，使用「覆蓋」機制 |
| 學習曲線 | 較陡峭 | 較平緩 |
| 功能豐富度 | 較高 | 較基礎 |
| 內建支援 | 需額外安裝 | kubectl 原生支援 |

#### 運作方式

Kustomize 不使用範本，而是讓你定義 **base YAML** 和 **overlays（覆蓋層）**：

```
├── base/
│   └── deployment.yaml
└── overlays/
    ├── dev/
    │   └── kustomization.yaml
    └── prod/
        └── kustomization.yaml
```

這類似於 Docker Compose 的 `docker-compose.override.yml` 概念。

---

### Compose on Kubernetes：Docker 的解決方案

Docker 提供了 **Compose on Kubernetes**，讓你可以直接使用 `docker-compose.yml` 部署到 Kubernetes：

```bash
docker stack deploy -c docker-compose.yml myapp --orchestrator kubernetes
```

#### 優點

- **統一工作流**：開發與生產環境使用同一份 YAML
- **降低學習門檻**：開發者不需要學習 Kubernetes YAML 語法
- **80/20 法則**：支援 80% 開發者需要的 20% Kubernetes 功能

#### 限制

- 無法使用 Kubernetes 的所有進階功能
- 適合簡單應用，複雜場景可能需要混合使用

> **講師建議**：如果你考慮使用 Docker Enterprise 或偏好 Docker 生態，這是非常值得嘗試的方案。

---

### CNAB 與 Docker App

**CNAB（Cloud Native Application Bundle）** 是 Docker 與 Microsoft 在 2018 年共同提出的標準：

- 將 YAML 打包成 **Image**
- 可以像容器一樣版本化、分發
- 使用 `docker app` 子命令操作

這讓「基礎設施即程式碼」可以像容器一樣被管理。

---

### 如何選擇？

| 場景 | 建議工具 |
| --- | --- |
| 小型團隊、初學 | 原生 `kubectl apply` 或 Kustomize |
| 部署第三方軟體（如 Prometheus、ELK） | Helm（有現成 Chart） |
| 想統一開發與生產 YAML | Compose on Kubernetes |
| 大型企業、多叢集環境 | Helm + Kustomize 混合使用 |

> **重要提醒**：這些工具都是 **可選的**。你可以只用原生 Kubernetes YAML，它們只是讓團隊協作更便利的第三方工具。

---

## 💡 重點摘要

- **Kubernetes 部署工具有超過 60 種，Helm 是其中的事實標準，幾乎無法在大規模環境中避開。**
- **Kustomize 是 Kubernetes 原生的配置管理工具，使用「覆蓋」機制而非範本，學習曲線較平緩。**
- **Compose on Kubernetes 讓開發者可以用熟悉的 docker-compose.yml 部署到 Kubernetes，適合偏好 Docker 生態的團隊。**
- **這些工具都是可選的，選擇時應考慮團隊規模、既有技術棧與具體需求。**
