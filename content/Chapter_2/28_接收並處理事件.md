# 接收並處理事件

## 📝 課程概述

本單元為 `posts` service、`comments` service 各自加入 `POST /events` 端點，使其能接收來自 Event Bus 廣播的事件。收到事件時，Service 只需要 `console.log` 記錄下來（目前對 `PostCreated` 和 `CommentCreated` 都不做任何實際處理），為日後 Query Service 真正處理這些事件做準備。

## 核心觀念與實作解析

### Posts Service 的事件接收端點

```js
// posts/index.js
app.post('/events', (req, res) => {
  const { type, data } = req.body;
  console.log('Event received:', type);
  res.send({});
});
```

### Comments Service 的事件接收端點

```js
// comments/index.js
app.post('/events', (req, res) => {
  const { type, data } = req.body;
  console.log('Event received:', type);
  res.send({});
});
```

### 為什麼 `posts` 和 `comments` 只做 `console.log`？

在目前的應用場景中，`posts` 和 `comments` 這兩個 Service **不需要**知道其他 Service 的狀態變化：
- `posts` 是文章的管理者，它不關心留言。
- `comments` 是留言的管理者，它不關心文章（除了 `postId` 作為關聯鍵）。

加入 `POST /events` 端點只是為了「讓 Event Bus 的廣播不會 404」——它們在這階段是**被動接收者**，日後若業務需求變更，可能就會用到這些事件（例如刪除某篇文章時，需要通知 Comments Service 刪除該文所有留言）。

### `res.send({})` 的必要性

Express 的 request handler 必須**回傳一個 HTTP 回應**。如果 handler 執行完但沒有呼叫 `res`，HTTP 連線會一直懸著（hang），最終 client（Event Bus）會得到超時錯誤。即使只是空物件 `{}`，也必須 `res.send()` 或 `res.json()`。

### 驗證完整通訊流程

完成後，Terminal 中會看到：
```
posts:     Event received: PostCreated
comments:  Event received: PostCreated
comments:  Event received: CommentCreated
```

這表示 Event Bus 成功廣播，Posts 和 Comments Service 都收到了事件——完整的事件驅動通訊鏈路已建立。

## 💡 重點摘要

- 所有 Service 都需要 **`POST /events` 端點**——即使目前不需要處理任何事件，也必須存在，否則 Event Bus 廣播時會收到 404。
- **`res.send({})`**（或 `res.json()`）是必要的，否則 HTTP 請求會超時。
- 被動接收事件不代表「沒事做」——日後業務邏輯擴展時，這些端點就會派上用場。
- 完整通訊鏈路的驗證方式是觀察各 Service Terminal 的 `console.log` 輸出。
