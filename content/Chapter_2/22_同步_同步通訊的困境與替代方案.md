# 同步通訊的困境與替代方案

## 📝 課程概述

本單元診斷了 React 前端目前面臨的 **N+1 效能問題**，並提出兩種解決方向：**同步（Synchronous）通訊**與**非同步（Asynchronous）通訊**。老師預告同步方案雖然直覺，但會帶來多個微服務架構中的典型缺陷——這為下一堂課引入 Event Bus 與 Query Service 做了鋪墊。

## 核心觀念與實作解析

### 同步通訊方案（Synchronous Communication）

在這個方案中，`GET /posts` 端點不只是回傳文章列表，還會**主動呼叫** Comments Service 取得留言，然後把留言「內嵌」進文章物件中再回傳：

```
Browser → GET /posts → Posts Service
                        ↓（同步呼叫）
                        Comments Service → 回傳留言
                        ↓
                  組裝 → 回傳 { posts: [{ ..., comments: [...] }] }
```

**缺點（為什麼不採用）：**

1. **服務間的直接依賴**：Posts Service 現在依賴 Comments Service。如果 Comments Service 掛了，`GET /posts` 也跟著失敗——使用者連文章列表都看不到。
2. **串聯延遲**：一次請求變成兩次網路跳（Browser→Posts→Comments）。Comments Service 反應慢，Posts Service 的回應也跟著慢。
3. **擴展性惡化**：如果日後再加入第三、第四個 Service（例如 Likes、Tags），請求鏈會變成一棵樹，延遲與失敗率都會指數成長。
4. **違反微服務的「獨立性」原則**：每個 Service 應該是自治的（autonomous），不應直接依賴另一個 Service 的存在與健康狀態。

### 非同步通訊方案（Asynchronous Communication）

引進一個**第三方的查詢服務（Query Service）**，它：
- 不處理文章或留言的建立（CRUD）
- 只負責「彙整」——監聽所有 Service 發出的事件（Event），維護一份自己的資料副本
- 提供一個端點讓前端只需要一次請求，就能拿到完整資料

```
Post 建立 → Event Bus 廣播 → Query Service 更新內部狀態
                                      ↓
Browser → GET /posts → Query Service → 一次回傳所有文章 + 留言
```

這正是下一個單元要實作的方案。

## 💡 重點摘要

- **同步通訊**直覺但會在微服務中造成：服務依賴、串聯失敗、擴展性惡化等問題。
- **非同步通訊**透過第三方 Query Service 做資料彙整，實現真正意義上的「去耦合」。
- 查詢（Query）職責與建立（Command）職責分開，是 CQRS（Command Query Responsibility Segregation）概念的簡化體現。
- 採用非同步方案時，前端只向 Query Service 發送一個請求，解決了 N+1 問題。
