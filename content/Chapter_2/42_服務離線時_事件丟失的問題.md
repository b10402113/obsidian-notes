# 服務離線時事件丟失的問題

## 📝 課程概述

本單元揭示了 Event-Driven 架構的一個根本性問題：**當某個 Service 離線時，它會錯過離線期間發出的所有事件**。Moderation Service 在離線時建立的留言會永遠停留在 `pending` 狀態——這稱為「事件丟失（Missing Events）」問題，是所有事件驅動系統必須面對的課題。

## 核心觀念與實作解析

### 問題的具體表現

假設 Moderation Service 在時間區間 [t2, t3] 離線：

```
t1: Event 1 (正常發送) → Moderation ✅ 收到
t2: Event 2 (Moderation 離線) → ❌ 丟失
t3: Event 3 (Moderation 離線) → ❌ 丟失
t4: Moderation 上線 → Event 4 ✅ 收到
```

結果：Event 2 和 Event 3 代表的留言永遠不會被 Moderation Service 處理，永遠停留在 `pending` 狀態。

### 更嚴峻的場景：Query Service 延後上線

如果 Query Service 在應用程式已經運行了「很長一段時間」後才啟動，它將**完全錯過所有歷史事件**——所有歷史文章和留言都無法從 Query Service 取回，前端頁面一片空白。

### 為什麼這是「根本性」問題？

在我們目前的手刻 Event Bus 中，事件**沒有持久化**——Event Bus 把事件廣播出去後，如果目標 Service 不在線，該事件就此消失。這在 Monolith 架構中幾乎不可能發生（因為所有模組都在同一個 process 中），但在分散式系統中是一個**內在特性**，必須靠外部機制來彌補。

### 三種修復方向預覽

老師預告了即將介紹的三種解決方案（細節在下一個單元）：

| 方案 | 思路 | 缺點 |
|------|------|------|
| 方案一 | Query Service 上線時向 Posts 和 Comments Service 同步發送 HTTP 請求取回所有資料 | 需要在 Posts/Comments 中新增專門的 bulk endpoint |
| 方案二 | 直接讓 Query Service 連接 Posts 和 Comments 的資料庫 | Service 間耦合到 DB 具體實作（MySQL vs MongoDB） |
| 方案三 | Event Bus 持久化所有事件，Service 上線時向 Event Bus 拉取歷史事件 | Event Bus 需要有資料庫（成本考量但其實不貴） |

> 老師預告：我們將採用**方案三**——這是業界最常見的實作方式。

## 💡 重點摘要

- **事件丟失**是 Event-Driven 架構的內在風險，無法避免，只能靠外部持久化機制來確保 Service 離線期間的事件不被遺忘。
- **Query Service 延後上線**是最常見的實務場景——新功能上線時，歷史資料不能靠「等新事件」取得，必須能夠補歷史事件。
- 修復這個問題的核心思路是：**Event Bus 必須持久化所有事件**，讓任何 Service 在任何時候都能補回錯過的事件。
- 「資料丟失」的代價（磁碟空間）其實遠低於想像——100 億個事件每個月可能只需要 $15。
