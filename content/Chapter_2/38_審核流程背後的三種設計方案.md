# 審核流程背後的三種設計方案

## 📝 課程概述

本單元深入探討了引入 Moderation Service 時的三種可能的架構設計，並比較了它們的優缺點。老師最終選擇了**方案三**——讓 Comments Service 接收 `CommentModerated` 事件並發出通用的 `CommentUpdated` 事件，確保 Query Service 不需要理解任何「業務特定」的更新邏輯。

## 核心觀念與實作解析

### 留言的 `status` 欄位

無論採用哪種方案，所有留言都需要多一個 `status` 欄位來表示審核狀態。在 Comments Service 建立留言時就預設為 `pending`：

```js
// comments/index.js — POST /posts/:id/comments handler
comments.push({
  id: commentId,
  content,
  status: 'pending'  // ← 新增欄位
});
```

`CommentCreated` 事件也需要包含 `status`：

```js
{
  type: 'CommentCreated',
  data: { id, content, postId, status: 'pending' }
}
```

### 三種方案的比較

**方案一：Query Service 直接接收 `CommentModerated`**

```
Comments → CommentCreated → Moderation → CommentModerated → Query Service
```

缺點：當留言的「更新方式」越來越多（編輯、刪除、舉報、匿名化等），Query Service 必須知道所有這些事件的處理邏輯，造成**業務邏輯洩漏**到 Query Service 中。

**方案二：Query Service 接收 `CommentCreated` 並預設 `pending`**

```
Comments → CommentCreated → Query Service（直接顯示 pending）
                         → Moderation → CommentModerated → Query Service（更新狀態）
```

好處：使用者提交留言後馬上能看到（pending 狀態），不必等 Moderation 完成。
缺點：與方案一同樣存在「Query Service 需要理解所有更新事件」的問題。

**方案三（採用）：Comments Service 接收 `CommentModerated`，發出通用的 `CommentUpdated`**

```
Comments → CommentCreated → Query Service（馬上顯示 pending）
                         → Moderation → CommentModerated → Comments Service
                                                      → CommentUpdated → Query Service
```

老師推薦這個方案的原因：Query Service 只知道「有東西更新了，請用新資料覆蓋我的舊資料」——它**不需要知道也不需要理解**是誰發起的更新、是什麼類型的更新。這是真正的**關注點分離**。

### `CommentUpdated` 事件的設計

```js
{
  type: 'CommentUpdated',
  data: {
    id: 'comment-id',
    postId: 'post-id',
    content: 'This is my comment',
    status: 'approved'   // 或 'rejected'
  }
}
```

Query Service 收到後，直接用 `data` 中的所有欄位覆蓋該留言的舊資料——**不需要知道覆蓋的是哪個欄位發生了變化**。

### Comments Service 處理 `CommentModerated` 並發出 `CommentUpdated`

```js
if (type === 'CommentModerated') {
  const { id, postId, status } = data;
  // 更新本地 comment 的 status
  const comments = commentsByPostId[postId];
  const comment = comments.find(c => c.id === id);
  comment.status = status;

  // 發出通用的 CommentUpdated 事件
  await axios.post('http://localhost:4005/events', {
    type: 'CommentUpdated',
    data: { id, postId, content: comment.content, status }
  });
}
```

## 💡 重點摘要

- **方案三的核心思想**：Domain-specific 更新（`CommentModerated`）由 Comments Service 處理，對外只暴露通用的 `CommentUpdated`——Query Service 只需要「覆蓋資料」，不需要知道背後的業務邏輯。
- **`status: 'pending'`** 讓使用者在 Moderation 還沒完成時就看到自己的留言處於「等待審核」狀態，提供良好的 UX。
- 隨著系統功能增加，**服務邊界**（哪些邏輯該由哪個 Service 負責）會越來越重要——這是微服務架構中最困難的設計決策之一。
- `CommentUpdated` 是「通用事件」的典型例子：用一個事件類型涵蓋所有可能的留言屬性更新，讓所有 Consumer 都能通用處理。
