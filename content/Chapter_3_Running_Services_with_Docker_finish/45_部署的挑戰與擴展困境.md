# 部署的挑戰與擴展困境

## 📝 課程概述

本單元從一個簡單的假設情境切入：若將本機上運行的多個微服務直接部署到一台虛擬機器，會遇到什麼樣的擴展瓶頸？我們將看到**水平擴展**（scale out）所帶來的三大致命問題——IP / Port 耦合、程式碼耦合，以及動態擴縮容的複雜度。這正是為什麼我們需要引入 Kubernetes 的核心驅動力。

---

## 核心觀念與實作解析

### 直接部署到虛擬機器的直覺做法

將所有微服務（Posts、Comments、Query、Moderation、Event Bus）複製到一台 VM 上，各自綁定固定 Port（如 4000、4001…），透過 `localhost` 直接互相通訊。**這種做法在小規模、靜態服務數量下是可行的**，但隨即會面臨三個現實挑戰。

### 挑戰一：水平擴展時的 Port 耦合

當 Comments 服務承載量暴增，需要啟動第二、第三個副本時，VM 上會多出 4006、4007 等新 Port。然而 **Event Bus 的程式碼必須事先知道每個副本的 IP 與 Port**，否則無法將事件廣播出去。

> 這裡的問題在於：我們被迫將「基礎設施的動態狀態」寫進「應用程式程式碼」。

只要多開一台副本，就要去 Event Bus 的程式碼裡多寫兩行 URL；多關閉一台，就要去刪。這種耦合在實務上幾乎無法維護。

### 挑戰二：跨機器的網路拓撲耦合

若 Comments 的多個副本分布在兩台不同的 VM 上，Event Bus 除了要知道 Port，還需要知道「目標機器的 IP 位址」。同一份 Event Bus 程式碼被迫維護一個靜態的 IP + Port 清單。**一旦 VM 的 IP 變動（例如雲端自動遷移），整個服務就會斷線**。

### 挑戰三：動態擴縮容的時間耦合

想像一個只在白天流量高峰需要擴展的場景：凌晨 1 點沒有人訪問，可以關閉多餘副本以節省費用。但「何時該發請求到哪些副本」的邏輯**不該寫在 Event Bus 裡**，否則會變成：

```javascript
if (isPeakHour()) {
  // 發送給額外的副本
} else {
  // 只發給主副本
}
```

這段邏輯完全屬於「基礎設施」的範疇，放進應用程式是架構上的災難。

### 根本問題的本質

以上三個挑戰都指向同一個核心問題：**我們要求 Event Bus 用「直接指定」的方式維護所有服務的網路位址**。這是一種 `imperative`（命令式）的溝通模式——Event Bus 必須「知道」每一個服務在哪裡，才能把事件送過去。

當服務數量動態變化、分布位置不固定時，這種模式的維護成本趨近無窮大。

---

## 💡 重點摘要

- **直接部署到 VM 在靜態場景可行，但擴展時 Event Bus 的 URL 清單會與應用程式程式碼形成硬耦合。**
- 每新增一個服務副本，就要在 Event Bus 裡新增一行 URL——這是不可擴展的維護方式。
- 跨機器的服務通訊若靠 IP + Port 硬編碼，雲端 VM 遷移後服務立刻斷線。
- 動態擴縮容的「時段邏輯」屬於基礎設施職責，不應寫進應用程式。
- **微服務架構需要一個「第三人」來自動處理服務發現與請求路由，而非讓服務直接互知彼此位置。**
