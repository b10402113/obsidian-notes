# 非同步通訊的優勢與資料複製策略

## 📝 課程概述

本單元介紹**[[Asynchronous Communication]]的第二種形式（也是本課程真正要採用的方案）**：每個 Service 維護自己的 [[Database per Service]]，並透過 **[[Event Sourcing]]（事件溯源）**模式，讓其他 Service 在本地持續同步所需的資料。這種方式雖然有資料重複（Data Duplication）的代價，但換來了**零依賴**與**極致的回應速度**，是老師在課程中明確推薦的做法。

---

## 核心觀念與實作解析

### 第二種非同步方式的核心思想

不同於第一種「用 Event Bus 等回應」的方式，這次的策略是：

> **Service DX 擁有自己的 Database，裡面存放「經過轉化」的資料（只存它實際需要的欄位）。其他 Service（Service A、B、C）在發生業務事件時，主動發射 Event，Service DX 收到後據此更新自己的資料。**

這樣 Service DX 在收到查詢請求時，**完全不需要呼叫任何人**，直接從自己的 Database 裡取出答案。

---

### 重新定義 Service DX 的職責

在實作之前，先精確定義需求：

> **Given：一個使用者的 ID。Return：該使用者曾經訂購過的所有商品標題與圖片。**

根據這個需求，Service DX 的 Database 只需要兩個 Collection：

**Users Collection（只存必要欄位）：**
```json
{ "userId": 1, "orderedProductIds": [5, 20] }
```
> 注意：這裡不存使用者的姓名、Email、年齡等——因為 Service DX 的職責根本不需要這些資訊。

**Products Collection（只存必要欄位）：**
```json
{ "productId": 5, "title": "Pants", "image": "pants.jpeg" }
{ "productId": 20, "title": "Shoes", "image": "shoes.jpeg" }
```

**查詢流程（接收到「使用者 1 的訂單」）**：
1. 去 Users Collection 找到 `{ userId: 1 }`，取得 `orderedProductIds: [5, 20]`
2. 去 Products Collection 找到 ID 為 5 和 20 的商品，取出 `title` 和 `image`
3. 直接回傳結果——**零網路呼叫，零等待**

---

### Event Sourcing：如何把資料「同步」進 Service DX？

**這是整個模式最關鍵的部分。** 流程如下：

當 Service B 新增一個商品時：
```
Service B 收到建立商品的請求
  ├── 將商品存進自己的 [[Database per Service]]
  └── 同時 [[Emit]] [[Event]] → { type: "ProductCreated", data: { id: 1, title: "Pants", image: "pants.jpeg" } }
                                                      ↓
                                              [[Event Bus]] / Broker
                                                      ↓
                                              Service DX 接收到 [[Event]]
                                                      ↓
                                              在自己的 Products Collection 中寫入對應記錄
```

當 Service A 有新使用者註冊時：
```
Service A 收到註冊請求
  ├── 將使用者存進自己的 [[Database per Service]]
  └── 同時 [[Emit]] [[Event]] → { type: "UserCreated", data: { userId: 1 } }
                                                      ↓
                                              Service DX 接收 [[Event]]
                                                      ↓
                                              在 Users Collection 中建立 { userId: 1, orderedProductIds: [] }
```

當 Service C 有新訂單時：
```
Service C 收到訂單請求
  ├── 將訂單存進自己的 [[Database per Service]]
  └── 同時 [[Emit]] [[Event]] → { type: "OrderCreated", data: { userId: 1, productId: 1 } }
                                                      ↓
                                              Service DX 接收 [[Event]]
                                                      ↓
                                              在 Users Collection 中找到 userId: 1 的記錄
                                              將 productId: 1 加入 orderedProductIds 陣列
```

---

### 這種非同步方式的優點

| 優點 | 說明 |
|---|---|
| **[[Zero Dependency]]** | Service DX 回答請求時，完全不需要聯繫 Service A、B、C。即使它們全部掛了，Service DX 依然正常運作。 |
| **極致的回應速度** | 所有資料都在本地，回應時間僅取決於本地資料庫查詢速度，沒有網路 [[Latency]]。 |
| **[[Schema]] 隔離** | Service DX 的 [[Database per Service]] 結構完全由自己掌控，不會被其他 Service 的 [[Schema]] 變更影響。 |

---

### 關於「[[Data Duplication]]」——代價真的那麼大嗎？

這是老師特別強調的重點。資料確實在多個地方重複儲存了，但這**不應該被視為缺點**：

> **「Storage 在現代世界相對免費，抱怨資料重複就像在抱怨空氣中有太多氧氣。」**

老師做了一個計算：
- 一個商品（約 1200 bytes）在雲端儲存 1 GB 的成本：
  - Azure MySQL：**$0.015 / 月**
  - AWS MySQL：**$0.017 / 月**
- 儲存 **1 億個商品**的額外費用：**約 $14 / 月**

所以儲存成本根本不是問題。真正需要關注的是：

- **[[Eventual Consistency]]**：其他 Service 的資料更新了，Service DX 的本地副本可能一時之間是舊的，這是需要在後續章節中處理的問題。
- **過度複製**：不要複製不需要的欄位（這就是為什麼我們只存 `title` 和 `image`，而不存 `price` 或 `reviews`）。

---

### 缺點：學習曲線高、程式碼量大

| 缺點 | 說明 |
|---|---|
| **概念複雜** | 需要同時理解 Event 發射機制、Event Bus 路由、以及多個 Service 的同步邏輯。 |
| **初始建置成本高** | 每個 Service 除了處理自己的主要業務邏輯，還要多寫發射事件與處理事件的程式碼。 |

---

## 💡 重點摘要

- **「每個 Service 維護自己所需的 [[Database per Service]] + [[Event Sourcing]]」是本課程採用的進階 [[Asynchronous Communication]] 模式。**
- **Service DX 的 [[Database per Service]] 結構是由它的「查詢需求」決定的，只存必要欄位，拒絕無謂 [[Data Duplication]]。**
- **這種架構下，Service DX 回答請求時完全不需要呼叫任何其他 Service——[[Zero Dependency]]，零等待。**
- **[[Data Duplication]] 的代價在現代雲端儲存成本下幾乎可以忽略不計，真正要關注的是「[[Eventual Consistency]]」的處理。**
- **[[Event Sourcing]] 讓每個 Service 成為真正意義上的「自律個體」——它既是自己業務的擁有者，也是自己資料的守護者。**
