# 事件匯流排的非同步通訊方式

## 📝 課程概述

本單元介紹**[[Asynchronous Communication]]**的第一種形式。我們將引入**[[Event Bus]]**作為所有 Service 之間的中介層，讓 Service 透過發送與監聽「[[Event]]」來溝通，而非直接呼叫彼此。雖然這種方式能解決同步通訊的依賴問題，但它仍繼承了相當多的缺點，是個「不完美但必須了解」的過渡方案。

---

## 核心觀念與實作解析

### Event Bus 的基本概念

Event Bus 是一個所有 Service 都能連接的共用中介層。當某個 Service 發生重要的事情（例如：建立了一筆訂單），它會**發射一個 Event** 到 Event Bus；Bus 負責把這個 Event **路由到有需要處理它的 Service**。

每個 [[Event]] 本質上是一個攜帶資料的物件，通常包含：
- **`type`**：描述 [[Event]] 的類型（例如 `UserCreated`、`OrderPlaced`）
- **`data`**：與 [[Event]] 相關的上下文資料（例如新使用者的 ID）

```
Service A ──(Event: UserQuery { userId: 1 })──→ Event Bus ──→ Service D
Service A ←──(Event: UserQueryResult { name: "Jill" })───── Event Bus ←── Service A
```

Service 連接到 [[Event Bus]] 後，可以：
- **[[Emit]]**：主動發送 [[Event]]
- **[[Listen]]**：接收感興趣的 [[Event]] 並執行對應邏輯

> **注意：** [[Event Bus]] 本身是整個系統中的**單一故障點（Single Point of Failure）**。因此在實際部署時，會需要特別做好 [[Event Bus]] 的高可用性（High Availability）設計。

---

### 實例演練：Event Bus 模式下的 Service DX

沿用同一個 E-Commerce 案例，Service DX 需要顯示「某位使用者購買過的所有商品」。

**處理流程：**

1. **Service DX [[Emit]] [[Event]]**：`{ type: "UserQuery", data: { userId: 1 } }`
2. **[[Event Bus]] 將 [[Event]] 路由給 Service A**（因為 Service A 處理 User 相關邏輯）
3. **Service A 接收到 [[Event]]，執行查詢，再 [[Emit]] 回應 [[Event]]**：`{ type: "UserQueryResult", data: { userId: 1, name: "Jill" } }`
4. **[[Event Bus]] 將回應 [[Event]] 路由回 Service DX**
5. Service DX 收到後，再對 Service C 發射 `OrderQuery` [[Event]]……重複相同模式取得訂單與商品資訊

整個過程中，**Service DX 完全沒有直接呼叫 Service A，也沒有觸碰任何 [[Database per Service]]**。

---

### 為什麼這種非同步方式「不完美」？

雖然 [[Event Bus]] 模式避免了直接依賴，但仍存在顯著問題：

| 缺點 | 說明 |
|---|---|
| **[[Dependency]] 關係依然存在** | 如果 Service A 沒有正常處理 [[Event]] 並回傳結果，Service DX 的請求依然會失敗或超時。 |
| **速度瓶頸不變** | 整體回應速度仍然由「最慢的 [[Event]] 處理」決定，沒有實質改善。 |
| **[[Dependency Web]] 更複雜** | 多了 [[Event Bus]] 的路由邏輯，追蹤一個請求經過了哪些 Service 變得更加困難。 |
| **概念上不易理解** | 相較於同步的「發請求→等回應」，[[Event]] → Handler → [[Event]] → Handler 的模式需要更多心智負擔。 |
| **Service DX 仍然沒有自己的 [[Database per Service]]** | 無法離線回答查詢，每次都需要等待 [[Event]] 處理完成。 |

> **這種「用 [[Event Bus]] [[Emit]] [[Event]]、等待回應 [[Event]]」的模式，通常被稱為「[[Request-Response over Events]]」——它在本質上仍然是 [[Synchronous Communication]] 思維，只是包了一層 [[Asynchronous Communication]] 的殼。**

---

## 💡 重點摘要

- **[[Event Bus]] 提供了 Service 之間的 [[Asynchronous Communication]] 管道，讓 Service 不需要知道其他 Service 的存在，僅需理解感興趣的 [[Event]] 格式（[[Event-Driven]]）。**
- **[[Event Bus]] 本身是單一故障點，需要做好高可用性設計。**
- **[[Event Bus]] 模式仍繼承了 [[Synchronous Communication]] 的 [[Dependency]] 與速度瓶頸問題，是過渡方案而非最終解答。**
- **「[[Request-Response over Events]]」在本質上仍保留了 [[Synchronous Communication]] 思維的缺點。**
- **真正理想的 [[Asynchronous Communication]]，是讓每個 Service 擁有自己完整的資料，能在**不回應其他 Service** 的情況下直接回答查詢**——這就是下一章要介紹的第二種非同步方式。
