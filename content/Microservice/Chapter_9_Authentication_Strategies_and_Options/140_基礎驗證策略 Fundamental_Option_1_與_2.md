# 基礎驗證策略：Fundamental Option 1 與 2

## 📝 課程概述

本單元正式進入微服務驗證（Authentication）的核心難題。我們將介紹兩種**基礎驗證策略**，理解「如何在分散式架構中確認使用者已登入」這個看似簡單、實則極為複雜的問題。本單元不會提供完美解答，而是讓學員理解各種方案的取捨與代價。

---

## 核心觀念與實作解析

### 驗證問題的核心提問

> **如何在微服務架構中，判斷一位發出請求的使用者是否已登入？**

當使用者嘗試購買票券時，`order` 服務需要知道：
- 這位使用者是否已通過身份驗證（Authentication）
- 這位使用者是否具有授權（Authorization），例如是否為 admin

這個問題在單體式架構中很簡單，但在微服務架構中卻極為棘手。

---

### Fundamental Option 1：依賴集中式驗證服務

**核心概念**：每個服務在處理需要驗證的請求時，主動向一個集中的 **Auth Service** 發出同步（Sync）請求，詢問「這位使用者是否有效」。

```
使用者請求 → Order Service → [同步呼叫] → Auth Service
                                    ← 回傳驗證結果 ←
```

**變體 1.1**：不讓請求直接打到各服務，而是先經過一個 **Auth Gateway**，由 Gateway 先檢查驗證狀態，通過後才轉發到目標服務。

**Option 1 的優點**：
- 驗證狀態變更（例如封鎖使用者）**立即生效**——下次請求時，各服務會立刻去 Auth Service 詢問最新狀態

**Option 1 的致命缺點**：
- **單點故障（Single Point of Failure）**：Auth Service 掛掉，**整個系統所有需要驗證的請求全部失敗**
- 完全喪失服務自治性

---

### Fundamental Option 2：各服務自己會驗證

**核心概念**：每個服務本身就知道如何驗證一個 JWT 或 Cookie，不需要依賴外部服務。

```
使用者請求 → Order Service → [本地解析 JWT] → 直接處理
```

**Option 2 的優點**：
- **完全自治**：每個服務不依賴其他服務，即使整個叢集其他服務全掛，這個服務仍可正常運作
- **失敗隔離**：某個服務崩潰不會影響其他服務的驗證能力

**Option 2 的缺點**：
- 驗證邏輯會在各服務中**重複出現**（可透過 shared library 緩解，但非根本解決）
- **隱藏的重大問題**：當 Auth Service 宣布某使用者被封鎖時，其他服務**不會立即知道**——使用者仍可拿著舊 JWT 繼續操作，直到 JWT 過期為止

> **這就是 Option 2 最核心的矛盾**：封鎖使用者的動作無法即時傳播到所有依賴 JWT 的服務。

---

## 💡 重點摘要

- **微服務驗證沒有完美解法，只有不同代價的取捨，課程會選擇 Option 2 作為最終方案。**
- **Option 1 的驗證狀態即時同步，但代價是單點故障風險——Auth Service 掛掉，系統全部失敗。**
- **Option 2 的各服務完全自治，但封鎖使用者的效果有時間窗口（取決於 JWT 有效期）。**
- **Option 2 的邏輯重複問題可透過 shared library 解決，這不是真正的阻礙。**
