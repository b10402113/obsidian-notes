# 微服務驗證機制的需求分析：Cookie 還是 JWT？

## 📝 課程概述

本單元確立了微服務驗證機制必須滿足的**五大需求**，並透過這些需求推導出「我們應該使用 JWT 作為驗證機制」的結論。理解這些需求的來源，能幫助學員在未來遇到不同場景時做出正確的技術選擇。

---

## 核心觀念與實作解析

### 五大需求

#### 需求一：驗證機制必須攜帶使用者資訊

當使用者購買票券時，`order` 服務不只要知道「這個人已登入」，還要知道「這是誰、他的 email、他是否已設定信用卡」。驗證機制必須**將這些資訊編碼進去**，而不是只回傳「是/否」。

#### 需求二：驗證機制必須包含授權資訊

某些操作（如建立免費優惠券）僅限 admin 使用。驗證機制必須能告訴後端「這位使用者的角色是什麼」，而不能只做簡單的「是否登入」檢查。

#### 需求三：驗證機制必須有防篡改設計

驗證機制必須有**內建的、不依賴後端資料庫的過期機制**。使用者無法透過修改瀏覽器設定繞過這個過期邏輯。

#### 需求四：跨語言支援

`auth` 用 Node.js/Express，`orders` 用 Ruby on Rails，`payments` 用 Java Spring——驗證機制必須**所有語言都能輕易實作**。

#### 需求五：不依賴各服務的後端資料庫

驗證機制本身不能要求每個服務都必須擁有某種特定的資料庫或儲存系統。

---

### 為什麼 Cookie 不是首選？

單純使用 Cookie-based 驗證的問題：
- Cookie 的**過期由瀏覽器管理**，使用者可以手動複製 Cookie 內容並忽略過期時間
- 跨語言實作時，不同語言的 Cookie 加密方式差異很大
- 若使用 Session ID（常見做法），則需要後端資料庫來儲存 Session 狀態——**違背需求五**

---

### 為什麼 JWT 是正確選擇？

| 需求 | JWT 如何滿足 |
|------|-------------|
| 攜帶使用者資訊 | JWT 的 Payload 可儲存任意結構的資料（ID、email、角色等） |
| 包含授權資訊 | JWT 內建支援 Role 相關欄位 |
| 防篡改 + 過期 | 過期時間 `exp` 直接**編碼在 Token 本身**，無法被使用者修改 |
| 跨語言支援 | 每種語言都有官方 JWT library，且格式一致 |
| 不依賴後端資料庫 | JWT 是**無狀態的**，驗證時不需要查詢資料庫 |

> **關鍵洞察**：Cookie 是**傳輸機制（Transport Mechanism）**，JWT 是**驗證機制（Authentication Mechanism）**。兩者並非互斥——我們可以用 **JWT 作為驗證內容，用 Cookie 作為傳輸工具**，這正是課程最終的選擇。

---

## 💡 重點摘要

- **JWT 滿足所有五大需求，特別是「過期時間內嵌於 Token 本身」是 Cookie 無法做到的關鍵特性。**
- **Cookie 與 JWT 並非對立——Cookie 是 transport，JWT 是 authentication mechanism，兩者可以同時使用。**
- **使用 Session ID + Cookie 的傳統方式需要後端資料庫支援，對微服務架構而言過於笨重。**
