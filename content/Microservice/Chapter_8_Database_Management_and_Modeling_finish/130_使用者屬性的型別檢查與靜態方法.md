# 130_使用者屬性的型別檢查與靜態方法

## 📝 課程概述

本單元實作「解決問題一」的方案：透過建立 `UserAttrs` 介面描述建立新用戶時必須提供的屬性，並包裝一個 `buildUser()` 靜態函式，讓 TypeScript 能夠在呼叫時檢查傳入的參數是否正確。後續更進一步將此函式註冊為 Mongoose Schema 的 static method，使 API 呼叫時語意更直觀（`User.build({...})` 而非獨立函式）。

---

## 核心觀念與實作解析

### 為什麼不直接呼叫 `new User()`？

因為直接呼叫 `new User()` 時，TypeScript 從 Mongoose 的型別定義中無法得知構造函式需要哪些具名參數，所以不會有任何型別檢查。

**解決思路**：把構造過程包進一個新函式，強迫 TypeScript 透過參數型別介入檢查流程。

### `UserAttrs` 介面

```typescript
interface UserAttrs {
  email: string;
  password: string;
}
```

這只是純 TypeScript 介面，用來描述「建立新用戶時，屬性物件的形狀」。它本身不會影響 Mongoose 的行為，只是作為我們的型別約束基礎。

### 第一次嘗試：`buildUser` 獨立函式

```typescript
function buildUser(attrs: UserAttrs) {
  return new User(attrs);
}
```

透過明確標註 `attrs: UserAttrs`，TypeScript 這時就會介入：

```typescript
buildUser({ email: 'test', passwordd: 'typo' }); // TypeScript 錯誤！passwordd 不存在
buildUser({ email: 'test', password: 123 });       // TypeScript 錯誤！password 應該是 string
```

### 升級為 Schema 的 static method

把 `buildUser` 直接掛在 Schema 上，呼叫時語意更自然：

```typescript
userSchema.statics.build = (attrs: UserAttrs) => {
  return new User(attrs);
};
```

現在可以這樣呼叫：

```typescript
User.build({ email: 'test', password: 'pass' });
```

### 讓 TypeScript 認識這個 static method

光是 `userSchema.statics.build = ...` 還不夠——TypeScript 不知道 Model 上會有 `build` 這個方法。我們需要一個新介面：

```typescript
interface UserModel extends mongoose.Model<UserDoc> {
  build(attrs: UserAttrs): UserDoc;
}
```

並將其傳入 `mongoose.model<UserDoc, UserModel>(...)` 的第二個泛型參數。

```typescript
const User = mongoose.model<UserDoc, UserModel>('User', userSchema);
```

### TypeScript Generics 的角色

`model<T, U>()` 接受兩個泛型參數：
- `T`（第一個）：描述 Document 的型別（即 `UserDoc`）
- `U`（第二個）：描述 Model 本身的型別（即 `UserModel`）

`U` 會直接成為 `model()` 的**回傳型別**。所以當我們寫 `const User = mongoose.model<UserDoc, UserModel>(...)` 時，TypeScript 會知道 `User` 這個變數的型別就是 `UserModel`，從而能識別 `.build()` 方法。

---

## 💡 重點摘要

- `UserAttrs` 介面定義了「建立用戶時需要什麼」，而非「用戶文件有哪些屬性」——這兩個是不同的概念
- 將 `buildUser` 封裝成 static method 是為了讓 API 更直觀：不再需要 import 獨立函式，直接 `User.build(...)`
- `mongoose.model<T, U>()` 的第二個泛型參數決定了回傳值（Model 本身）的型別，讓 TypeScript 能識別靜態方法
- 這段 workaround 程式碼將在所有 service 中複製使用，了解原理後複製就不會迷失
