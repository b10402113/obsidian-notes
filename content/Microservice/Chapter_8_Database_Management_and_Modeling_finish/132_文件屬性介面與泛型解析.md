# 132_文件屬性介面與泛型解析

## 📝 課程概述

本單元解決「問題二」：當 Mongoose Document 上出現我們沒明確傳入的屬性（如 `_id`、`createdAt`）時，TypeScript 無法認知這些屬性。我們會建立 `UserDoc` 介面並將其設定為 `mongoose.model` 的第一個泛型參數，讓 TypeScript 完整掌握 Document 上所有可能出現的屬性。同時補充說明「泛型」在這段程式碼中的具體意義。

---

## 核心觀念與實作解析

### `UserDoc` 介面

```typescript
interface UserDoc extends mongoose.Document {
  email: string;
  password: string;
}
```

`mongoose.Document` 是 Mongoose 所有 Document 類別的基底介面，繼承它代表「我們的 Document 會有基底的所有能力，再加上以下這些屬性」。

如果我們在 Document 上有 `createdAt` 或 `updatedAt`（由 Mongoose 自動維護），也應該在這裡聲明：

```typescript
interface UserDoc extends mongoose.Document {
  email: string;
  password: string;
  createdAt: string;  // Mongoose 自動維護
  updatedAt: string;  // Mongoose 自動維護
}
```

### 介面的分層設計

完整看下來，這個 model 檔案中會有 **三層介面**，各自職責不同：

| 介面 | 職責 | 誰用 |
|------|------|------|
| `UserAttrs` | 建立新 Document 時要傳入的資料 | `build()` 的參數型別 |
| `UserDoc` | Document 實例上所有可存取的屬性 | `model<T>` 的 T |
| `UserModel` | Model 類別本身（含 static 方法） | `model<T, U>` 的 U |

### 泛型 `<T, U>` 的完整解析

```typescript
const User = mongoose.model<UserDoc, UserModel>('User', userSchema);
```

以 `model` 函式的視角來看：

```
泛型參數一（T）→ 文件實例的型別（UserDoc）
泛型參數二（U）→ Model 類別本身的型別（UserModel）
                 └── 這個型別會成為 model() 的回傳型別
```

所以當我們寫：

```typescript
const user = User.build({ email: 'test', password: 'pass' });
```

TypeScript 推斷出：
- `User.build()` 的回傳值是 `UserDoc`（因為 `UserModel.build` 的回傳型別是 `UserDoc`）
- 因此 `user.email`、`user.password` 都可以安全存取

### 如果不指定泛型會發生什麼？

```typescript
// 故意把第二個泛型參數改成 any
const User = mongoose.model<UserDoc, any>('User', userSchema);
```

這時 TypeScript 認為 `User` 的型別是 `any`，所以：
- `.build()` 可以呼叫（但 TypeScript 不知道它存在）
- 回傳值的型別變成 `any`，所有屬性存取都沒有型別檢查

這正好說明了我們為什麼要費力建立 `UserModel` 介面——就是要讓 TypeScript 有足夠的資訊來做靜態檢查。

---

## 💡 重點摘要

- `UserDoc` 介面讓 TypeScript 知道 Document 上所有可存取的屬性，包括 Mongoose 自動維護的那些
- 三層介面分工明確：`UserAttrs`（輸入）→ `UserDoc`（文件實例）→ `UserModel`（類別本身）
- `model<T, U>()` 的第二個泛型參數決定了回傳值的型別，這是讓 `.build()` 方法能被 TypeScript 識別的關鍵
- 泛型的核心價值在於「把型別本身當作參數傳入函式」，讓函式能針對不同型別做出不同的型別推斷
