# 136 密碼雜湊的重要性與運作原理

## 📝 課程概述

本單元介紹密碼儲存的正確觀念。**絕對不能以明文儲存密碼**——若資料庫被竊取，使用者很可能在多個網站使用相同密碼，攻擊者就能用同一組帳密組合登入其他網站。密碼雜湊（Hashing）搭配 Salt，是解決這個問題的標準做法。

---

## 核心觀念與實作解析

### 錯誤做法：明文儲存密碼

```
React App ──密碼明文──→ auth service ──明文──→ MongoDB
```

一旦攻擊者取得資料庫存取權，所有使用者的 Email + 密碼組合全部外洩，且攻擊者可立刻拿這些組合去其他網站嘗試登入。

### 正確做法：Hashing + Salt

**Hashing** 的特性：
- 同一段文字經過同一個 Hash 函式，永遠得到相同的 Hash 值。
- 無法從 Hash 值反推原始輸入（單向函式）。
- 輸入稍有不同，Hash 值就完全不同。

**Salt** 的特性：
- 攻擊者可用「彩虹表」（預先計算好的常見密碼 Hash 表）來破解常見密碼。
- Salt 是附加在每個密碼上的隨機字串，確保相同密碼也會產生不同 Hash。
- 即使攻擊者拿到 Salt，也難以用彩虹表破解。

### Sign Up 時的處理流程

```
收到密碼 "wordpass"
    → 產生隨機 Salt（例如 "abc123"）
    → Hash("wordpass", "abc123") → "9f86d0...abc123"
    → 儲存至資料庫（Hash + Salt）
```

### Sign In 時的比對流程

```
收到密碼 "wordpass"
    → 從資料庫取出完整儲存值（含 Salt）
    → Hash("wordpass", 取出的Salt) → 比對儲存的 Hash 值
    → 相同 → 登入成功
```

---

## 💡 重點摘要

- **密碼永遠不該以明文儲存**，這是資訊安全的基本原則。
- Salt 的作用是確保相同密碼在不同使用者之間產生不同 Hash，防止彩虹表攻擊。
- 實務上使用 `bcrypt` 這類專門設計的雜湊函式，它內建 Salt 機制並支援可設定的運算成本。

## 🔑 關鍵字

Password Hashing, Salt, bcrypt, 彩虹表, 單向函式
