# 134_使用者註冊流程實作

## 📝 課程概述

有了完整的 User Model 之後，我們終於可以實作註冊流程的核心邏輯。整個流程依序為：檢查 Email 是否已被使用 → 建立新用戶 Document 並雜湊密碼 → 存回資料庫。本單元完成了前兩個步驟，並設計了統一的錯誤處理機制——當 Email 已被使用時拋出 `BadRequestError`，讓所有錯誤都走同一條處理管道。

---

## 核心觀念與實作解析

### 檢查 Email 是否已被使用

```typescript
const existingUser = await User.findOne({ email });

if (existingUser) {
  throw new BadRequestError('Email in use');
}
```

`User.findOne()` 是 Mongoose 提供的查詢方法，會在 users collection 中找到第一個符合條件的文件並以 `UserDoc` 型別傳回。如果找不到，回傳值是 `null`。

### 建立用戶 Document

```typescript
const user = User.build({ email, password });
await user.save();
```

請注意：`User.build()` 只是**建立記憶體中的 Document 物件**，並不會寫入資料庫。要真正持久化儲存，必須呼叫 `.save()`。這個分離設計讓我們可以在正式儲存前對 Document 做更多處理（比如雜湊密碼）。

### 自訂錯誤類別 `BadRequestError`

```typescript
export class BadRequestError extends CustomError {
  statusCode = 400;

  constructor(public message: string) {
    super(message);
    Object.setPrototypeOf(this, BadRequestError.prototype);
  }

  serializeErrors() {
    return [{ message: this.message }];
  }
}
```

這個錯誤類別的設計原則：

- **`statusCode = 400`**：通用客戶端錯誤（如輸入驗證失敗、資源衝突），比 500 更準確描述錯誤來源
- **`serializeErrors()`** 的回傳格式是陣列，因為一個請求可能同時有多個錯誤（例如多個欄位驗證失敗）
- **`Object.setPrototypeOf`** 是因為在 TypeScript 中擴展內建類（如 `Error`）時，需要手動設定原型鏈，否則 `instanceof` 可能失效

### 為什麼 `BadRequestError` 不同於 `RequestValidationError`？

`RequestValidationError` 是專門用來處理 `express-validator` 回傳結果的錯誤類別，**只能用在 validator 驗證環節**。

`BadRequestError` 是通用錯誤類別，用於所有「請求本身有問題」的情境（Email 已存在、資源不足、格式錯誤等）。它的設計目標是：**任何在 route handler 內發現的業務邏輯錯誤，都應該丟出 `BadRequestError`**，讓統一錯誤中介層統一處理回傳格式。

---

## 💡 重點摘要

- `User.build()` 只建立 Document 物件，`user.save()` 才真正寫入 MongoDB
- `User.findOne({ email })` 是查詢現有用戶的主要方式，找不到時回傳 `null`
- `BadRequestError` 是通用客戶端錯誤類別，任何業務邏輯層面的「壞請求」都應該用它
- 統一錯誤格式的好處：前端 React 只需要處理一種錯誤結構，不需要為每種錯誤情境寫不同的解析邏輯
