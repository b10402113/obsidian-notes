# 136_密碼雜湊的重要觀念

## 📝 課程概述

在實作密碼雜湊程式碼之前，本單元先建立正確的安全觀念。我們會看到「將密碼明文儲存」為什麼是危險的設計，以及「密碼雜湊」在 Sign Up 與 Sign In 兩個流程中各自扮演什麼角色。這個概念章節看似簡單，卻是理解後續所有實作細節的基礎。

---

## 核心觀念與實作解析

### 為什麼不能儲存明文密碼？

當我們把密碼以明文儲存在資料庫時，資料庫被入侵（Data Breach）的後果遠比「攻擊者登入我們的網站」嚴重得多——**使用者通常會在多個網站重複使用同一組帳密**。攻擊者取得資料後，可以用同樣的組合去嘗試其他網站（所謂的「撞庫攻擊」）。

這不是假設性的威脅，而是現實中反覆發生的安全事故。身為後端工程師，**永遠不儲存明文密碼是基本責任**。

### Sign Up 時的雜湊流程

```
1. 使用者提交：email = "test@test.com", password = "wordpass"（明文）
2. 我們對 "wordpass" 做雜湊 → 得到一個固定長度的乱碼字串（"1c2b3d4e..."）
3. 只儲存雜湊後的密碼，**绝不**儲存原始密碼
```

同一個明文密碼每次雜湊的結果都相同（給定相同的 Salt，見下節），所以日後可以比對。

### 什麼是 Salt（鹽）？

Salt 是一段隨機產生的字串，加在密碼前面再做雜湊。作用是：**讓相同的明文密碼產生不同的雜湊值**，防止攻擊者用「彩虹表」（預先計算好的常見密碼對應表）反向查詢。

```
"wordpass" + 隨機鹽 → 雜湊 → "1c2b3d..."
```

每次建立新用戶時都會產生新的 Salt，Salt 本身不需要保密（它會和雜湊結果一起儲存在資料庫中）。

### Sign In 時的比對流程

```
1. 使用者提交：email = "test@test.com", password = "wordpass"
2. 用 email 找到資料庫中的記錄（包含 hash + salt）
3. 用找到的 salt，對使用者這次提交的密碼重新做雜湊
4. 比對：重新計算的 hash === 資料庫儲存的 hash？
   - 相等 → 密碼正確，登入成功
   - 不等 → 密碼錯誤
```

這個比對流程的關鍵在於：**我們永遠不解密**，而是把使用者提供的密碼用同樣的演算法再跑一次，比對結果是否一致。

### 為什麼 bcrypt 是合適的選擇？

`bcrypt` 是專為密碼雜湊設計的演算法，特點包括：
- 內建 Salt 機制
- 計算成本可調整（隨硬體進步可以提高難度，防止暴力破解）
- 輸出長度固定，無法從輸出反推輸入長度

Node.js 的 `crypto` 模組原生提供 `crypto.scrypt()`，是另一個可接受的選擇（課程使用它）。

---

## 💡 重點摘要

- 永遠不儲存明文密碼——資料庫被入侵的代價不只是「別人登入你的網站」，而是使用者所有其他帳戶的安全都受威脅
- Salt 的作用是防止相同密碼產生相同雜湊，讓彩虹表攻擊失效
- Sign In 時的比對是「重新計算雜湊後比對」，不是「解密後比對」
- bcrypt/scrypt 這類專門為密碼設計的雜湊演算法，比一般 MD5/SHA 系列更適合儲存密碼
