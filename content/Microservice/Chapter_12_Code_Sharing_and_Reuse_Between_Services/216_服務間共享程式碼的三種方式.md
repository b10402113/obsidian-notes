# 服務間共享程式碼的三種方式

## 📝 課程概述

本單元是 Chapter 12 的開篇。我們即將進入下一個重要服務——Ticket Service（票券服務），但在實作它之前，必須先建立一套跨服務共享程式碼的機制。我們會分析三種常見的程式碼共享策略，並選擇最符合本專案需求的方案。

---

## 核心觀念與實作解析

### 為什麼我們需要程式碼共享？

在進入 Ticket Service 之前，讓我們先思考一個問題：Auth Service 已經實作了 `require auth` middleware 與自訂錯誤系統，而 Ticket Service 同樣需要這些功能。如果我們從頭重寫一次，無疑是浪費時間，也會造成日後維護的噩夢。

**核心需求是：將 Auth Service 中可複用的邏輯（例如 middleware 與錯誤處理）抽取出來，讓所有服務都能使用。**

---

### 三種程式碼共享方案

#### 方法一：直接 Copy & Paste

最直覺的做法：直接複製 `middleware` 與 `errors` 資料夾到 Ticket Service。

**致命缺點**：日後當你在 Auth Service 更新了 shared code，沒有任何機制可以自動同步到 Ticket Service。你只能手動再複製一次，而且難以追蹤哪些檔案被修改過。

這種方式幾乎不具任何長期可行性。

---

#### 方法二：Git Submodule

將共用的程式碼单独建立一個 Git Repository，再將它以 submodule 的方式分別加入到 Auth 與 Ticket 的 repo 中。

**優點**：
- 所有共用程式碼都在版本控制之下
- 可以精確知道每個專案目前使用的版本

**缺點**：
- Git submodule 的指令較為繁瑣，新增、更新、移除 submodule 都有一定的學習曲線
- 在 CI/CD 流程中，submodule 的處理並不直覺

---

#### 方法三：發布為 NPM Package（我們選擇的方案）

將共用程式碼封裝成一個獨立的 NPM 模組，發布到 NPM Registry（或私人 Organization），其餘服務透過 `npm install` 來取用。

**運作方式**：

```bash
# 發布 package
cd common
npm publish                  # 發布到 @org-name/common

# 在各服務中安裝
npm install @org-name/common
```

**為什麼這是最優方案？**

- **版本控制**：每個服務可以固定使用某個版本，不必強制一起升級
- **團隊協作**：新成員 `npm install` 後立即擁有所有共用模組
- **維護簡單**：修改 Common Module → 發布新版本 → 各服務 `npm update` 即可

**缺點也要正視**：

每次修改 Common Module 都需要經歷：`修改程式碼 → commit → `npm version patch` → `npm run build` → `npm publish``，然後各服務再 `npm update`。這個來回流程確實比 copy & paste 多了幾個步驟。不過團隊已經設計了一個 `pub` script 來簡化這個流程。

---

### 過渡到 Ticket Service

一旦 Common Module 準備好，Ticket Service 的實作速度將遠比 Auth Service 快，因為所有 Foundation 層（middleware、錯誤處理、驗證邏輯）都已經準備就緒。

> **Ticket Service 的核心功能**：列出票券、顯示單一票券詳情、建立票券、編輯票券。這四個 CRUD 操作中，「建立」與「編輯」需要 authentication，這正好就是我們從 Auth Service 抽取出的 `require auth` middleware 所擅長處理的事情。

---

## 💡 重點摘要

- **直接 copy & paste 看似快速，卻沒有任何版本同步機制，長期維護會成為災難。**
- **Git Submodule 解決了版本控制問題，但指令複雜，在 CI/CD 環境中處理不直覺。**
- **將共用程式碼發布為 NPM Package，是多數大型團隊採用的最佳實踐，支援版本管理與精準更新。**
- **Ticket Service 的建立與編輯路由都依賴 `require auth` middleware，這正是我們要先抽取共用邏輯的原因。**
- **Common Module 建立起來之後，後續所有新服務的啟動速度將大幅提升。**

---
