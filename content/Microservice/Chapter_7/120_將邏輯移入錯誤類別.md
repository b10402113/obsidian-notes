# 120_將邏輯移入錯誤類別

## 📝 課程概述

本單元重構了錯誤格式化的職責分配：我們不再讓 Error Handling Middleware 知道每種 Error 類型的內部結構，而是將 `serializeErrors()` 方法與 `statusCode` 屬性直接定義在各自的 Error 子類別中。這大幅簡化了 Middleware 的邏輯，也讓新增錯誤類型時無需修改 Middleware。

---

## 核心觀念與實作解析

### 目前的問題：Middleware 知道太多

若繼續將格式化邏輯寫在 Middleware 中，當新增更多 Error 類型時，Middleware 會變成這樣：

```typescript
// ❌ 隨著錯誤類型增加，這個檔案會不斷膨脹
if (err instanceof RequestValidationError) { /* 格式化邏輯 */ }
if (err instanceof DatabaseConnectionError) { /* 格式化邏輯 */ }
if (err instanceof NotAuthorizedError) { /* 格式化邏輯 */ }
if (err instanceof NotFoundError) { /* 格式化邏輯 */ }
// ... 這種模式會無限期延續
```

### 反轉職責：每個 Error 自己知道怎麼被序列化

**設計原則**：每個 Error 類別應該自己實作：
1. **`statusCode` 屬性**：告訴 Middleware 應該回傳哪個 HTTP 狀態碼
2. **`serializeErrors()` 方法**：告訴 Middleware 應該回傳什麼格式的錯誤陣列

```typescript
// errors/database-connection-error.ts
export class DatabaseConnectionError extends Error {
  public statusCode = 500;

  serializeErrors() {
    return [{ message: this.reason }];
  }
}

// errors/request-validation-error.ts
export class RequestValidationError extends Error {
  public statusCode = 400;

  serializeErrors() {
    return this.errors.map((e) => ({
      message: e.msg,
      field: e.param,
    }));
  }
}
```

### Middleware 簡化後的模樣

現在 Middleware 完全不知道任何 Error 類型的內部實作，只需要：

```typescript
// ✅ Middleware 變得極度簡潔
export const errorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  if (err instanceof RequestValidationError || err instanceof DatabaseConnectionError) {
    return res.status(err.statusCode).send({ errors: err.serializeErrors() });
  }

  // 最後的 fallback
  return res.status(400).send({
    errors: [{ message: 'Something went wrong' }],
  });
};
```

> **好處**：未來新增任何 Error 類型，只需要確保它有 `statusCode` 與 `serializeErrors()`，Middleware 無需任何修改。

### 為什麼要用反轉的設計？

這個模式遵循了 **Tell, Don't Ask** 原則：

- **舊做法（Ask）**：Middleware 問 Error：「你的錯誤陣列在哪？我幫你組裝。」
- **新做法（Tell）**：Middleware 告訴 Error：「你自己格式化後給我。」

---

## 💡 重點摘要

- **將 `statusCode` 與 `serializeErrors()` 放進 Error 類別，讓 Middleware 不再需要知道每種錯誤的實作細節。**
- **`serializeErrors()` 回傳的格式（`{ message, field? }[]`）完全由 Error 類別自己決定，確保一致性。**
- **這個重構讓 Middleware 的複雜度不再隨 Error 類型數量線性成長，達成了 Open/Closed Principle。**
