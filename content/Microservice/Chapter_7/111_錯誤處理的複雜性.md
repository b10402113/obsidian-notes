# 111_錯誤處理的複雜性

## 📝 課程概述

本單元揭示了**微服務架構中錯誤回應一致性的核心問題**：當多個 Service 使用不同語言、不同框架各自處理錯誤時，回傳給 client（React App）的錯誤結構會完全不同——這會讓前端工程師負擔極重的 Error Parsing 邏輯。我們將從這個問題出發，定義統一錯誤結構的必要性。

---

## 核心觀念與實作解析

### 微服務的多語言現實

在微服務架構中，不同 Service 可能由不同團隊、使用不同技術栈實作：

- **Auth Service**：Node.js + Express + express-validator
- **Order Service**：Ruby on Rails
- **Payment Service**：Java Spring

當前端 React App 提交錯誤資料給任意一個 Service 時，**每個 Service 回傳的錯誤結構都不一樣**：

```json
// Express Validator 的錯誤結構（陣列 of objects）
[
  { "msg": "Email must be valid", "param": "email", "location": "body" },
  { "msg": "Password must be between 4 and 20 characters", "param": "password", ... }
]

// Ruby on Rails 的錯誤結構（可能是完全不同的格式）
{ "errors": { "email": ["is invalid"], "password": ["is too short"] } }
```

### 這個問題為什麼很嚴重？

若不統一錯誤結構，前端工程師必須：

1. 知道如何解析 Express 的錯誤格式
2. 知道如何解析 Rails 的錯誤格式
3. 知道如何解析 Spring 的錯誤格式
4. 每新增一個 Service，就得多寫一種解析邏輯

> **關鍵 insight**：讓前端知道每個 Service 的錯誤實作細節，是一種**職責錯置（Separation of Concerns Violation）**。前端只需要一種統一的錯誤結構。

### 錯誤發生的多個情境

不只是 Validation 可能產生錯誤，以下情境都可能需要回傳錯誤：

| 情境 | 錯誤來源 |
| --- | --- |
| **情境一** | 資料格式驗證失敗（express-validator）|
| **情境二** | 業務邏輯驗證失敗（例：Email 已被註冊）|
| **情境三** | 資料庫連線失敗（Database 層）|

無論錯誤來自哪一層，**都必須回傳相同結構的錯誤回應**。

### 解決方向

我們需要確保：

> **所有 Service 在任何錯誤發生時，都回傳完全一致的錯誤結構。**

這個結構的具體格式尚未定義，但我們已經確認了設計原則：讓 React 前端只需要處理**一種**錯誤格式，而非 N 種。

---

## 💡 重點摘要

- **微服務的多語言特性使得各 Service 的錯誤結構天然不同，這是架構層面的問題，不是單一程式碼能解決的。**
- **統一錯誤結構能解放前端工程師，讓他們不需要理解每個 Service 的實作細節。**
- **錯誤可能來自 Validation、Business Logic、Database 等多個層次，捕獲與轉換必須統一處理。**
