# 117_透過子類別建立自訂錯誤

## 📝 課程概述

本單元正式實作 `RequestValidationError` 與 `DatabaseConnectionError` 兩個自訂錯誤類別，並將它們整合進 Route Handler 中。同時預告了下一階段的 refactor 方向：將錯誤格式化的邏輯從 Middleware 移到各個 Error 子類別本身，實現真正的**關注點分離（Separation of Concerns）**。

---

## 核心觀念與實作解析

### RequestValidationError 的完整實作

```typescript
// errors/request-validation-error.ts
import { ValidationError } from 'express-validator';

export class RequestValidationError extends Error {
  public errors: ValidationError[];

  constructor(errors: ValidationError[]) {
    super('Invalid request parameters');
    this.errors = errors;
    Object.setPrototypeOf(this, RequestValidationError.prototype);
  }
}
```

在 Route Handler 中使用：

```typescript
import { RequestValidationError, DatabaseConnectionError } from '../errors';

if (!errors.isEmpty()) {
  throw new RequestValidationError(errors.array());
}

// ...
if (dbConnectionFailed) {
  throw new DatabaseConnectionError();
}
```

### TypeScript 中 extends 內建類的特殊注意事項

當 `class A extends Error`（而非 `class B`）時，有一個 TypeScript 特有的陷阱：**編譯出來的 JavaScript 會讓 `instanceof` 判斷失效**。

解決方法是在 constructor 最後加上：

```typescript
Object.setPrototypeOf(this, RequestValidationError.prototype);
```

這行程式碼讓 JavaScript 引擎知道 `this` 的原型鏈已經更新到 `RequestValidationError`，否則 `err instanceof RequestValidationError` 會錯誤地回傳 `false`。

> 這行在純 JavaScript 繼承中不需要，但 TypeScript 在編譯 `class` 時會產生不同的原型鏈，因此是必要的。

### Middleware 中的 instanceof 驗證

實作完成後，透過 Postman 測試可以確認：

1. 傳入無效資料 → `instanceof RequestValidationError` 為 `true`，Middleware 正確識別
2. 傳入有效資料（DB 連線失敗模擬）→ `instanceof DatabaseConnectionError` 為 `true`

```typescript
if (err instanceof RequestValidationError) {
  console.log('Handling this error as a request validation error');
}

if (err instanceof DatabaseConnectionError) {
  console.log('Handling this error as a DB connection error');
}
```

### 下一階段：將邏輯反轉

目前 Middleware 知道太多細節：它需要知道如何從 `RequestValidationError.errors` 取值並 map 成格式化陣列。這樣當我們新增更多 Error 類型時，Middleware 會不斷膨脹。

**解決方向**：把「如何序列化自己」的邏輯從 Middleware 移到 Error 類別本身，讓 Middleware 只需要問：「你應該怎麼被格式化？」

---

## 💡 重點摘要

- **Subclass Error 時，constructor 中呼叫 `Object.setPrototypeOf(this, ClassName.prototype)` 是 TypeScript 環境下的必要步驟，否則 `instanceof` 會失效。**
- **`instanceof` 是 Middleware 識別錯誤類型的標準方式，可用來觸發不同的處理邏輯。**
- **將邏輯保持在 Middleware 中會造成未來擴充時的維護困難，需要將職責歸還給各 Error 類別。**
