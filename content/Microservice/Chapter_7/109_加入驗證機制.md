# 109_加入驗證機制

## 📝 課程概述

本單元為 `signup` 路由處理器加入**輸入驗證（Validation）**功能。我們使用 `express-validator` 套件對 request body 中的 `email` 與 `password` 進行結構檢查與消毒（sanitization），並說明如何在驗證失敗時將錯誤資訊回傳給用戶端。

---

## 核心觀念與實作解析

### 為什麼不能直接假設 input 存在？

初學者常犯的錯誤是直接假設用戶一定會傳入正確的資料格式：

```typescript
// ❌ 不安全的寫法
const { email, password } = req.body;
// 直接使用，沒有任何檢查
```

這在正式環境中極度危險——攻擊者可以傳入任意資料，甚至利用特殊字元進行 Injection 攻擊。因此**所有外部輸入都必須經過驗證**。

### express-validator 的核心概念

`express-validator` 包含兩個核心功能：

| 功能 | 說明 |
| --- | --- |
| **Validation** | 確認資料的結構與格式是否正確（如：是否為合法 Email 格式）|
| **Sanitization** | 修改或淨化資料內容（如：去除多餘空白、移除特殊字元）|

兩者經常搭配使用，因為驗證通過的資料，有時也需要預先處理。

### 實際安裝與使用

**安裝：**
```bash
npm install express-validator
```

**匯入與設定：**

```typescript
import { body, validationResult } from 'express-validator';

// 在路由處理器中作為 middleware 陣列加入
router.post(
  '/api/users/signup',
  [
    body('email')
      .isEmail()
      .withMessage('Email must be valid'),
    body('password')
      .trim()
      .isLength({ min: 4, max: 20 })
      .withMessage('Password must be between 4 and 20 characters'),
  ],
  (req: Request, res: Response) => {
    // 取出驗證結果
    const errors = validationResult(req);

    if (!errors.isEmpty()) {
      return res.status(400).send(errors.array());
    }

    res.send({});
  }
);
```

### validationResult 的運作原理

當 `body('field')` 這類 middleware 執行時，它會在 `req` 物件上附加驗證結果。之後在 Handler 中呼叫 `validationResult(req)` 就能取出這個結果物件。

若 `errors.isEmpty()` 為 `false`，代表有驗證失敗，我們應該：

1. **early return**，不再執行後續邏輯
2. 將錯誤陣列以 `errors.array()` 取出並傳回 client

> **為什麼要用 early return？** 驗證失敗時，後續的 business logic（建立用戶、寫入資料庫等）根本不需要執行。提早回應也能讓 client 更快收到回饋。

---

## 💡 重點摘要

- **所有外部輸入都必須經過驗證，絕不能假設用戶一定會提供正確格式的資料。**
- **`express-validator` 的 `body()` 是 middleware，可直接插入路由定義中作為驗證層。**
- **`validationResult(req)` 是讀取驗證結果的標準方式；`errors.array()` 將結果轉為普通陣列供 JSON 回傳。**
- **Sanitization（如 `trim()`）通常在 Validation 之後執行，確保資料進入 business logic 前已被淨化。**
