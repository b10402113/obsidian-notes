# 121_驗證自訂錯誤實作

## 📝 課程概述

本單元為 Error 類別引入了 **TypeScript 編譯時期的型別檢查機制**：我們建立一個 `CustomError` Abstract Class，作為所有自訂錯誤的基底類別。透過 `implements` 與 Abstract Class 的結合，確保每個 Error 子類別都正確實作了 `statusCode` 與 `serializeErrors()`，預防執行時期才發現錯誤的悲劇。

---

## 核心觀念與實作解析

### 方案一：Interface（介面）

最直覺的做法是定義一個 Interface，讓各 Error 類別 `implements` 它：

```typescript
interface CustomError {
  statusCode: number;
  serializeErrors(): { message: string; field?: string }[];
}
```

這確實能在編譯時期檢查實作完整性，但有一個問題：**Interface 在編譯成 JavaScript 後會完全消失**，無法用於執行時期的型別判斷（如 `instanceof`）。

### 方案二：Abstract Class（抽象類別）

我們採用 Abstract Class 的做法：

```typescript
// errors/custom-error.ts
export abstract class CustomError extends Error {
  abstract statusCode: number;

  abstract serializeErrors(): { message: string; field?: string }[];

  constructor(message: string) {
    super(message);
    Object.setPrototypeOf(this, CustomError.prototype);
  }
}
```

**Abstract Class 與 Interface 的關鍵差異**：

| 特性 | Interface | Abstract Class |
| --- | --- | --- |
| 編譯成 JS 後存在？ | 否（完全消失） | 是（仍是 class 定義）|
| 支援 `instanceof`？ | 否 | 是 |
| 可定義 constructor？ | 否 | 是 |
| 可被 `extends`？ | 否 | 是 |

### 讓所有 Error 類別 extends CustomError

```typescript
// errors/request-validation-error.ts
export class RequestValidationError extends CustomError {
  public statusCode = 400;

  constructor(public errors: ValidationError[]) {
    super('Invalid request parameters');
    Object.setPrototypeOf(this, RequestValidationError.prototype);
  }

  serializeErrors() {
    return this.errors.map((e) => ({
      message: e.msg,
      field: e.param,
    }));
  }
}
```

### TypeScript 編譯時期的自動檢查

一旦使用了 `extends CustomError`，TypeScript 會**自動檢查**以下項目：

- 是否定義了 `statusCode: number`
- 是否實作了 `serializeErrors()` 方法且回傳型別正確
- 若漏寫或打錯字，**編譯當下就會噴錯**，而不是等到執行時期才爆問題

### Middleware 的最終形態

由於所有 Error 都 extends `CustomError`，Middleware 只需要一個 `instanceof` 判斷就夠了：

```typescript
export const errorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  if (err instanceof CustomError) {
    return res.status(err.statusCode).send({ errors: err.serializeErrors() });
  }

  return res.status(400).send({
    errors: [{ message: 'Something went wrong' }],
  });
};
```

**只要是 `CustomError` 的子類別，無論新增多少種，Middleware 都能自動正確處理。**

---

## 💡 重點摘要

- **Abstract Class 在編譯成 JS 後仍存在，支援 `instanceof` 判斷，這是它比 Interface 更適合這個場景的關鍵原因。**
- **`extends CustomError` 後，TypeScript 會在編譯時期強制檢查所有子類別是否正確實作了 `statusCode` 與 `serializeErrors()`。**
- **Middleware 現在只需要一個 `instanceof CustomError` 判斷，就能處理所有自訂錯誤，新增 Error 類型不再需要修改 Middleware。**
