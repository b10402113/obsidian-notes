# 119_將錯誤轉換為標準回應

## 📝 課程概述

本單元正式定義了「統一錯誤回應結構」，並將格式化邏輯直接實作在 Error Handler Middleware 中。所有錯誤最終都會被轉換為 `{ errors: [{ message: string, field?: string }] }` 的格式回傳給 client，確保 React 前端只需要處理一種錯誤結構。

---

## 核心觀念與實作解析

### 統一錯誤回應結構

經過前面的討論，我們現在正式定義統一的錯誤回應格式：

```typescript
// 統一的錯誤回應格式
{
  errors: [
    { message: "Email must be valid", field: "email" },
    { message: "Password must be between 4 and 20 characters", field: "password" }
  ]
}
```

這個結構的設計原則：

- **`errors` 永遠是陣列**：即使只有一個錯誤也包成陣列，保持 client 端處理邏輯一致
- **`message`（必填）**：人類可讀的錯誤說明
- **`field`（選填）**：發生錯誤的欄位名稱，方便前端直接對應到表單欄位

### Middleware 中的格式化邏輯

```typescript
// middleware/error-handler.ts
export const errorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  if (err instanceof RequestValidationError) {
    // 從 err.errors 取出 ValidationError[]，map 成標準格式
    const formattedErrors = err.errors.map((e) => ({
      message: e.msg,
      field: e.param,
    }));
    return res.status(400).send({ errors: formattedErrors });
  }

  if (err instanceof DatabaseConnectionError) {
    return res.status(500).send({
      errors: [{ message: err.reason }],
    });
  }

  // 最後的 fallback：未知錯誤也轉成相同格式
  return res.status(400).send({
    errors: [{ message: err.message }],
  });
};
```

### HTTP 狀態碼的選擇

不同錯誤類型應使用不同的 HTTP 狀態碼：

| 錯誤類型 | 狀態碼 | 含義 |
| --- | --- | --- |
| `RequestValidationError` | `400` Bad Request | 用戶端傳入了格式錯誤的資料 |
| `DatabaseConnectionError` | `500` Internal Server Error | 伺服器端發生非預期錯誤 |
| 其他未知錯誤 | `400` | 通用錯誤回應 |

> **為什麼 DB 錯誤用 500 而非 400？** HTTP 規格中，`4xx` 系列代表「client 端的錯誤」，而 `5xx` 代表「server 端的錯誤」。資料庫連線失敗不是用戶造成的，所以用 500 更符合語義。

### 為什麼統一結構如此重要？

有了這個一致的结构，React 前端的 Error Handling 可以極度簡化：

```typescript
// React 中的錯誤處理（示意）
if (response.data.errors) {
  setFormErrors(response.data.errors);
}
```

無論錯誤來自哪個 Service、前端只需要知道「看 `response.data.errors`」即可——不需要針對每個 Service 寫不同的解析邏輯。

---

## 💡 重點摘要

- **統一的錯誤結構 `{ errors: [{ message, field? }] }` 是微服務架構中前端與後端分離的關鍵設計。**
- **`400` 適用於用戶端造成的錯誤（如驗證失敗）；`500` 適用於伺服器端錯誤（如 DB 連線失敗）。**
- **Middleware 的最後一個 fallback 確保任何未知錯誤也遵循相同結構，避免前端收到非預期的格式。**
