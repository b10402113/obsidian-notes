# 113_錯誤處理方案

## 📝 課程概述

本單元正式提出解決方案：我們將使用 **Error Handling Middleware** 統一攔截所有錯誤，並確保 Express 的同步（sync）與非同步（async）Handler 都能正確地將錯誤傳遞給 Middleware。同時建立 `errors/` 目錄與 `middleware/` 目錄的初始結構。

---

## 核心觀念與實作解析

### Express 錯誤處理的兩個情境

Express 對錯誤的處理分為兩種情境：

**情境一：同步（Synchronous）Route Handler**

```typescript
router.get('/test', (req, res) => {
  throw new Error('Something went wrong!');
});
```

在同步程式碼中，`throw` 的錯誤會被 Express **自動捕獲**，並轉發給 Error Handling Middleware——**不需要手動呼叫 `next()`**。

**情境二：非同步（Async）Route Handler（含 async/await）**

```typescript
router.get('/test', async (req, res) => {
  await someAsyncOperation();
});
```

在 async 函式中，`throw` 的錯誤**不會自動被 Express 捕獲**。我們必須在 `try/catch` 中手動呼叫 `next(error)`：

```typescript
router.get('/test', async (req, res, next) => {
  try {
    await someAsyncOperation();
  } catch (err) {
    next(err); // 手動轉發給 Error Handling Middleware
  }
});
```

> **為什麼有這個差異？** Express 最初是為同步程式碼設計的。async/await 是後來才引入的語法，Express 需要透過 `next()` 才能知道有錯誤發生。

### Error Handling Middleware 的撰寫方式

Error Handling Middleware 的函式簽名**固定為四個參數**：

```typescript
// middleware/error-handler.ts
import { Request, Response, NextFunction } from 'express';

export const errorHandler = (
  err: Error,
  req: Request,
  res: Response,
  next: NextFunction
) => {
  console.log('something went wrong', err);
  res.status(400).send({ message: err.message });
};
```

**關鍵規則**：Express 透過參數數量來判斷這是不是 Error Handler——**四個參數就是 Error Handler**，少一個都不行。

### 掛載 Error Handling Middleware

在 `index.ts` 中，Error Handler 必須掛在最末端，確保它能捕獲所有路由的錯誤：

```typescript
// ... 所有 app.use(router) 都必須在這行之前 ...
app.use(errorHandler);
```

### 在 Route Handler 中主動 Throw

現在當錯誤發生時，我們應該用 `throw` 而非手動 `res.send()` 來處理：

```typescript
// ❌ 舊方式：手動回應
if (!errors.isEmpty()) {
  res.status(400).send(errors.array());
}

// ✅ 新方式：throw，讓 Error Handler 統一處理
if (!errors.isEmpty()) {
  throw new Error('Invalid email or password');
}
```

如此一來，無論錯誤來自 Validation 失敗、資料庫連線失敗，還是其他任何環節，**都會進入同一個 Error Handling Middleware**，確保回應格式一致。

---

## 💡 重點摘要

- **Express 的 Error Handling Middleware 透過「四個參數」自動識別；任何帶有四個參數的中間件都是 Error Handler。**
- **同步 Handler 中的 `throw` 會自動被 Express 捕獲；async Handler 必須手動呼叫 `next(error)`。**
- **所有錯誤統一走 Error Handler，可確保錯誤回應格式的一致性。**
- **Route Handler 中應優先使用 `throw` 而非手動 `res.send()`，將回應邏輯交給 Middleware 處理。**
