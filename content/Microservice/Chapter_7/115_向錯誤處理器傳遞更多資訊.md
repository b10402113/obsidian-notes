# 115_向錯誤處理器傳遞更多資訊

## 📝 課程概述

本單元處理一個關鍵問題：Error Handling Middleware 目前只知道 `err.message`（一個字串），但我們需要傳遞更豐富的錯誤資訊（例如 express-validator 的完整錯誤陣列）。由於 TypeScript 不允許對 Error 物件任意新增屬性，我們需要一個更結構化的方式來擴充錯誤資訊。

---

## 核心觀念與實作解析

### 為什麼不能直接給 Error 加上自訂屬性？

在純 JavaScript 中，我們可以這樣做：

```javascript
// ✅ JavaScript 可以這樣做（不推薦）
const err = new Error('Invalid email or password');
err.reasons = errors.array(); // 動態附加屬性
throw err;
```

但在 TypeScript 中，`Error` 型別**沒有 `reasons` 屬性**，編譯器會直接噴錯：

```typescript
// ❌ TypeScript 編譯錯誤
const err = new Error('Invalid email or password');
err.reasons = errors.array(); // Property 'reasons' does not exist on type 'Error'
```

> **核心矛盾**：Error 物件本身的型別定義太過封閉，我們無法直接附加有意義的結構化資料。

### 解決方案：Subclass（繼承）Error

當我們想要「像 Error 一樣」但能附加自訂屬性時，標準做法是**建立 Error 的子類別（Subclass）**：

```typescript
// errors/request-validation-error.ts
import { ValidationError } from 'express-validator';

export class RequestValidationError extends Error {
  constructor(public errors: ValidationError[]) {
    super('Invalid request parameters');
    // 必須加這行，確保 instanceof 判斷正確
    Object.setPrototypeOf(this, RequestValidationError.prototype);
  }
}
```

這裡的 `public errors` 是 TypeScript 的 **Parameter Property** 語法，相當於：

```typescript
this.errors = errors; // 同時宣告並賦值
```

### 子類別的具體優勢

透過 Subclassing，我們可以為不同類型的錯誤附加不同的自訂屬性：

| 錯誤類別 | 自訂屬性 | 用途 |
| --- | --- | --- |
| `RequestValidationError` | `errors: ValidationError[]` | 儲存 express-validator 的完整驗證失敗資訊 |
| `DatabaseConnectionError` | `reason: string` | 儲存資料庫連線失敗的原因描述 |

### Error Handling Middleware 中的 instanceof 判斷

有了具體的子類別，Middlewar 就能透過 `instanceof` 判斷錯誤類型並取出對應的自訂屬性：

```typescript
if (err instanceof RequestValidationError) {
  const formattedErrors = err.errors.map(e => ({
    message: e.msg,
    field: e.param,
  }));
  return res.status(400).send({ errors: formattedErrors });
}

if (err instanceof DatabaseConnectionError) {
  return res.status(500).send({
    errors: [{ message: err.reason }],
  });
}
```

### 為什麼這麼設計？

> **核心邏輯**：Error Handling Middleware 不需要知道每種錯誤的實作細節。它只需要會「問」錯誤本身：「你應該怎麼被格式化？」——這個「回答」的能力就由每個子類別自己提供。

這就是物件導向的「**Tell, Don't Ask**」原則的體現：不要 Middleware 自己去拆解錯誤，而是讓錯誤自己知道如何呈現自己。

---

## 💡 重點摘要

- **TypeScript 的 Error 型別無法直接附加自訂屬性，必須透過 Subclassing 來擴充。**
- **`public` 關鍵字放在 constructor 參數前，可同時完成屬性宣告與賦值（Parameter Property）。**
- **`Object.setPrototypeOf(this, ClassName.prototype)` 是在 TypeScript 中繼承內建類（如 Error）時的必要步驟。**
- **`instanceof` 讓 Middleware 可以依錯誤類型做出不同的處理邏輯。**
