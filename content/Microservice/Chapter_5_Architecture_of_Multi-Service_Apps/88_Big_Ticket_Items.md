# 回顧第一個應用程式的核心課題：痛點與解決方向

## 📝 課程概述

本單元作為第一個應用程式與第二個更大型專案之間的橋樑，回顧了我們在第一個 microservice 應用中遭遇的主要挑戰，並預告即將在下一個專案中採用的根本性改進方向。這些「痛點」並非微不足道，而是會隨著服務數量與資料模型複雜度增加而急遽惡化的系統性問題，提早正視它們是必要的。

## 核心觀念與實作解析

### 資料同步是最大的挑戰

在第一個應用程式中，我們面臨的最核心問題就是**跨服務的資料共享**。當資料模型變得複雜時，這個問題只會越來越嚴重。在有多個服務彼此相依的架構下，如何確保資料的一致性與正確性，是建構 microservice 時最困難、也最關鍵的課題。

### Sync 與 Async 溝通方式的取捨

我們學到了服務之間分享資料有兩種主要方式：**同步（Sync）** 與**非同步（Async）** 溝通。這兩者之間存在一個連續的光譜——從完全同步到完全非同步，甚至可以是兩者的混合。

講師選擇在課程中介重點介紹 **Async 溝通**，而非 Sync。原因是：同步溝通在實作上**非常容易**，但缺點是會在 production 環境中帶來耦合與可用性問題。選擇非同步溝通，讓各服務透過事件匯流排（Event Bus）以事件（Event）來傳遞變化，是更困難但更正確的做法。

> 在非同步架構下，當某個服務發出一個事件，Event Bus 負責將這個事件廣播給所有需要知道它的服務。這種模式讓每個服務都能高度自給自足（self-sufficient），因為服務之間**沒有嚴格的依賴關係**——即便某個服務突然故障，其他服務仍能繼續運作。

### Docker 與 Kubernetes 的價值

Docker 讓包裝與部署服務變得簡單；而 Kubernetes 雖然在初期設定時比較繁瑣（要理解 Pod、Deployment、Service 等基本概念），但它是管理與擴展 microservice 的正確方向。**困難的路徑是值得的**——學會 Kubernetes，長期來看會大幅簡化部署與擴展工作。

### 第一個應用程式的四大痛點

講師明確指出第一個應用程式中存在四個必須解決的「痛點」，這些痛點在下一個專案中會用截然不同的方式來處理。

#### 痛點一：重複性程式碼過多

在每個 service 中，我們寫了大量結構相似的程式碼——相同的 Express 設定、相同的根路由處理器、查詢服務與文章服務之間共享資料的邏輯也是重複的。

**解決方案**：建立一個 **共享 library（中央 library）**，將所有服務都會用到的 Express 相關程式碼、認證 middleware 等全部集中在一個可共用的模組中，各服務統一 `npm install` 這個 library，大幅提升程式碼重複利用率。

#### 痛點二：難以想像事件流向

當有多個服務、多种事件在系統中流動時，光靠腦補很難掌握事件何時發送、流向哪些服務、以及觸發了什麼後續行為。即使是講師自己，在整理影片素材時也發現：「等一下，這個事件是怎麼運作的？」

**解決方案**：在共享 library 中**明確定義所有事件**，包含事件名稱、屬性名稱、資料型別等。這讓整個團隊能清楚知道有哪些事件、各事件的精確結構，以及事件之間的時序關係。

#### 痛點三：事件屬性難以記憶、易出錯

由於事件定義散落在多個專案之間，跨服務傳遞事件時很容易發生**屬性名稱拼寫錯誤**——例如在某個 service emit 的事件中使用 `username`，但在接收事件的 service 中卻錯誤地使用了 `name`。這樣的小錯誤在大型系統中難以察覺，卻會導致資料不一致。

**解決方案**：全面採用 **TypeScript** 來撰寫整個專案。TypeScript 的強型別系統能自動檢查出屬性名稱不符、型別錯誤等問題，將這類錯誤在編譯時期就攔截下來，徹底解決「憑記憶拼寫屬性」的困擾。

#### 痛點四：事件流程測試繁瑣

以我們的言論審核（Moderation）功能為例，測試一個完整的事件流程需要：先建立文章 → 建立留言 → 重新整理頁面 → 再針對不同的邊界情況（留言含 Orange、有 Orange 但大小寫不同、Orange 拼寫錯誤等）重複上述步驟。這樣的手動測試不僅繁瑣，隨著服務數量增加，根本不可能完整覆蓋所有情境。

**解決方案**：開始為關鍵功能撰寫**自動化測試**，尤其是與資料處理和 Express 邏輯相關的部分。透過測試，我們可以確保事件流程的正確性，而不必每次都手動在瀏覽器中操作。

### 雲端開發叢集：解決本機效能瓶頸

如果你的機器在執行 Kubernetes、Skaffold 加上瀏覽器等工具時開始變得 Lag，在正式的公司環境中更是如此——可能同時有數十個服務在運行。這時候**在雲端建立一個開發用的 Kubernetes 叢集**就是解決方案。這個叢集不是用於 production，而是專門用來在大型專案中進行開發與測試。

> 需要留意的是，雲端開發叢集可能需要付費，請自行評估是否有免費額度可用。

### 事件順序與資料一致性的深層風險

這是一個在第一個應用程式中**從未被正視**的問題：如果使用者同時（幾乎在同一微秒）建立一篇文章以及附加在該文章上的留言，系統有沒有可能先收到「留言建立事件」再收到「文章建立事件」？

答案是**有可能**。我們的 Comment Service 在收到留言建立事件時，預設文章已經存在於記憶體資料結構中。若文章尚未建立，Comment Service 就無法正確將留言與文章關聯，整個系統就會斷裂。這種**事件順序不確定（out-of-order events）** 的情境，是接下來專案要大力關注的核心課題。

### 解決方案的總覽

| 痛點 | 解決方案 |
|------|----------|
| 重複程式碼 | 建立共享 library |
| 事件流向難以掌握 | 在 library 中明確定義所有事件結構 |
| 屬性名稱易錯 | 全面採用 TypeScript |
| 手動測試繁瑣 | 撰寫自動化測試 |
| 本機效能瓶頸 | 使用雲端開發 Kubernetes 叢集 |
| 事件順序不一致 | 在下一個應用程式中系統性處理資料一致性 |

## 💡 重點摘要

- 非同步（Async）溝通雖然比同步更難實作，但能讓服務之間保持低耦合，提升整體系統的可用性。
- 跨服務共享程式碼與事件定義的最佳實踐，是建立一個集中管理的 **shared library**。
- TypeScript 的強型別檢查是防止跨服務屬性名稱不一致的利器，**必須從專案一開始就使用**。
- 雲端 Kubernetes 開發叢集能解決本機無法負荷大量服務的效能問題，是大型 microservice 專案必備的開發利器。
- 系統必須假設事件可能以**任意順序**抵達，並據此設計處理資料的邏輯。
