# 測試架構與檔案重構

## 📝 課程概述

要讓測試工具正常運作，我們需要先對現有架構做一個關鍵的重構：將 `index.ts` 拆分成 `app.ts` 與 `index.ts` 兩個檔案。這是因為 `index.ts` 內含自動監聽 Port 3000 的啟動邏輯，若在測試環境中被重複引入，會導致多個服務同時搶佔同一個連接埠。本單元說明重構背後的原因與實際操作方式。

## 核心觀念與實作解析

### 為何需要重構？

讓我們回到問題的根源。`index.ts` 目前做了兩件事：

1. **建立並設定 Express app**（包含所有 middleware 與 route handler）
2. **啟動伺服器**（`app.listen(3000, ...)`）

當我們引入 SuperTest 來測試 Express app 時，**SuperTest 需要直接存取 Express app 物件本身**，而非透過啟動伺服器的方式。問題在於：如果 `index.ts` 自動監聽 Port 3000，那麼當不同服務的測試同時執行時，它們全部都會嘗試監聽同一個 Port，馬上就會發生連接埠衝突的錯誤。

### SuperTest 的 Ephemeral Port 機制

SuperTest 提供了一個很棒的功能：**若 Express app 目前沒有在監聽任何連接埠，它會自動挑選一個閒置的臨時連接埠（Ephemeral Port）來使用**。這讓我們可以同時執行多個服務的測試，而不需要擔心連接埠衝突。

要啟用這個功能，前提條件是：**Express app 本身不能已經在監聽連接埠**。所以我們必須把「建立 app」和「啟動 app」這兩個職責分開。

### 重構實作步驟

**步驟一：建立 `app.ts`**

將原本 `index.ts` 中所有與「建立並設定 app」相關的程式碼（middleware、route handler、Cookie Session 設定等）移動到新的 `app.ts`，**但不要包含 `app.listen()` 的啟動邏輯**。接著把 app 匯出：

```typescript
// app.ts
export { app };
```

**步驟二：修改 `index.ts`**

讓 `index.ts` 變成純粹的「啟動腳本」：

```typescript
// index.ts
import { app } from './app';
import { mongoose } from './nats-wrapper';

mongoose.connect(process.env.MONGO_URI!);

app.listen(3000, () => {
  console.log('Auth service listening on port 3000!');
});
```

**步驟三：在測試檔案中引入 app**

```typescript
// __test__/signup.test.ts
import request from 'supertest';
import { app } from '../../app';
```

### 這個重構的額外價值

這個重構不只是為了測試，它實際上讓我們的應用程式結構更加清晰：

- **`app.ts`**：純粹描述「如何設定這個應用程式」，不關心啟動細節
- **`index.ts`**：純粹描述「如何啟動這個應用程式」，包含資料庫連線等環境設定

未來在其他微服務中，這個模式會帶來更大的幫助——測試、生產環境、開發環境可以共享同一個 app 設定，卻使用不同的啟動方式。

> ⚠️ **這裡有一個易錯點**：在移動程式碼時，千萬不要把 `mongoose` 的 import 也移動到 `app.ts`。mongoose 是用來連線資料庫的，屬於「啟動」邏輯，應該留在 `index.ts`。

## 💡 重點摘要

- `index.ts` 同時承擔「建立 app」與「啟動伺服器」兩種職責，必須拆分才能支援測試。
- 把 app 匯出成獨立檔案後，SuperTest 才能直接對其發送請求，並利用 **Ephemeral Port** 避免連接埠衝突。
- 這個重構讓應用程式的職責更加分離：**設定（app.ts）與啟動（index.ts）分開**，是好設計的體現。
