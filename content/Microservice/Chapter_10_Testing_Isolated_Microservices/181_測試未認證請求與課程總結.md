# 測試未認證請求與課程總結

## 📝 課程概述

本節是 Auth Service 測試章節的收尾。我們為 `GET /api/users/currentuser` 補上最後一個測試案例：當未攜帶任何 Cookie 發送請求時，`currentUser` 應為 `null`。這個測試確認了系統在未認證狀態下的正確行為，同時也順帶回顧了本章涵蓋的所有測試策略與實作要點。

## 核心觀念與實作解析

### 測試未認證請求

```typescript
it('responds with NULL if not authenticated', async () => {
  const response = await request(app)
    .get('/api/users/currentuser')
    .send({});

  expect(response.body.currentUser).toBeNull();
});
```

這個測試**不呼叫 `global.signin()`**，直接發送請求。因為沒有 Cookie，伺服器無法驗證身份，`currentUserRouter` 中的 `req.session = { currentUser: null }` 會是預設值。

### 完整測試覆蓋回顧

整個 Auth Service 的測試涵蓋了以下 Handler 與場景：

| Handler | 測試場景 |
|---|---|
| `signup.ts` | 成功註冊（201）、無效 Email（400）、無效密碼（400）、缺少欄位（400）、重複 Email（400）、Cookie 設定 |
| `signin.ts` | 不存在帳號（400）、錯誤密碼（400）、正確憑證登入（200）+ Cookie |
| `signout.ts` | Cookie 清除行為 |
| `currentUser.ts` | 已認證請求（取得正確 email）、未認證請求（currentUser 為 null） |

### 本章未涵蓋的測試類型

**Model 測試**：User Model 目前很簡單，沒有太多商業邏輯需要測試。未來複雜的 Model（如 Order、Ticket）會需要單獨的 Unit Test。

**Event 測試**：Auth Service 目前尚未與 Event Bus 連接，所以沒有 Event 發送與接收的測試。其他 Service 會需要這類測試來模擬跨服務溝通。

### 測試的最大價值：防止 Regression

手動用 Postman 測試過的功能，只要別人（或自己）改了一行 code，馬上就可能出現 Bug。自動化測試的目的不是「測一次」，而是「每次變動都能自動確認舊功能還沒壞」——這才是測試真正的價值所在。

## 💡 重點摘要

- 未認證請求回傳 `currentUser: null` 是預期行為，反映了 Session 的預設狀態
- 所有 Handler 的錯誤路徑（400 錯誤）都值得寫測試，因為這些路徑在 UI 上很難手動觸發
- Auth Service 的 Model 與 Event 測試留待未來服務擴展時再補上
- 測試的終極目標是：**每次 Code 變動，自動確認沒有破壞既有功能**
