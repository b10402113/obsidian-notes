# 微服務測試的範疇與策略

## 📝 課程概述

在進入實際寫測試之前，我們需要先理解：在微服務架構下，「測試的範疇（Scope）」是個至關重要的決策。本單元說明為何我們選擇對各個服務進行隔離測試，而非模擬多服務之間的互動，並預告我們將聚焦的三種測試類型。

## 核心觀念與實作解析

### 測試範疇的光譜

在微服務環境中，測試並不是一個簡單的命題。我們可以選擇測試一個非常小的範圍，也可以選擇測試一個非常龐大的範圍。從小到大排列如下：

1. **單一程式碼片段（Unit）**：例如只測試一個 middleware 是否正確運作。這是最傳統的單元測試（Unit Test）。
2. **多個片段協作（Integration）**：例如模擬一個請求流經多個 middleware，最後抵達 route handler。
3. **元件協作（Component）**：測試我們的服務如何與外部相依項目互動，例如與 MongoDB 或 Event Bus 的互動。
4. **多服務協作（End-to-End）**：同時啟動多個服務（例如 Order Service 與 Ticketing Service），讓它們透過事件彼此溝通，驗證整體行為。

### 為何不選擇多服務測試？

表面上看，測試「所有服務一起運作」最能模擬真實情況。然而在微服務架構下，這樣做的代價極高：

- 需要架設測試用的 Kubernetes 叢集
- 需要設法對這些服務發送請求並斷言結果
- 環境建置複雜、成本高昂

因此，**我們選擇對每個服務進行隔離測試（Isolated Testing）**。

### 三種測試目標

對於 Ticketing 這個專案，我們將聚焦於三種測試：

| 測試類型 | 描述 | 適用範圍 |
|---|---|---|
| **請求處理測試（Request Handling）** | 對 route handler 發送請求，斷言回應狀態碼與資料是否正確 | Auth Service ✅ |
| **模型測試（Model）** | 針對資料模型的商業邏輯進行斷言 | 較複雜的模型適用 |
| **事件處理測試（Event Handling）** | 驗證服務能否正確接收與發出事件 | 未來其他服務適用 |

### Auth Service 的測試範圍

對於 Auth Service 本身，目前尚未連接事件系統，User Model 也相對簡單，因此**目前只專注於請求處理測試**。

### 測試的執行方式

我們將在終端機直接執行 `npm run test`，啟動本機的 Node.js 伺服器與 MongoDB 來跑測試。這意味著：

- **本地環境必須能完整運行每個服務**（目前只需要 Node.js 與 MongoDB）
- 未來若服務依賴更複雜的環境，可能需要更進階的測試策略

> 💡 **這裡之所以採用本地隔離測試**，是因為它的設定成本最低，且足以驗證每個服務的核心邏輯是否正確。隨著專案規模擴大，我們會逐步引入更複雜的測試基礎設施。

## 💡 重點摘要

- 微服務測試的核心挑戰是**選擇正確的測試範疇**，而非一味追求大範圍的整合測試。
- **多服務同時啟動測試**雖然模擬真實場景，但環境建置成本過高，實務上並不推薦。
- 我們將對每個服務進行**隔離測試**，確保單一服務的正確性，再透過事件機制模擬服務間的協作。
- Auth Service 目前主要測試 **route handler 的請求處理邏輯**，其他類型的測試將在相關功能實作後陸續加入。
- 測試將直接在本機執行（`npm run test`），不需要 Docker 或 Kubernetes。
