# 微服務測試的範圍與策略

## 📝 課程概述

本章節開啟了 Auth Service 的自動化測試實戰。在進入程式碼之前，我們先宏觀地探討：在微服務架構下，「測試的範圍」是首要必須思考的問題。我們會了解為何不該執著於跨服務整合測試，而是採用隔離（Isolated）的方式逐一測試每個服務。

## 核心觀念與實作解析

### 測試範圍的光譜

在微服務環境中，測試並非只有「有沒有寫」那麼單純，而是要問：**我們到底要測多遠？** 以下由小到大依序說明：

| 範圍層級 | 描述 | 範例 |
|---|---|---|
| **單一程式碼** | 只測試一個函式或 Middleware 的行為 | 測試 `require-auth` Middleware |
| **多個程式碼協作** | 模擬一個 Request 流經多個 Middleware 到達 Handler | Request 經過 `require-auth` → 到 `signup` Handler |
| **跨元件協作** | 測試 Express App 與外部資源（MongoDB、Event Bus）的互動 | 確認 Sign Up 後資料有寫入 MongoDB |
| **跨服務協作** | 同時啟動多個服務，測試它們之間的事件來往 | 同時啟動 Order Service 與 Ticketing Service |

### 為何不直接做跨服務測試？

直覺上我們會說：「直接測所有服務一起跑，最接近真實環境。」但實務上，這在微服務架構中非常困難：

- 需要一個可快速啟動的測試環境（例如測試用 Kubernetes Cluster）
- 多個 Service 可能同時監聽同一個 Port（衝突問題）
- 測試的建置與維護成本極高

所以我們的策略是：**盡量將每個服務隔離測試，用事件收發（Event Emitting / Receiving）來模擬跨服務協作**。

### 三種測試目標

針對 Auth Service，我們將聚焦以下三種測試：

1. **Request Handling 測試**：發送 HTTP Request，驗證 Response 與資料庫寫入是否正確
2. **Model 測試**：針對特定 Model（如 User Model）的商業邏輯撰寫 Unit Test
3. **Event 測試**：驗證 Service 接收與發出事件的處理流程

目前 Auth Service 尚未連接 Event Bus，所以先專注第一類。

### 測試執行方式：本地直接跑

我們會在終端機執行 `npm run test`，直接在本機啟動 Node.js + MongoDB 來跑測試，**不走 Docker 也不走 Kubernetes**。這對 Auth Service 來說完全可行，因為它只需要 Node.js 與 MongoDB 兩個依賴。但若未來服務更複雜，則需要更彈性的測試策略。

## 💡 重點摘要

- 微服務測試的關鍵問題是「範圍有多大」—— 並非測越大越好
- 跨服務整合測試在實務上建置成本極高，應優先採用隔離測試
- 單一 Service 的測試只需要本機能啟動的依賴即可
- Auth Service 的主要測試目標是 Request Handling 行為
- `npm run test` 會直接在本機啟動測試環境，不需要 Docker
