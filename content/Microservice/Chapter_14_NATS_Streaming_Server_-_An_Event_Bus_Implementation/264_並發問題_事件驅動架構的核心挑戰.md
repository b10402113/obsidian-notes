# 並發問題：事件驅動架構的核心挑戰

## 📝 課程概述

本單元是 Chapter 14 中**最關鍵的一集**。老師透過一個銀行帳戶系統的案例，展示在事件驅動的非同步架構下，可能導致資料不一致甚至業務邏輯錯誤的**四種典型並發問題**。理解這些問題，是學會後續所有解決方案的起點——這些問題不是 NATS Streaming 特有的，而是**所有事件匯流排實作都會面臨的挑戰**。

---

## 核心觀念與實作解析

### 情境設定：銀行帳戶系統

```
Publisher 發布事件：
  - account:deposit     → 存款
  - account:withdraw   → 提款
  (不能讓帳戶餘額 < 0，否則為 Critical Error)
```

假設存款：$70 + $40，然後提款 $100，正常結果應該是餘額 = $10。

---

### 問題一：事件處理失敗，後續事件搶先執行

```
步驟 1: deposit $70 → 發送給 Service A → 處理失敗（檔案鎖住、資料庫錯誤等）
步驟 2: deposit $40 → NATS 等待 30 秒後重送給 Service B → 成功（餘額 = $40）
步驟 3: withdraw $100 → 發送給 Service B（很快，已經在線上）→ 餘額 $40，提款 $100 → Critical Error！
```

**問題本質**：第一筆事件處理失敗，但第二、三筆事件搶先完成，導致提款事件在存款生效前就執行了。

---

### 問題二：各實例處理速度差異

```
步驟 1: deposit $70 → 發給 Service A（A 負擔很重，事件排隊中）
步驟 2: deposit $40 → 發給 Service A（仍在排隊）
步驟 3: withdraw $100 → 發給 Service B（B 很空，立即處理）→ Service B 的餘額仍是 $0 → Critical Error！
```

**問題本質**：即使沒有錯誤，多個實例之間的處理速度差異也可能導致事件亂序執行。

---

### 問題三：Client 離線但 Server 尚未偵測到

```
步驟 1: deposit $70 → Server 以為 Service A 還活著（事實上已 crash）
步驟 2: deposit $40 → 同上
步驟 3: withdraw $100 → 發給 Service B → 餘額 $0 → Critical Error！
（等 Service A 重連後，存款事件才會重新發送，但那已經太晚了）
```

**問題本質**：graceful shutdown 機制失效時，Server 仍然「相信」離線的實例，造成事件在空窗期內被發送到錯誤的實例。

---

### 問題四：網路延遲與事件重送的交錯

```
步驟 1: deposit $70 → Service A 開始處理（硬碟很慢，需要 29.99 秒）
步驟 2: 29.99 秒後 → Service A 剛讀出 $0，準備寫入 $70
步驟 3: 0.01 秒後（30 秒到了）→ NATS 判定 Service A 失敗，重送事件給 Service B
步驟 4: Service B 立即寫入 $70 → 檔案已鎖（由 Service A 持有）→ B 也失敗
步驟 5: 又 30 秒後 → 重送給 Service A（此時 A 終於寫入成功，餘額 = $70）
步驟 6: 重送給 Service B → B 現在可以寫入，覆蓋為 $70
步驟 7: withdraw $100 → 餘額 = $70 → 扣款後 = -$30 → Critical Error！
```

> 這個案例聽起來極端的，但它描述的正是**網路延遲 + 30 秒超時 + 硬碟鎖定**三個條件同時滿足時，**幾乎必然會發生的真實場景**。

---

### 為什麼這是所有事件匯流排的共同問題？

這些問題**並非 NATS Streaming 獨有**。任何支援非同步訊息傳遞 + 自動重送的系統，都會面臨同樣的挑戰：

- 事件處理失敗 → 需要重送
- 重送需要等待超時 → 時間視窗內其他事件搶先執行
- 多實例同時運行 → 同一事件只發給其中一個
- 網路延遲 + 處理時間差異 → 事件實際處理順序與發布順序不符

---

## 💡 重點摘要

- **事件處理失敗時，NATS 會在 30 秒後自動重送，期間其他事件可能已經執行完畢，導致業務邏輯錯誤。**
- **多個實例處理速度不同，fast instance 可能搶先處理依賴尚未生效的事件。**
- **Client 離線後，Server 需要等待心跳失敗才能移除該 Subscription，期間事件會發送到「已死亡」的實例。**
- **以上所有問題都是「事件重送機制」帶來的固有代價，是所有事件匯流排實作都需要面對的挑戰，而非 NATS Streaming 的 Bug。**
