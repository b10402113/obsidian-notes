# 發布與訂閱事件

## 📝 課程概述

本單元實作 Publisher 與 Listener 的完整程式碼。我們會學習如何在 NATS Streaming 中**發布事件（Publish）**到指定的 Channel，以及如何**訂閱事件（Subscribe）**並接收訊息。課程中也說明了 Subject、Channel、Subscription 這三個核心術語的具體含義與它們之間的關係。

---

## 核心觀念與實作解析

### 核心術語對照

| 術語 | 在程式碼中的意義 |
|------|----------------|
| **Subject（主體）** | 發布事件時指定的 Channel 名稱 |
| **Channel** | 儲存在 NATS Streaming Server 端的虛擬管道，以 Subject 名稱為識別 |
| **Subscription（訂閱）** | Client 端建立的監聽器，負責接收並處理來自某個 Channel 的訊息 |

### 發布事件（Publisher）

```typescript
import nats from 'node-nats-streaming';

const stan = nats.connect('ticketing', 'abc', {
  url: 'http://localhost:4222'
});

stan.on('connect', () => {
  const data = {
    id: '123',
    title: 'concert',
    price: 20
  };

  // 所有資料必須先轉成 JSON 字串
  stan.publish('ticket:created', JSON.stringify(data), () => {
    console.log('Event published');
  });
});
```

**`publish` 的三個參數：**

1. **Subject** — 要發布到的 Channel 名稱（這裡是 `ticket:created`）
2. **Data** — 要傳遞的資料，**必須是字串**（原生 NATS 不支援傳送 JS 物件）
3. **Callback** — 可選，發布完成後被呼叫

> **重要限制：NATS 原生只支援字串傳輸。** 如果要傳送 JS 物件，必須先 `JSON.stringify()`，接收端再 `JSON.parse()`。

### 訂閱事件（Listener）

```typescript
import nats, { Message } from 'node-nats-streaming';

const stan = nats.connect('ticketing', '123', {
  url: 'http://localhost:4222'
});

stan.on('connect', () => {
  const subscription = stan.subscribe('ticket:created');

  subscription.on('message', (msg: Message) => {
    console.log('Message received');
  });
});
```

**`subscribe` 的設計邏輯（重要觀念）：**

多數 Library 接受 Callback 作為第二參數——收到訊息時直接呼叫。但 `node-nats-streaming` 的做法不同：

1. **先建立 Subscription 物件**
2. **再在 Subscription 上監聽 `message` 事件**

這是一個明確的**事件發送器（Event Emitter）**模式，與 Node.js 內建的 `EventEmitter` 概念相同。

---

### 如果沒有 Listener，發布會發生什麼？

如果目前沒有任何 Service 訂閱某個 Channel，直接發布事件不會導致錯誤。Server 會執行發布動作，但訊息直接進入虛空中，沒有人收到。這在初期開發階段很正常，不是 Bug。

---

### Client ID 衝突問題

嘗試用**相同的 Client ID** 連線兩次，會收到錯誤：

```
Client ID already registered
```

這是因為 NATS Streaming Server 不允許重複的 Client ID。如果要讓多個程式實例同時連線，**每個實例都必須有不同的 Client ID**。

在 Kubernetes 環境中，通常由系統動態生成唯一的 Client ID。但在本地測試時，可以用 `crypto.randomBytes()` 產生隨機 ID：

```typescript
import { randomBytes } from 'crypto';

const clientId = randomBytes(4).toString('hex'); // 例如 'abc1234f'
```

---

## 💡 重點摘要

- **`publish` 只能傳送字串，物件必須先 `JSON.stringify()`；接收端用 `JSON.parse()` 還原。**
- **`subscribe` 回傳的是 Subscription 物件，訊息處理發生在 Subscription 的 `message` 事件上，而非 `subscribe` 的 Callback 中。**
- **Client ID 必須全局唯一，衝突時 Server 會拒絕連線。**
- **沒有人訂閱的 Channel 不會導致錯誤，訊息只是「消失了」。**
