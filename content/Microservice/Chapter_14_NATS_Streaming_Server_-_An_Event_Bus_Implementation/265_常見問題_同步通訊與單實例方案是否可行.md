# 常見問題：同步通訊與單實例方案是否可行？

## 📝 課程概述

本單元回答三個在理解並發問題後必然會浮現的疑問：**同步通訊是否就沒有這些問題？只跑一個實例是否可以避免競爭？手動列舉所有並發情境是否可行？** 老師用具體的分析告訴我們，這三個直覺上的解決方案都有各自的根本性缺陷。

---

## 核心觀念與實作解析

### 問題一：改用同步通訊是否就解決了？

**並不是。** 即使回到同步的 HTTP 請求架構，如果背後有多個 Server 實例，同樣的問題仍然會發生：

```
Load Balancer 隨機分配請求：
  Request 1 (deposit $70)  →  Instance A（很忙，排隊中）
  Request 2 (deposit $40)  →  Instance B（很空，立即處理）
  Request 3 (withdraw $100) →  Instance B（又很快）→ 餘額 $0 → Critical Error！
```

這說明：**問題的根源不是「事件匯流排」或「HTTP」，而是「有多個處理單元同時運行」**。事件驅動架構只是讓這些問題更凸顯罷了。

---

### 問題二：只跑一個實例是否可以？

**不行。** 從兩個層面來看：

**層面一：仍可能發生失敗與重送**

即使只有一個實例，事件處理仍然可能失敗（NAT 斷線、硬碟鎖住等），Server 仍會在 30 秒後重送。問題仍然存在。

**層面二：單一實例是無法擴展的瓶頸**

> 單一實例是系統可用性的巨大風險，而且完全犧牲了水平擴展的能力。任何 production 級的系統都必須假設會同時運行多個實例。

---

### 問題三：列舉並處理所有可能的並發情境？

**不可能。** 潛在的並發問題數量趨近無限，無法窮舉所有可能組合。

> **工程判斷的關鍵問題**：這個問題**真的需要解決嗎？** 如果你是打造 Twitter 系統，兩條推文順序顛倒或重複出現，用戶能接受嗎？如果是一個銀行系統，餘額計算錯誤能接受嗎？
>
> **答案通常是「在合理範圍內不需要處理」。** 過度工程只會拖慢開發速度，而不會帶來對應的價值。

---

## 💡 重點摘要

- **同步通訊同樣會面臨並發問題，因為 Load Balancer 同樣會將請求隨機分配到不同實例。**
- **單一實例是系統可用性的巨大風險，完全放棄了水平擴展能力，生產環境不可接受。**
- **不可能列舉並處理所有並發情境，需要根據業務需求做出工程判斷，取捨成本與風險。**
- **非同步事件驅動不會讓並發問題消失，只是讓它們變得更明顯、更需要刻意處理。**
