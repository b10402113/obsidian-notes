# 更多並發解決方案及其局限性

## 📝 課程概述

本單元展示三種看似可行但各有致命缺陷的並發控制方案：**全局序列號檢查**、**按用戶分組序列號**（需要大量 Channel）、以及**Publisher 端自行追蹤序列號**。每一個方案都會被指出為什麼在實作上最終行不通，以及背後的根本原因。

---

## 核心觀念與實作解析

### 方案一：全局序列號檢查

**想法**：所有事件共享一個序列號池。每個 Service 收到事件時，檢查上一個序列號是否已被處理過。

```
Event #1 → Service A 處理 → 標記已處理
Event #2 → Service B 處理 → 檢查：#1 已在清單中 ✅ → 處理
Event #3 (withdraw) → Service A 處理 → 檢查：#2 已在清單中 ✅ → 處理
```

**問題**：所有帳戶的交易事件必須嚴格依序處理，**不同用戶的事件也會互相阻塞**。

> 例如：Jim 的存款和 Mary 的存款完全無關，但因為要等待前一序號完成，Mary 的存款必須等 Jim 的存款處理完才能開始。當系統有數十萬個帳戶時，這會成為災難性的效能瓶頸。

---

### 方案二：按用戶分組序列號

**想法**：每個用戶有自己獨立的序列號池，透過為每個用戶建立**獨立的 Channel** 來實現。

```
Channel: account-deposit-jim
Channel: account-withdraw-jim
Channel: account-deposit-mary
Channel: account-withdraw-mary
```

**問題**：

- **NATS Streaming 預設只允許 1000 個 Channel**，大量用戶會瞬間超出限制
- 創建過多 Channel 會增加 Server 的處理開銷

**結論**：對使用者數量龐大的系統，這個方案在技術上不可行。

---

### 方案三：Publisher 端自行追蹤序列號

**想法**：Publisher 在發送事件時，攜帶「上一個影響該用戶的事件之序列號」。

```
Publisher 發送 Event #3 (withdraw $100 for Jim)
  攜帶資訊：上一次修改 Jim 的事件序列號 = #1

Consumer 收到 Event #3：
  檢查：我最後處理過的 Jim 相關事件序列號是 #1 嗎？
  是 → OK，處理這個提款
  否 → 拒絕，等待序列號 #2 的事件先抵達
```

**問題**：Publisher **無法得知事件被 NATS Server 分配到的實際序列號**。

```
NATS 的 Publish 流程：
  Publisher ──publish──▶ NATS Server ──分配序列號──▶ Consumer
       │                                           ▲
       │ (Publisher 端沒有收到回覆，不知道 # 是多少)
       ▼
   不知道！
```

Publish 是一個**單向操作**，Server 不會把分配的序列號回傳給 Publisher。因此 Publisher 無法在事件中嵌入「上一個序列號」的資訊。

---

## 💡 重點摘要

- **全局序列號檢查讓所有不相關的事件被迫依序處理，造成災難性的效能瓶頸。**
- **按用戶建立獨立 Channel 會被 NATS Streaming 的 1000 Channel 限制與額外開銷阻止。**
- **Publisher 端追蹤序列號不可行的根本原因：Publish 是單向操作，Publisher 無法得知 Server 分配的序列號。**
- **這些方案的失敗揭示了一個更深層的問題：需要重新思考「事件來源的 Service」的設計方式。**
