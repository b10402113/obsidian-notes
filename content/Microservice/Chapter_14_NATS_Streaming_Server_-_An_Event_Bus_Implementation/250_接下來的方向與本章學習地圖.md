# 接下來的方向與本章學習地圖

## 📝 課程概述

本單元作為 Chapter 14 的起點，老師先向學員預告接下來三個可能的工作方向（ Client App、Orders Service、Event Bus），並說明為何我們選擇先深入理解 Event Bus 的內部運作機制。理解 NATS Streaming Server 的行為特性，對於後續建構 Orders Service 與 Ticket Service 的協作模式有著決定性的影響。

---

## 核心觀念與實作解析

### 本章的核心任務：理解 Event Bus

在先前的章節中，我們已經完成了 Ticket Service 的實作與測試。現在有三条路可以走：

1. **直接進入 Client App** — 加入查看、建立、編輯 Ticket 的功能
2. **先建 Orders Service** — 實作與 Ticket Service 類似的 CRUD 服務
3. **先理解 Event Bus** — 將 Event Bus 接入 Ticket Service，預先掌握其限制與特性

老師選擇**第三條路**，原因是：**Event Bus 的行為會直接影響 Orders Service 與 Ticket Service 的協作方式**。如果現在不把背後的機制弄清楚，之後在建構 Orders Service 時將不斷踩坑。

---

### 為什麼要用 NATS Streaming Server？

在第一個小型練習專案中，我們從零打造了一個**自訂 Event Bus**——它本質上就是一個跑著 Express 的小程式，用 Axios 轉發事件。這種做法極度簡化，適合教學。

但當我們的系統規模變大、自有服務數量變多時，這種簡易 Event Bus 的能力就遠遠不夠了。**NATS Streaming Server** 是建立在核心 NATS 之上的更進階實現，提供了事件持久化、訂閱管理、重送機制等豐富功能。

---

### NATS 與 NATS Streaming Server 是兩回事

> **千萬注意：我們用的是 NATS Streaming Server，不是 NATS。**

- **NATS** — 非常輕量、極簡的事件轉發核心
- **NATS Streaming Server** — 在 NATS 之上封裝了更多功能的進階版本，提供事件持久化、序列號、Client ID 管理等特性

在查文件時，請務必確認找到的是 **NATS Streaming** 的相關資料，而非純 NATS 文件。官方文件的左側導覽列中，「NATS Streaming Concepts」與「Developing with NATS Streaming」才是我們需要參考的部分。

---

### Docker Hub 上的 NATS Streaming Image

NATS Streaming Server 有官方 Docker Image（`nats-streaming`），在部署到 Kubernetes 之前，可以先在本地端透過 Docker Hub 查看其文件。最重要的是**命令列選項（Command Line Options）**——這些參數會直接控制伺服器的行為，其中有些預設值並不適合生產環境，需要我們主動調整。

## 💡 重點摘要

- **本章的核心目標是深入理解 NATS Streaming Server，而非急著實作 Client 或 Orders Service。**
- **NATS 與 NATS Streaming Server 是不同的專案，請確認文件來源正確。**
- **Event Bus 的行為特性會直接影響服務之間的協作邏輯，必須預先掌握。**
