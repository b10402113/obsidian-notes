# Vim 即時 Q&A 直播第二場

## 📝 課程概述

本單元是 Vim Master Class 的第二場即時問答直播，Jason Cannon 老師延續第一場的互動形式，深入回應了關於 Vim 未來發展、安裝 GitHub 腳本的安全性考量、Vim 8 原生插件系統、文字大小寫切換，以及 Visual Mode 與直接命令取決等實務問題，是一堂兼顧安全性、進階操作與未來方向的全方位 Q&A。

## 核心觀念與實作解析

### 安全執行 GitHub 上的腳本：該一行指令執行還是逐行檢視？

有學員提問：「我想從 GitHub Gist 安裝一個 MQ 安裝腳本，直接用一行指令下載並執行比較好，還是應該逐行確認？」

老師強調：**安全永遠是第一考量，而信任的前提是理解。**

#### 判斷流程

1. **不確定腳本內容是否安全** → 先在**虛擬機（VM）**中隔離測試，並逐行手動執行每個命令，觀察每一步的輸出與副作用
2. **已完整閱讀並理解腳本內容**，確認它只做你想讓它做的事（例如：單純執行 `yum install` 或 `apt install` 安裝軟體）→ 可以放心地寫成腳本或用一行指令執行

#### 如何從 Gist 下載並執行腳本

在瀏覽 Gist 頁面時，點擊 **Raw** 按鈕，複製該連結。Raw 版本的 Gist 就是乾淨的純文字腳本，不含任何 HTML 排版。

下載並直接執行（適用於信任的腳本）：

```bash
curl https://gist.githubusercontent.com/.../raw | bash
```

或者分步驟進行：

```bash
curl -O https://gist.githubusercontent.com/.../raw/install.sh
chmod +x install.sh
./install.sh
```

> **核心問題永遠是：① 你信任這個來源嗎？② 你希望這個操作可重複且一致嗎？如果答案都是肯定的，就把腳本保存到你完全掌控的位置。**

### Vim 的未來：Bram Moolenaar 與 Neovim 的崛起

有學員關心 Vim 是否會因為維護者人數少而走向衰落。

老師對此抱持樂觀態度，理由如下：

- **Bram Moolenaar** 是 Vim 過去 25 年的主要維護者與原始作者，目前任職於 **Google**，公司允許他將 Vim 作為 Side Project 持續開發，有 Google 的資源與他數十年的經驗作為後盾
- Vim 8 在數月前剛發布了重大更新，代表專案依然活躍
- 即使有朝一日 Vim 主專案停止維護，**Neovim** 已經是成熟的 Fork，直接繼承了 Vim 的所有命令與操作方式，並在非同步支援、內嵌 Lua 腳本等方向做出創新

> **Open Source 的核心優勢在於：任何人如果需要，都可以 fork 原始碼自行維護。Vim 的設計與命令不會輕易消失。**

### Vim 8 原生插件系統：不需要額外套件管理器

過往要管理 Vim 插件，必須安裝 Vundle、Pathogen 等第三方套件管理器。但從 **Vim 8** 開始，插件支援已**內建於 Vim 本身**，大幅降低了使用門檻。

#### 目錄結構

插件的目錄結構如下：

```
~/.vim/
└── pack/
    └── [任意名稱 plugins]/
        └── start/
            └── [插件名稱]/
                ├── autoload/
                │   └── ...
                └── plugin/
                    └── ...
```

#### 安裝流程（以 Goyo 為例）

Goyo 是一款「專注模式」插件，啟動後會隱藏所有介面元素，只保留一個居中的遊標，模擬類似 Focus Writer 的無干擾寫作體驗。

安裝方法：
1. 從 GitHub 下載插件的 ZIP 或 tarball
2. 解壓縮到 `~/.vim/pack/[任意名稱]/start/[插件名稱]/` 目錄
3. 重啟 Vim，插件自動載入

大多數插件的 README 都有詳細的安裝說明，整個過程非常直覺，不需要任何額外的設定檔。

### 文字大小寫切換指令

#### Visual Mode 操作

進入 Visual Mode (`Shift + V` 或 `Ctrl + V`)，選取目標範圍後：

- `U`（大寫 U）：將選取範圍**全部轉為大寫**
- `u`：將選取範圍**全部轉為小寫**

#### Normal Mode 直接命令

將遊標停留在單字上，直接在 Normal Mode 下執行：

- `gUiw`：將**當前單字**轉大寫（`iw` 是一個 Text Object）
- `guaw`：將**當前單字**轉小寫

#### 交換大小寫：`~` 命令

`~` 命令會將單個字元的大小寫翻轉——小寫變大寫，大寫變小寫。在 Visual Mode 中對整個選取範圍執行 `~`，也會翻轉所有大小寫。

### Visual Mode 與直接命令：何時該用哪一種？

這是直播中討論最深入的實務問題之一。

#### Visual Mode 的價值

Visual Mode 的最大優點是**所見即所得**——在執行操作之前，你能清楚看到 Vim 將會影響哪些文字。對於跨越多個螢幕畫面的長範圍選取，或是結構複雜的 HTML 區塊，先選取再操作能避免失誤。

操作範例（以雙引號內的文字為例）：

```
"block a block"
```

- `vi"`：選取**引號內的內容**（不包含引號本身）
- `va"`：選取**包含引號的整個範圍**

#### 直接命令的價值

當你已經熟悉 Vim 的操作邏輯，能在腦中預判命令的效果時，直接執行 Text Object 命令會更快速：

- `ci"`：刪除引號內內容並進入插入模式（相當於 `vi"c` 的簡化版）

#### 老師的個人取捨

> 「如果是很簡單的動作，我直接在腦中想好就執行。如果複雜到跨越多個螢幕，我通常會進 Visual Mode，一路 page down 選取範圍，再執行操作。看情況。」

### `vi` 別名與系統連結

在一些 Linux 發行版中，`vi` 已經被設定為 `vim` 的軟連結（Soft Link）。因此即使直接輸入 `vi`，系統也會啟動 Vim。老師笑說自己是「老人習慣」，打字時下意識就輸入 `vi`，反正結果一樣。

Ubuntu/Debian 系統也可以用 `update-alternatives` 工具來管理多個 Vim 安裝版本之間的切換，不過在大多數情況下，安裝 `vim-full` 或 `vim-minimal` 時，系統就會自動設定好連結。

## 💡 重點摘要

- **執行來路不明的 GitHub 腳本前，務必先在 VM 隔離環境中逐行測試；信任來源後再用 `curl | bash` 或儲存腳本後執行。**
- **Vim 的未來並不黯淡：Bram Moolenaar 有 Google 資源支持，加上 Neovim 作為成熟替代，所有 Vim 操作技能都不會白費。**
- **Vim 8 內建插件支援，安裝插件只需將目錄放到 `~/.vim/pack/[名稱]/start/[插件名]/`，不需要額外套件管理器。**
- **文字大小寫：`gUiw` / `guaw` 處理當前單字；Visual Mode + `U`/`u` 處理任意範圍；`~` 翻轉單一字元的大小寫。**
- **Visual Mode 適合需要視覺確認的複雜選取，熟悉後可直接用 Text Object 命令提升操作速度。**
