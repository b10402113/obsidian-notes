# 微服務測試金字塔：概念與挑戰

## 📝 課程概述

本單元從傳統 Monolith 應用程式的 Testing Pyramid（測試金字塔）談起，逐步延伸至微服務架構中的測試策略。我們將看到，微服務之間透過網路或 Message Broker 進行溝通、且各自擁有資料庫的特性，使得原本的測試金字塔變得更加複雜——尤其是 End-to-End Tests 與 Integration Tests 兩層，在微服務環境中面臨著前所未有的挑戰。

## 核心觀念與實作解析

### 傳統 Monolith 的測試金字塔

在 Monolith 時代，測試通常分為三個層級，由下而上構成金字塔：

```
        ▲ End-to-End Tests（最少）
       ▲▲ Integration Tests（中等）
     ▲▲▲▲ Unit Tests（最多）
```

| 層級 | 測試內容 | 數量 | 執行速度 | 信心程度 |
|------|---------|------|---------|---------|
| **Unit Tests** | 測試 class 或 module 等小邏輯單元，**完全隔離** | 最多 | 最快（毫秒級） | 最低 |
| **Integration Tests** | 驗證不同單元與外部系統（如 Database、Message Broker）**能否正確協作** | 中等 | 較慢 | 中等 |
| **End-to-End Tests** | 模擬真實使用者旅程，涵蓋 UI → 整個 Application → Database 的完整流程 | 最少 | 最慢 | 最高 |

> **為什麼是金字塔形？** Unit Tests 最便宜（快速好寫），因此數量最多；E2E Tests 最昂貴（環境複雜、執行緩慢），因此數量最少。數量與成本呈反向關係。

---

### 將測試金字塔應用到微服務架構

微服務架構下的測試金字塔有兩層含義：

#### 第一層：每個微服務內部的測試金字塔

每個微服務團隊在各自程式碼庫內部，遵循**相同的傳統測試金字塔**——Unit Tests、Integration Tests（對自己的資料庫）、Functional Tests（對自己的服務）。

#### 第二層：整個系統的宏觀測試金字塔

將每個微服務視為系統中的「小型單元」，再往上建構：

```
          ▲▲ System E2E Tests（全部微服務 + Databases + Message Brokers + Frontends）
        ▲▲ Contract Tests（兩兩微服務之間的 API 契約驗證，Mock 掉其餘系統）
      ▲▲▲▲ Unit Tests（每個微服務各自內部的測試金字塔）
```

> **為什麼還需要額外的 Contract Tests？** 因為各微服務之間只能透過網路 API 溝通，**僅測試每個服務的內部邏輯是不夠的**——我們還需要驗證「當 Service A 向 Service B 發送請求時，雙方是否能正確溝通」。

---

### 微服務測試金字塔的三大挑戰

#### 挑戰一：End-to-End Tests 的成本與擁有權困境

理論上，E2E Tests 應該在測試環境中啟動所有微服務、資料庫、Message Broker 與前端，驗證整個系統的協作。但在實務上：

- **環境維護成本極高**：要運行一個完整複製生產環境的測試環境，代價昂貴。
- **擁有權模糊**：哪個團隊應該擁有並維護這個橫跨所有微服務的共享測試環境？當某個團隊的微服務 Build 壞了，即使只是一天，**整條測試管線都會被阻斷**，所有團隊無法發布任何服務到生產環境。
- **兩極化的公司策略**：有些公司投入過多資源維護極少數的 E2E Tests；有些公司則直接放棄，完全不投資——兩種極端都有風險。

#### 挑戰二：Integration Tests 造成團隊之間的緊密耦合

當團隊 A 的微服務（API Consumer）想要執行 Integration Tests 時：

- 團隊 A **需要會建置、設定與執行團隊 B 的微服務**——但團隊 A 可能根本不清楚如何操作團隊 B 的系統。
- 若團隊 B 的微服務依賴多個其他資料庫或服務，設定難度更高。
- 反過來，團隊 B（API Provider）若有多個 Consumer 服務，要確保自己的 API 變更沒有破壞任何 Consumer，就必須**同時啟動並執行所有 Consumer 服務來測試**。

> 這形成了一個快速失控的連鎖反應，讓整個組織的開發速度大幅下降。

#### 挑戰三：Event-Driven 架構讓 Integration Tests 幾乎不可能

Event-Driven 架構中，生產事件（Producer）的微服務**並不知道誰在消費這些事件**——這是刻意為之的 decouple 設計，卻成了測試的死穴：

- 要測試「事件格式是否正確」→ 必須實際啟動 Message Broker 與上下游服務。
- 要測試「某個 Consumer 能否正確解析事件」→ 必須啟動整個事件鏈路（Producer → Broker → Consumer）。

> 這與我們追求的「快速隔離測試」目標完全背道而馳。

---

## 💡 重點摘要

- **微服務架構下的測試金字塔有兩層：每個微服務內部的傳統金字塔，以及跨越所有微服務的宏觀金字塔（Contract Tests + System E2E Tests）。**
- **End-to-End Tests 在微服務環境中成本極高（環境複製費用大）、擁有權模糊（誰負責維護共享環境？），且任何一個團隊的 Build 失敗都會阻斷所有團隊的發布節奏。**
- **Integration Tests 在微服務環境中要求每個團隊都必須了解、啟動並操作其他團隊的服務——這將耦合從程式碼層級擴散到了組織協作層面。**
- **Event-Driven 架構刻意去除了 Consumer 與 Producer 之間的耦合，但代價是 Integration Tests 幾乎無法在隔離環境中執行。**
- **Contract Tests 是為了解決這些挑戰而出現的替代方案——下一堂課將深入探討。**
