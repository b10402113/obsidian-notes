# 可觀測性五大關鍵指標

## 📝 課程概述

本單元聚焦於 Metrics（指標）——可觀測性三大支柱中**最容易收集與視覺化**的一種。我們將理解為什麼「收集所有指標」是一種反模式，並學習五個在任何系統中都最值得關注的核心信號：**流量（Traffic）**、**錯誤（Errors）**、**延遲（Latency）**、**飽和度（Saturation）** 與 **利用率（Utilization）**。

---

## 核心觀念與實作解析

### 什麼是指標（Metrics）？

Metrics 是軟體系統的**可量化信號**，幫助我們監控系統的健康狀況與效能。

核心特性：
- **定期採樣**：產生連續的趨勢資料，而非離散的快照，藉此發現異常
- **數值形式**：因為是數字，可以輕鬆設定閾值警報，並與歷史資料做對比
- **最容易視覺化**：三大支柱中，Metrics 最容易製作成 Dashboard，緊急時刻工程師只需看一兩個畫面就能掌握全局

### 為什麼不能收集所有指標？

收集所有可測量的指標是一種**明確的反模式**，原因有兩個：

**成本問題**：數十甚至數百個 Microservice，每個都在產生大量指標，儲存與處理的成本會迅速失控。

**資訊過載（Information Overload）問題**：

> 凌晨三點收到警報「網站很慢」，打開 Dashboard 卻發現圖表數量多到螢幕放不下，根本找不到異常在哪。就算找到了，也分不清**哪個是症狀（Symptom）、哪個是根本原因（Cause）**。

### 五個最有價值的指標類別

借鑒 Google 的 **「SRE Gold Search」** 與 Brendan Gregg 的 **「USE Method」**，五個核心指標類別覆蓋了最有價值的系統訊號。

#### 1. Traffic（流量）

**定義**：單位時間內系統承受的需求量。

測量方式：
- **HTTP Service**：每秒/每分鐘的 HTTP 請求數
- **資料庫**：每秒/每分鐘的查詢或交易數
- **Message Broker**：收到的 Event 數量、發送給 Consumer 的 Event 數量

**額外建議**：當一個請求進入 Service 後觸發多個輸出請求時，**輸入請求數與輸出請求數應該分開測量**。因為對外服務的連線數量會直接消耗系統資源並影響該 Service 的效能。

#### 2. Errors（錯誤）

**定義**：系統中各環節的錯誤率與錯誤類型。

警報價值極高——錯誤率突然上升幾乎意味著**用戶已經受到影響**。

不同環節的錯誤測量方式：

| 環節 | 測量方式 |
|------|---------|
| HTTP 回應 | 非 200 的 Status Code 計數 |
| Application | Exception 計數（詳細類型交給 Logging） |
| Event-Driven Service | 處理失敗的 Event 數量 |
| Message Broker | 未能成功傳遞給 Consumer 的 Event 數量 |
| Database | 中止的交易數、磁碟故障數 |

**特殊情況**：在 Latency 敏感的系統中，**超過預設 Latency 閾值的成功回應也可以被視為「錯誤」**——這稱為 Latency Budget。

#### 3. Latency（延遲）

測量延遲看似直觀，但有兩個至關重要的考量：

**考量一：永遠不要只看平均值，要看完整的 Latency 分布**

來看一個驚人的例子：

> 假設某 Service 每分鐘處理 1000 個請求，其中 95% 在 50ms 內完成，5% 需要 5000ms。
> 平均延遲 = (0.95 × 50 + 0.05 × 5000) ≈ **300ms**，看起來還不錯。
> 但實際上有 **50 個用戶每分鐘要等 5 秒鐘！**
> 如果網站每天有 100 萬用戶，這 5 萬人（5%）很可能直接放棄並轉向競爭對手。

> 解決方案：永遠追踪 **P95（95th Percentile）Latency**，而不是平均值。P95 能揭示那 5% 長期被平均值掩蓋的糟糕體驗。

**考量二：成功請求與失敗請求的延遲應該分開測量**

- 如果我們使用 **Fail-Fast**（快速失敗）策略，失敗請求的處理時間會很短，混入成功請求的 Latency 資料會**讓整體 Latency 看起來比實際更好**
- 如果錯誤處理本身很慢（例如需要等待很長時間才回傳錯誤），把失敗 Latency 混在一起會讓你**完全看不到這個問題**

> 正確做法：分別監控成功與失敗請求的 Latency，才能獲得對用戶真實體驗的完整視角。

#### 4. Saturation（飽和度）

**定義**：某個 Service 或資源的**負載程度**，通常體現在 Queue 的堆積情况。

這是指標中最能預示「即將出問題」的一類：

| 觀察到的飽和度訊號 | 可能的根本原因 |
|-----------------|------------|
| Message Broker 中的 Purchase Order Topic 不斷增長 | 某個 Consumer Service 有擴展性問題 |
| 對 Database 的請求佇列不斷增長 | 儲存設備效能不足，或需要 Sharding |
| Service Instance 內部的工作佇列不斷增長 | 某段邏輯太慢，可能即將 OOM（記憶體溢出） |

> **飽和度上升的同時，通常伴隨著 Latency 上升或來自其他 Service 的 Timeout**——這是系統開始降級的明確信號。

#### 5. Utilization（利用率）

**定義**：某個有限容量資源在一段時間內的**繁忙程度**。

測量對象：CPU、Memory、Disk Space 等。

關鍵認知：

> 大多數系統在達到 100% 利用率**之前**就會出現效能下降。因此必須**在臨界點之前**設定警報。

實務建議：
- **CPU 利用率接近 80%** 時就應該擴展 Service Instance，否則超過臨界點後 Latency 會急劇上升
- **Database 即將耗盡儲存空間** 時需要立即新增資料庫實例
- 測量時要使用**足夠高的顆粒度**，不要只看幾分鐘的平均值——短暫的高利用率峰值往往預示著效能瓶頸或邏輯問題

---

## 💡 重點摘要

- **Metrics 是三大支柱中最容易視覺化的，緊急時刻一個 Dashboard 勝過翻閱幾百萬行日誌。**
- **「收集所有指標」是反模式——不僅成本高，更會造成資訊過載，讓真正重要的異常被淹沒。**
- **Latency 不能只看平均值：P95 Latency 才能揭示那被平均值掩蓋的少數用戶的糟糕體驗。**
- **成功與失敗請求的 Latency 必須分開測量，否則 Fail-Fast 會讓你的 Latency 數據看起來比實際好，隱藏真正的問題。**
- **Saturation 是系統即將降級的最強預警信號——Queue 堆積往往領先於 Latency 飆高，是主動發現問題的關鍵窗口。**
