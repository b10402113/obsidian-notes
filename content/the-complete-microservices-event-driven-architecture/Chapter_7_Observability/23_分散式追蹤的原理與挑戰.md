# 分散式追蹤的原理與挑戰

## 📝 課程概述

本單元深入探討可觀測性三大支柱中的最後一個——**Distributed Tracing（分散式追蹤）**。我們將從一個電子商務交易的真實場景出發，理解為什麼 Metrics 與 Logs 還不夠，並學習 Distributed Tracing 的核心概念（Trace、Span、Trace Context）、資料收集架構，以及它所帶來的三大挑戰：手動 Instrumentation、成本與資料量膨脹。

---

## 核心觀念與實作解析

### 一個真實場景：電子商務訂單確認信延遲

想像一個電子商務系統中，用戶下單後的處理流程：

```
用戶下單 → API Gateway → Auth Service → Order Service → Message Broker
    → Inventory Service → Payment Service → Shipping Service → Notification Service
    → 用戶收到 Email / Push Notification（含收據與追蹤連結）
```

突然間，部分用戶反映**沒收到確認信**，另一部分反映**過了好幾小時甚至一天才收到**。

**問題可能在哪裡？**

- Notification Service 本身有 Bug？
- Payment Service 回應太慢？
- Message Broker 的某個 Topic 堆積了？
- Shipping Service 出了錯？
- 還是第三方 API 的問題？

> 在這個複雜的分散式架構中，**沒有 Distributed Tracing，根本無從下手**。

### 為什麼需要 Distributed Tracing？

在真實系統中，可能涉及數百個 Microservice，每個 Service 在整個請求流程中各自扮演一部分角色。**沒有任何人類能完整記住所有 Service 的交互順序與依賴關係。**

**Distributed Tracing 的核心定義**：

> 追蹤請求從用戶端裝置，一路穿過所有後端 Service 與資料庫的完整路徑，並在過程中收集每個環節的關鍵效能資訊。

這讓工程師能：
- **視覺化整個請求流程**，了解所有參與的元件
- **精確測量每個環節的處理時間**
- **隔離出問題的環節**，大幅縮小 Debug 範圍

> Tracing 本身通常不足以直接告訴你「根本原因是什麼」，但它**足夠精確地指出「問題在哪個環節」**——之後再結合 Logs 與 Metrics 深入挖掘。

### 核心概念與術語

#### Trace ID

每個請求在**初始發起時**，系統會生成一個**全球唯一的 Trace ID**，這個 ID 會一路跟隨請求穿越所有 Service。

#### Trace Context（追蹤上下文）

將 Trace ID 封裝在一個稱為 **Trace Context** 的物件中，裡面還包含其他關於整個 Trace 的關鍵資料。Context 透過以下方式傳遞：

- **HTTP 通訊**：放在 HTTP Headers 中
- **非同步訊息**：放在 Message Headers 中（Event 的 Header）

#### Instrumentation Library / SDK

光靠傳遞 Context 是不夠的——Service Instance 還需要被**插樁（Instrumented）**，才能真正收集追蹤資料。各程式語言都有對應的 Tracing Library，用於：

- 接收並處理 Trace Context
- 收集該 Service 的度量資料
- 將 Context 傳遞給下一個 Service

#### Span（跨度）

Trace 由多個 **Span** 組成，每個 Span 代表一個**邏輯工作單元**：

- **粗粒度 Span**：「這個 Service 處理請求花了多長時間」、「這次資料庫查詢花了多長時間」
- **手動建立的細粒度 Span**：開發者可以對 Service 內部的不同邏輯步驟，分別建立 Span

Span 可以組成**父子階層關係（Parent-Child Hierarchy）**：

> 如果追蹤顯示某個 Service 的 Span 比平常慢，可以展開它的子 Span，發現某個資料庫查詢異常緩慢——這就是效能瓶頸。

### 資料收集與聚合架構

整個流程如下：

```
Service Instance 產生 Trace 資料
    → 本地 Agent（與 Service 同機器運行的獨立 Process）拉取資料
    → 傳送至中央 Queue / Message Broker Topic
    → Big Data Processor 分析、聚合、索引、儲存至資料庫
    → 工程師透過瀏覽器中的 Tracing UI 查詢與視覺化
```

### 三大挑戰

#### 挑戰一：需要手動進行 Code Instrumentation

大多數情況下這不是大工程，但仍有成本：
- 需要引入並依賴特定的 Tracing Library
- 需要學習正確的使用方式
- 有些場景需要**手動補充 Instrumentation**

如果 Instrumentation 不完整，會導致：
- **Span 太寬泛**，缺乏顆粒度
- **Trace 斷裂**（Broken Trace），完全無法使用

> 這與 Logging 的最佳實踐類似——好的 Tracing 效果，取決於工程師願意投入多少紀律來維護它。

#### 挑戰二：成本

Distributed Tracing 的基礎設施成本不容小覷：

| 資源消耗來源 | 說明 |
|------------|------|
| 每台 Service Host 上的 Agent | 消耗 CPU 與 Memory |
| 網路頻寬 | Trace 資料需透過網路傳輸 |
| Big Data Pipeline | 分析與處理追蹤日誌的專屬基礎設施 |
| 資料庫儲存 | 最主要的成本——需要保留數週的資料以備 Debug |

#### 挑戰三：Sampling 與 Trace 龐大問題

在每秒數百萬請求、每個請求穿越數十個 Microservice 的系統中，**Trace 資料量極為驚人**。

為控制成本，大多數公司採用 **Client-Side Sampling**：

> 例如每 1000 個甚至 10000 個請求中，只完整追蹤其中 1 個。

代價：
- 極高的 Sampling Ratio，使得重現特定問題的 Trace 難以找到
- 單一 Trace 可能**過於龐大**，連 Tracing UI 都無法載入

> 老師提到，他親眼見過 Production 環境中的 Trace 太大，Tracing UI 完全無法渲染——這種情況在業界並不罕見。

### 結論：即使有挑戰，Distributed Tracing 仍是必備工具

儘管存在這些挑戰，Distributed Tracing 在 Microservices 與 Event-Driven 架構中的價值難以替代。它常被認為是**恢復工程師對 Production Debug 信心**的關鍵工具——讓團隊相信：即使 Production 出問題，我們有手段找出根本原因。

---

## 💡 重點摘要

- **在橫跨數十個 Microservice 的複雜請求流程中，沒有 Distributed Tracing，工程師幾乎不可能定位問題環節。**
- **Trace ID 與 Trace Context 是追蹤的基礎，它們透過 HTTP Headers 或 Message Headers 在 Service 間傳遞，Instrumentation Library 負責實際的資料收集。**
- **Span 是追蹤的基本單位，可以組織成父子階層關係——當某個 Span 異常時，展開其子 Span 可以逐步鎖定具體的效能瓶滸。**
- **Client-Side Sampling 是控制成本的必要手段，但高 Sampling Ratio 會犧牲問題重現能力，需要在成本與可用性之間取得平衡。**
- **三大挑戰——手動 Instrumentation、昂貴的基礎設施、龐大的資料量——是採用 Distributed Tracing 時必須提前規劃的成本。**
