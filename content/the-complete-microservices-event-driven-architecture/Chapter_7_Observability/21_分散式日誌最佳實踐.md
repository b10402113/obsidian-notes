# 分散式日誌最佳實踐

## 📝 課程概述

本單元深入探討 Microservices 架構下日誌實作的核心挑戰，以及六項關鍵的最佳實踐。我們將理解為什麼「簡單地把 Log 寫到檔案」在分散式環境中完全不可行，並學習如何建立一套可搜尋、可結構化且能保護隱私的日誌系統。

---

## 核心觀念與實作解析

### 分散式環境下的日誌災難

在 Microservices 中，整個系統可能有**數千個不同的 Service Instance**，每天共同產生**數百萬行日誌**。當 Production 出問題時，工程師不可能逐個打開日誌檔案來搜尋。

這就是為什麼**集中式日誌系統（Centralized Logging System）**是第一步也是最基礎的需求。

### 最佳實踐一：集中式儲存與索引

日誌系統的核心需求：

- **收集（Collect）**：從所有 Service Instance 自動蒐集日誌
- **儲存（Store）**：使用可横向擴展的儲存引擎
- **解析與索引（Parse & Index）**：讓日誌能透過**關鍵字、Pattern** 快速搜尋，並能按 **Host、Service、Time Period、Region** 等屬性分組過濾

> 這是後續所有最佳實踐的基礎——如果日誌無法快速找到，一切都是空談。

### 最佳實踐二：統一日誌結構（Schema）

跨所有 Service 必須遵循**預先定義的結構與術語**，不只在單一 Service 內，還包括跨 Service。

好的日誌必須同時滿足兩個目標：
- **人類可讀**：工程師能在 Production 緊急事件中快速掃描理解
- **機器可解析**：能高效地被自動工具解析、分組與分析

常見的日誌結構格式：

- **Logfmt**：以 Key-Value 配對的形式組織，簡潔易讀
- **JSON**：最普遍的結構化格式，機器解析能力強
- **XML**：某些企業場景仍然使用

### 最佳實踐三：設定 Log Severity Level（日誌級別）

為每行日誌指定級別，是降低噪音（Noise）與避免警報疲勞（Alert Fatigue）的關鍵手段。

常見的級別層次（由輕到重）：

| 級別 | 用途 | 例子 |
|------|------|------|
| `TRACE` | 最細粒度的偵錯資訊 | 每個迴圈迭代的狀態 |
| `DEBUG` | 開發階段的詳細偵錯資訊 | 某個參數的值 |
| `INFO` | 一般性操作記錄 | 「收到新請求」 |
| `WARN` | 潛在問題的預警信號 | 某個下游請求延遲過高、收到異常格式的資料 |
| `ERROR` | 已經發生的錯誤，需處理 | 例外捕捉、HTTP 500 |
| `FATAL` | 嚴重崩潰，直接影響用戶 | 資料庫連接完全中斷 |

> **為什麼這很重要？** 凌晨三點收到警報，工程師需要能馬上 Filter 到 `ERROR` 與 `FATAL`，快速定位問題，而不是在幾百萬行 `INFO` 中大海撈針。

自動化工具也可以定期按 Severity 分組日誌，自動建立 Ticket 並指派給負責人。

### 最佳實踐四：Correlation ID（關聯 ID）

**每個使用者請求或交易，必須分配一個唯一的 Correlation ID，並將其附加到所有相關的日誌行。**

在 Microservices 中，單一 Instance 同時處理多個使用者請求是常態。如果沒有 Correlation ID，要把某個特定請求的所有日誌串聯起來，幾乎是不可能任務。

有了 Correlation ID：
- 可以只搜尋與該請求相關的所有事件
- 可以追蹤該請求**穿越多個 Microservice 的完整順序**
- 可以重現整個問題的發生過程

> 這是日誌可用性的根本——沒有 Correlation ID，日誌只是噪音。

### 最佳實踐五：豐富的上下文資訊（Contextual Information）

每行日誌應該盡可能包含有助於 Debug 的背景資訊：

**ERROR/Exception 記錄時必備：**
- **Stack Trace**：理解程式如何執行到這一行
- **導致錯誤的參數值**：否則很難重現問題

**緩慢的資料庫查詢必備：**
- **實際的 SQL 查詢語句**
- **觸發該查詢的 Request 內容**

**幾乎所有日誌行都應包含的通用資訊：**
- 發出日誌的 Service 名稱
- 事件發生的 Hostname
- 觸發操作的使用者 ID 或其他識別符
- Event 發生的精確 Timestamp

### 最佳實踐六：安全與隱私考量

有兩條高優先級的安全紀律：

**只記錄 Debug 所需的關鍵資訊**：在大規模系統中，儲存與處理的成本驚人。不必要的日誌不只是浪費金錢，也會讓真正重要的資訊被淹沒。

**絕對不能記錄敏感或個人識別資訊（PII）**：

> Usernames、Passwords、Social Security Numbers、Emails、Credit Card Numbers……

**為什麼？**
- 資安漏洞一旦發生，這些資料會造成巨大的法律與信譽風險
- 增加了安全合規與資料保留（Data Retention）的複雜度
- 讓不相關的工程師接觸用戶的私人資料，本身也是不道德的

> **一個好的紀律原則**：如果某筆資料對於 Debug 該問題不是必需的，就不要記錄它。

---

## 💡 重點摘要

- **分散式日誌的核心挑戰是數量龐大且分散——必須建立集中式、可索引的日誌系統，否則日誌只是無用的噪音。**
- **統一日誌結構（Logfmt/JSON）讓日誌同時被人類閱讀和機器解析，緊急時刻才能快速發揮價值。**
- **Log Severity Level 是降低噪音、避免 Alert Fatigue 的核心機制；工程師應該能只看 ERROR/FATAL 就快速上手。**
- **Correlation ID 是將跨 Service、跨 Instance 的日誌串聯成完整故事線的關鍵——沒有它，Debug 幾乎無從下手。**
- **日誌安全與隱私保護不只是合規要求，更是基本的工程倫理——不必要的敏感資料，絕不應該出現在日誌中。**
