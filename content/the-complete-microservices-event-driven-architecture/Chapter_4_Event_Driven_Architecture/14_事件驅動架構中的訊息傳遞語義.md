# 事件驅動架構中的訊息傳遞語義

## 📝 課程概述

本單元深入探討 Event-Driven 系統中最核心也最容易被忽略的問題：**訊息傳遞的可靠性**。我們將學習三種訊息傳遞語義（Delivery Semantics）——**At Most Once**、**At Least Once** 與 **Exactly Once**——它們各自的保障程度、適用場景，以及 Exactly Once 在實作上最容易被誤解的地方。

---

## 核心觀念與實作解析

### 問題的起源：訊息丟失的三種情形

在分散式系統中，一條訊息（Event 或 Request）從發送方到接收方，可能在中間任何環節「消失」。以 Request-Response 為例：

1. **接收方根本沒收到 Request**（網路丟包、接收方尚未啟動）
2. **接收方收到了，但處理到一半就崩潰了**（硬體/軟體故障）
3. **接收方處理完了，也寫入資料庫了，但 Response 在回程路上丟失了**

> **從發送方的角度，它永遠無法區分這三種情形。** 這就是所有 Delivery Semantics 爭論的起點：到底該不該重試？

### Event-Driven 中的傳遞複雜度更高

在 Event-Driven 架構中，流程變成了三方參與（Publisher → Message Broker → Subscriber），問題鏈更長：

- Publisher 發 Event 給 Broker，Event 在網路中丟失
- Broker 收到了並寫入 Log，但發給 Publisher 的 Ack 在網路中丟失
- Broker 發 Event 給 Subscriber，Subscriber 沒收到
- Subscriber 收到了並處理了，但回給 Broker 的 Ack 在網路中丟失

> 這就是為什麼在設計系統之前，**Producer、Broker 與 Consumer 必須事先約定好一致的 Delivery Semantics**。

### Delivery Semantics 一：At Most Once（最多一次）

**核心理念**：寧可丟訊息，也不要重複處理。

**Producer 端的行為**：
- 發 Event 給 Broker 後，如果一段時間內沒收到 Ack，**不回重試，直接放棄**。
- **最佳情況**：Event 確實送到了，只是 Ack 丟失了——沒有任何資料損失。
- **最壞情況**：Event 根本沒到，或 Broker 在寫入前就崩了——Event 永久丟失。

**Consumer 端的行為**：
- **先發 Ack，再處理 Event**。如果 Consumer 在處理過程中崩潰重啟，已經 Ack 過的 Event 不會再被傳送，因此這些 Event 已被處理（只不過崩潰發生在處理期間，可能造成半成品狀態）。

**適用場景**：
- 少量資料損失可接受、可從其他 Event 推斷資料的情境
- **即時日誌與 Metrics 收集**：位置更新、雲端伺服器指標等，少丟幾筆不影響大局，但重複處理則白白浪費資源
- 需要**最低延遲與最低 Overhead** 的高吞吐量系統

> 這是三種語義中最簡單、延遲最低的一種。適合「寧缺毋濫」的場景。

### Delivery Semantics 二：At Least Once（至少一次）

**核心理念**：寧可重複處理，也不要遺失任何訊息。

**Producer 端的行為**：
- 發 Event 給 Broker，如果在約定時間內**沒收到 Ack，就重新發送同一個 Event**。
- 結果：同一個 Event 可能在 Broker 中出現**多次**。

**Consumer 端的行為**：
- **先處理 Event，處理完後再發 Ack 給 Broker**。
- 如果 Consumer 在處理完之後、發 Ack 之前崩潰，Broker 會認為傳遞失敗，Consumer 重啟後會**再次收到同一個 Event** 並重新處理。

**結果**：Event 永遠不會丟失，但**同一個 Event 可能被處理兩次**。

**適用場景**：
- 不允許資料丟失，但**輕微的資料重複是可以接受的**
- 發送 Push Notification：寄送兩次通知問題不大，但漏寄就不好了
- 用戶發布評論：由於評論綁定用戶 ID，且系統有防重複邏輯，第二次的評論請求會被忽略或覆蓋，不造成實際問題

**代價**：
- 每次發送都需要等待 Ack 或等待重試，**延遲明顯增加**
- 不適合對延遲敏感的即時系統

### Delivery Semantics 三：Exactly Once（恰好一次）

**核心理念**：每個 Event 必須且僅被處理一次。這是所有金融系統的夢想，但也是最難實現的。

> **遺憾的是：即使某些 Message Broker 宣稱支援 Exactly Once，實際上它們也無法對所有情況提供完整保障。** 在選擇技術之前，請務必仔細閱讀文件。

#### Producer 端的 Exactly Once

做法是引入 **Idempotency ID（冪等性 ID）**：

1. 在發 Event 前，向 Broker（或專門的 ID 生成服務）取得一個**唯一識別符**
2. 連同 Event 一起發送給 Broker
3. 如果 Broker 提供 ID，它會檢查 Log 中是否已有相同 ID 的 Event——有的話直接忽略，沒有的話寫入
4. 如果 Publisher 需要自行重試，它會**帶著同一個 Idempotency ID 重新發送**，Broker 會自動去重

這確保了** Producer → Broker** 這段不會出現重複 Event。

#### Consumer 端的 Exactly Once：最容易被誤解的部分

> **當 Consumer 收到 Event 並需要將結果寫入外部資料庫時，Message Broker **無法**保證 Beyond At Least Once。**

原因：

```
Consumer 收到 Event → 處理並寫入資料庫 → 發 Ack 給 Broker
```

如果 Consumer 在「寫入資料庫後、發 Ack 前」崩潰，Broker 會重送同一個 Event，Consumer 會**再次處理並再次寫入資料庫**——造成重複。

**解決方案：必須自己實作 Idempotency**

1. 在資料庫的記錄中加入 **Idempotency ID 欄位**
2. Consumer 每次處理 Event 時，先檢查資料庫中是否已有相同 ID 的記錄
3. 如果有，**程式中斷處理，直接忽略該 Event**
4. 如果沒有，則正常處理並寫入，同時存入 Idempotency ID

> **這是 Consumer 端自己需要處理的事，不是 Broker 能代勞的。** 大多數人對 Exactly Once 的誤解，正在於此。

## 三種 Delivery Semantics 總表

| 特性 | At Most Once | At Least Once | Exactly Once |
|------|-------------|--------------|--------------|
| 訊息丟失風險 | 可能丟失 | 不會丟失 | 不會丟失 |
| 重複處理風險 | 不會重複 | 可能重複 | 不會重複 |
| 延遲 | 最低 | 較高 | 最高 |
| Overhead | 最低 | 中等 | 最高 |
| 適用場景 | 即時日誌、Metrics | Push Notification、評論發布 | 金融交易、帳務系統 |

---

## 💡 重點摘要

- **在分散式系統中，發送方永遠無法區分「訊息未送達」與「訊息已處理但 Ack 丟失」這兩種情形，這是所有 Delivery Semantics 問題的根源。**
- **At Most Once：優先避免重複，願意承受資料丟失，適合即時日誌、低延遲高吞吐量場景。**
- **At Least Once：優先避免資料丟失，願意承受重複處理，適合 Push Notification、評論系統等輕微重複無害的場景。**
- **Exactly Once 看似理想，但 Consumer 端如果涉及外部資料庫寫入，Message Broker 無法獨立完成 Exactly Once 保障——必須由 Consumer 自己透過 Idempotency ID 機制實作。**
- **任何涉及金融影響力的資料處理，都應格外謹慎地閱讀所使用的 Message Broker 技術文件，確認其 Delivery Semantics 的實際保障範圍。**
