# 事件驅動架構的動機與核心概念

## 📝 課程概述

本單元從一個具體的商業場景——付費訂閱服務——出發，逐步推導出為什麼 Synchronous Request-Response 模型在複雜的跨 Service 交易中力不從心。我們將認識 Event-Driven Architecture 的核心概念：Event、Producer、Message Broker 與 Consumer，並理解它與 Microservices Architecture 為何能如此完美地搭配。

---

## 核心觀念與實作解析

### 情境：付費訂閱服務的交易難題

想像一個線上影音訂閱平台，用戶從免費試用升級到付費訂閱時，背後有三個 Microservices 同時參與：

- **Subscription Service**：管理每位用戶的訂閱等級、過期日等資訊
- **Payment Service**：與第三方信用卡處理服務溝通，負責扣款
- **Recommendations & Personalization Service**：維護每位用戶的觀看歷史與個人化推薦

> 這裡有一個關鍵約束：**三個 Service 必須全部成功處理訂閱請求，用戶的訂閱狀態才算完整。** 任何一個環節失敗，整個交易就不算完成。

### 方案一：同步 Chain 呼叫的問題

最直覺的實現方式是用一條長長的同步呼叫鏈：

```
使用者 → Subscription Service → Payment Service → Recommendation Service
```

表面上這能保證「當用戶收到回應時，三個 Service 都已成功處理」。但代價是：

- **用戶必須等待整條 Chain 完成**：即使一切順利，等待時間是所有 Service Latency 的總和。
- **第三方服務延遲會拖垮整個體驗**：如果 Payment Service 所依賴的信用卡處理方效能下降，用戶的等待時間會變得完全無法接受。

更糟的是，我們無法靠「提前回應用戶」來優化：

> 如果 Payment Service 尚未成功回覆就發確認給用戶，Payment Service 可能根本沒收到請求，用戶將面臨**未被扣款但也沒拿到訂閱**的不一致狀態。

因此，Request-Response 模型中，參與的 Service 越多，用戶等待時間就越長，卻仍要維持資料一致性——這在很多場景下根本不可接受。

### 方案二：Orchestration Pattern 的極限

另一種做法是引入一個 **Orchestration Service** 來統一協調，广播请求给所有相關 Service 並行執行。

**好處**：等待時間從「所有 Service Latency 的總和」降到「最慢那個 Service 的 Latency」。

**但仍存在問題**：

- 如果 Payment Service 依賴的第三方服務緩慢，整體延遲仍然過高（而且這不是我們能控制的）
- **Orchestration Service 與所有參與的 Service 緊密耦合**——任何一個 Service 的 API 變動，都需要同步更新 Orchestration Service
- 如果某個 Service 臨時故障，整個訂閱請求都會被拒絕，失去潛在的付費用戶

### Event-Driven Architecture 的核心概念

#### Event 是什麼？

> **Event 代表系統中發生的「事實」（Fact）、「動作」（Action）或「狀態變化」（State Change），而且是不可變的（Immutable）。**

與 Request 的關鍵差異：
- **Request 是短暫的（Ephemeral）**，只被消費一次、只由一個 Server 接收
- **Event 是持久的（Immutable & Storable）**，可以被多個不同的 Service 重複消費

#### 三個核心角色

Event-Driven Architecture 中有三大參與者：

| 角色 | 職責 |
|------|------|
| **Producer（發布者）** | 產生 Event |
| **Message Broker** | 儲存並路由 Event，帶來冗餘（Redundancy）與緩衝能力 |
| **Consumer（消費者）** | 接收並處理 Event，一個 Event 可同時被多個 Consumer 消費 |

> **Message Broker 是關鍵**：一旦 Event 被寫入 Broker，它不會消失，Consumer 可以隨時取用。這帶來了系統級別的 Redundancy。

### Request-Response 與 Event-Driven 的兩大根本差異

#### 差異一：同步 vs 非同步通訊

- **Request-Response（同步）**：發送方發出 Request 後**必須等待 Response**，即使 Response 裡沒有實際有用的資訊。如果一直沒收到 Response，通常表示系統有問題。
- **Event-Driven（非同步）**：Publisher 發布 Event 後**不需要也不期待收到任何回覆**。Event 的傳遞完全在 Publisher 的控制範圍之外。這讓 Publisher 能立刻繼續處理下一個任務。

#### 差異二：控制反轉（Inversion of Control）

在 Request-Response 模型中：

> **發送方必須預先知道接收方的存在，甚至必須知道如何呼叫它的 API。** 要廣播給多個接收方，就必須對每個接收方分別處理。

在 Event-Driven 模型中：

> **Publisher 完全不需要知道誰在消費它發布的 Event，甚至可能根本不知道有哪些 Consumer 存在。** 這實現了 Producer 與 Consumer 之間的完全解耦合（Decoupling）。

這正是我們在前面章節提到的 Microservices 核心品質之一——**Loose Coupling**。Event-Driven Architecture 與 Microservices 的成功結合，絕非偶然。

### 實作應用：訂閱服務重構

用 Event-Driven 重新設計訂閱流程：

```
[用戶請求] → Subscription Service → [寫入 DB，發布 Event 到 Broker]
                                              ↓
                        Payment Service 消費 Event → [處理付款，發布新 Event]
                                              ↓
                        Recommendation Service 消費 Event → [更新推薦，發布新 Event]
                                              ↓
                        Notification Service 消費 Event → [發送確認 Email / Push Notification]
```

關鍵變化：
- **用戶在第一時間就得到回覆**，不必等待漫長的後端處理
- **即使 Payment Service 因第三方問題延遲很長時間，用戶完全不受影響**
- **每個 Service 獨立運作，API 變更不會連鎖影響其他 Service**

---

## 💡 重點摘要

- **Request-Response 模型的同步特性，讓跨 Service 的長呼叫鏈成為用戶體驗的災難，而且越多的 Service 參與，一致性與延遲的矛盾就越尖銳。**
- **Event 是不可變的（Immutable）事實或狀態變化，可以被多個 Consumer 重複消費，這是與短暫的 Request 最根本的差異。**
- **Event-Driven 的非同步特性，讓 Publisher 發布 Event 後立即解脫，不必等待任何回覆，極大改善了用戶體驗。**
- **Event-Driven 的控制反轉，讓 Publisher 與 Consumer 完全解耦合——這正是 Loose Coupling 這個 Microservices 核心原則的最佳實現方式。**
- **Message Broker 是 Event-Driven Architecture 的核心樞紐，提供 Redundancy、路由與緩衝能力。**
