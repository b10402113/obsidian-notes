# 事件驅動架構的適用場景與傳遞模式

## 📝 課程概述

本單元探討 Event-Driven Architecture 的最佳適用場景，並明確指出何時應該繼續使用傳統的 Request-Response 模型。我們將學習兩種核心的 Event 傳遞模式——**Event Streaming** 與 **Pub/Sub**——它們各自的工作原理與選擇時機，為架構設計決策提供明確的判斷框架。

---

## 核心觀念與實作解析

### Event-Driven Architecture 的適用場景

Event-Driven 不是萬能解。接下來列出六個最典型、最適合使用 Event-Driven 的場景，以及背後的原因。

#### 場景一：Fire and Forget（觸發即忘）

當動作的發送方**不需要立即收到回覆**時，這是最自然的選擇。

- **生成報告**：用戶請求一個可能需要幾分鐘甚至幾小時才能完成的報告，結果完成後主動發送 Email 通知即可。用戶不需要在線等待。
- **發布評論**：用戶發布評論後，不需要任何立即回傳的資料，只要系統確認評論已寫入即可。

> 這類場景的共同點：**發送方不需要等待結果，Event 是最好的觸發媒介。**

#### 場景二：Reliable Delivery（可靠傳遞）

在**金融交易**等不允許任何資料丟失的場景中，Event 的持久化特性提供了可靠的傳遞保障。

- **電子商務下單**：告訴用戶「您已下單成功」後，絕對不能出現訂單在傳輸過程中被遺失的情況，否則用戶未收到貨，卻也未被退款——這是災難性的使用者體驗。
- **跨銀行轉帳**：每一步資金流動都需要有完整的 Audit Trail，Event 的 Log 提供了天然的不可變記錄。

> 這類場景需要「至少一次」（At Least Once）或「恰好一次」（Exactly Once）的 Delivery Semantics，我們在下一堂課會深入討論。

#### 場景三：Infinite Streams of Data（無限資料流）

當系統需要處理**持續不斷、無法預估數量**的資料流入時，Event 是最自然的抽象。

- **IoT 感測器資料**：自動駕駛車、智慧吸塵器等設備，無時無刻都在發送位置或狀態資料。
- **即時日誌與指標**：大量雲端伺服器產生的 Logs 或 Metrics，需要即時分析、彙總或轉換。

> 這類場景的 Event 是「時間序列資料」，需要 Event Streaming Pattern 來支撐。

#### 場景四：Anomaly Detection / Pattern Recognition（異常偵測與模式識別）

當**單一資料點本身意義不大，但串聯起來能揭示重要洞察**時，Event Stream 提供了完美的分析基礎。

- **請求頻率監控**：每台伺服器的每秒請求數（Requests per Second）是單一資料點，但如果 RP 突然飆高，代表即將遇到效能瓶頸，需要擴展；如果突然降為零，代表某台伺服器發生了硬體或軟體故障，需要立即處理。
- **信用卡異常消費偵測**：單一筆消費記錄不構成警示，但短時間內大量不同地點的交易就會觸發 fraud alert。

#### 場景五：State Change Broadcasting（狀態變化廣播）

當一個 Service 的狀態發生變化，而且**其他多個 Service 可能對此感興趣**時，Event Broadcasting 是最優雅的解法。

- **用戶點擊數位廣告**：Ad Service 收到請求後發布一個 Event，Message Broker 將其广播给所有感興趣的 Service（計費系統、分析系統、反欺詐系統等），每個 Service 各司其職，但 Ad Service 完全不需要知道它們的存在。

> 這正是 Event-Driven 與 Microservices 完美互補的最佳例證：**Producer 與 Consumer 完全解耦合。**

#### 場景六：Buffering / Traffic Spike Tolerance（緩衝與流量尖峰吸收）

當系統突然面臨**不可預測的流量洪峰**時，Message Broker 可以作為 Buffer，保護下游 Service 不被打垮。

- **社群媒體熱門事件**：突發的全球事件引發大量貼文或留言湧入，Message Broker 緩衝這些 Event，以各個下游 Service 能承受的速度逐一傳遞，確保系統整體不被流量擊潰。

### 何時應該用 Request-Response？

並不是所有場景都適合 Event-Driven。以下兩種情況，傳統模型更合適：

- **需要立即回傳資料**：例如用戶請求一個產品分類頁面，必須**同步**取得產品資料並立即呈現，否則用戶會直接離開。這種場景用 Event 只會徒增複雜度與延遲。
- **場景過於簡單**：架設與維護 Message Broker 有其**複雜度與成本**。如果一個簡單的雙向呼叫就能解決問題，硬上 Event-Driven 只是自找麻煩。

> **最佳實踐**：一開始先用簡單的 Request-Response，僅在確有必要的關鍵路徑上，才升級為 Event-Driven。這種漸進式的架構演進，比一開始就過度設計要健康得多。

### 兩種 Event 傳遞模式

#### Event Streaming Pattern

**核心特性**：Message Broker 作為 Event 的**永久或半永久儲存**，類似日誌（Log）。

- Consumer 可以訪問**完整的 Event Log**，即使某些 Event 已被消費過
- **新加入的 Consumer** 也能從頭或從任意時間點Replay 歷史 Event
- Message Broker 會保留 Event 一段時間（可能是永久），用於災難恢復與 Audit

**適用場景**：
- Reliable Delivery（需要完整的歷史記錄來進行復原與稽核）
- Pattern / Anomaly Detection（Consumer 需要訪問整個時間窗口內的所有歷史 Event）

#### Publisher/Subscriber（Pub/Sub）Pattern

**核心特性**：Consumer 訂閱某個 Topic/Queue/Channel，**僅接收訂閱之後的新 Event**。

- Consumer 無法取得歷史 Event
- 所有 Current Subscribers 收到 Event 後，Message Broker 通常會將其從 Queue 中刪除
- 新訂閱者只會收到未來的新 Event，不會收到過去的

**適用場景**：
- Fire and Forget、Broadcasting、Buffering 類型的場景
- 當 Message Broker 僅作為**臨時儲存或廣播機制**，Event 被消費後就轉入資料庫永久保存

### 模式選擇的直覺

| 問題類型 | 適合的模式 |
|---------|-----------|
| 需要 replay 歷史資料？ | Event Streaming |
| 只需要廣播新事件？ | Pub/Sub |
| 需要災難恢復/Audit Trail？ | Event Streaming |
| Event 消費後直接進資料庫，不再需要？ | Pub/Sub |

---

## 💡 重點摘要

- **Fire and Forget、Reliable Delivery、Infinite Streams、Anomaly Detection、State Broadcasting、Traffic Buffering 是 Event-Driven 的六個典型適用場景。**
- **需要立即回傳資料或場景過於簡單時，堅持用 Request-Response 即可，沒有必要為了用而用。**
- **典型的 Microservices 系統會結合兩種模型：非關鍵路徑用 Request-Response，關鍵路徑逐步升級為 Event-Driven。**
- **Event Streaming Pattern 適合需要歷史Replay 與 Audit 的場景；Pub/Sub Pattern 適合作為廣播與緩衝機制的場景。**
- **Message Broker 的維護有其成本代價，漸進式架構演進比一開始過度設計更為健康。**
