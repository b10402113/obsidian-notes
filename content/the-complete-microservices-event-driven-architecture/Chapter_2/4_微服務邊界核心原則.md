# 微服務邊界核心原則

## 📝 課程概述

本單元是 Chapter 2 的第一堂課。當我們決定要將 Monolith 遷移到 Microservice 之際，**「該怎麼拆？」**才是最核心的問題。這堂課從三個真實失敗案例出發，逐步推導出設定微服務邊界時必須遵守的三個核心原則：Cohesion（內聚性）、Single Responsibility Principle（單一責任原則）與 Loose Coupling（鬆耦合）。

---

## 核心觀念與實作解析

### 案例一：按技術層級拆分（按 Tier 拆分）—— 為什麼行不通？

一開始，我們可能直覺地沿用程式碼中已經存在的分層邏輯來拆分：

- **Presentation Layer** → Frontend Service（處理使用者請求、安全驗證、HTML/JS/CSS）
- **Business Logic Layer** → Business Service（處理結帳、折扣、促銷、產品評論）
- **Data Layer** → Data Service

這種做法看似合理，因為幾乎不需要重構。但實際上**這個方向會讓我們一事無成**。

原因在於：我們開發的幾乎每個新功能，都會同時涉及 API 層、業務邏輯層與資料層的變動。這意味著三個團隊必須同時規劃、協調、一起上線——**組織的擴展性（organizational scalability）完全沒有提升**。

> 我們所做的，只不過是把「三層架構」變成了「五層架構」，而這仍然不算是 Microservice。

---

### 原則一：Cohesion（內聚性）

從第一個失敗案例中，我們學到了第一個核心原則：**每個 Microservice 必須具備內聚性（Cohesion）**。

> **Cohesion 的定義：「會一起變動的元素應該待在同一個地方。」**

如果所有邏輯都跟著同樣的需求一起變動，就應該被放在同一個 Microservice 的邊界內。如此一來，每個團隊才能真正獨立運作，不需要與其他團隊反覆協調才能推進任何一個功能。

在我們的案例中，每個 Microservice 的內聚性不足，是遷移失敗的根源。

---

### 案例二：按技術邊界拆分——另一個常見陷阱

工程團隊提出：用不同語言與技術棧重寫不同模組，來追求更好的效能：

- Store Frontend 部分 → Node.js + JavaScript（程式碼量減少 50%）
- 另一部分 → Java（效能提升 20%）
- 推薦引擎部分 → C++ 函式庫
- 推薦引擎另一部分 → Python（引入 ML 函式庫）
- 與支付服務溝通的模組 → Golang

拆分後效能確實提升了，但邊界純粹是技術導向的。這帶來了新的問題：

- **Product Manager** 不知道一個功能 request 該發給哪個團隊
- **Support Engineer** 收到 bug 回報時，不知道該轉給哪個團隊
- 兩個 Store Front Service 的職責邊界模糊，**前端開發者也不確定改一個功能該改哪個程式碼庫**
- API 的術語極度混亂——把 User、Product、Bank Account 等不同語義全部混在同一個 API 裡

---

### 原則二：Single Responsibility Principle（單一責任原則）

從第二個失敗案例中，我們學到了第二個核心原則：

> **每個 Microservice 只做一件事，並把這件事做到極致。**

這條原則帶來了幾個重要的好處：

- **職責邊界清晰**：沒有人需要猜測新功能應該屬於哪個 Service
- **API 設計更純粹**：每個 API 只綁定於一個特定的上下文（context），術語不再混淆
  - 跟 User Service 溝通時，`id` 就是 User ID，`name` 就是 User Name
  - 跟 Product Service 溝通時，同樣的術語在這個上下文中意思完全不同，**不會產生任何歧義**

---

### 案例三：把 Microservice 拆得太小——Nano Service 陷阱

受「Micro」這個字眼誤導，團隊把每個 Package、Module 甚至每個 Class 都獨立成一個 Microservice。

結果在生產環境中遇到了更嚴重的問題：

- 每一個使用者請求，都會觸發一連串跨服務的通訊
- 每個 Service 又需要呼叫更多 Service 才能完成工作
- **效能大幅下降，問題排查幾乎不可能**

根本原因是：**這些 Microservice 彼此之間高度耦合（tightly coupled）**，每一個操作都需要大量跨服務的溝通。

---

### 原則三：Loose Coupling（鬆耦合）

從第三個失敗案例中，我們學到了第三個核心原則：

> **Microservice 應該是鬆耦合的：每個 Service 可以獨立完成工作，盡量減少對其他 Service 的依賴。**

此外，這個案例還澄清了一個常見的迷思：

> **「Micro」不代表 Service 要越小越好。** Microservice 中的「Micro」是有點誤導的用詞。只要服務具備內聚性、遵循 SRP 且相互鬆耦合，**大小根本不重要**。每個 Service 的大小自然會有差異，這完全正常。

---

### 三個原則的實質意義

| 原則 | 核心含義 | 目標 |
|---|---|---|
| **Cohesion** | 一起變動的元素待在一起 | 團隊能真正獨立運作 |
| **SRP** | 每個 Service 只做一件事 | 消除職責歧義、API 術語清晰 |
| **Loose Coupling** | Service 間依賴最小化 | 效能可控、故障隔離 |

這三個原則是**成功微服務架構的必備前提**。如果不遵守，遲早會遇到問題。但這三個原則仍然沒有告訴我們**具體怎麼拆分**——這部分會在下一堂課中介紹。

---

## 💡 重點摘要

- **按技術層級拆分 Monolith 只是在建立更多 Tier，並不會提升組織擴展性。**
- **Cohesion 是第一原則：「一起變動的邏輯就該待在同一個 Service。」**
- **Single Responsibility Principle 使職責邊界清晰，消除跨團隊的溝通歧義。**
- **Loose Coupling 意味著每個 Service 可以獨立完成工作，減少跨 Service 依賴。**
- **Microservice 的「Micro」不等於「越小越好」——服務大小不是目標，內聚性與鬆耦合才是。**
