# 遷移到微服務的步驟、技巧與模式

## 📝 課程概述

本單元是 Chapter 2 的最後一堂課，專注於回答「**具體該怎麼遷移**」這個實務問題。我們將從失敗率最高的 Big Bang 策略談起，引出正確的增量式漸進策略；接著說明遷移前的準備工作，最後介紹在遷移執行階段最關鍵的**Strangler Fig Pattern**（絞殺者無花果模式），以及一個確保遷移順利的核心建議：技術棧盡量保持不變。

---

## 核心觀念與實作解析

### 錯誤策略：Big Bang（一次到位）

Big Bang 策略的思路很直覺：

1. 一次规划好所有 Microservice 的邊界
2. 說服管理層暫停所有新功能開發
3. 全員專注在遷移上

表面上看起來「專心」能加速完成，但實際上是**生產力與商業影響最差**的策略。原因有三：

1. **太多廚師一起做同一道菜**：大型重構專案，參與的人越多，摩擦越大——而我們選擇 Microservice 的主要原因之一，正是團隊遇到了組織擴展性問題
2. **估算成本極高**：大型專案的時程估算本來就不準確，途中一定會遇到預料之外的技術難關。承諾四個月完成卻拖到五、六個月，會讓管理層失去信心
3. **業務會因此停滯**：停滯新功能開發幾個月，Product Manager 無法產出成果可能選擇離開，Sales 無新功能可推銷，使用者甚至會誤以為公司出了問題

---

### 正確策略：Incremental & Continuous（增量漸進式）

這才是業界推薦的做法：**不追求一次到位，而是持續找出最佳候選者，逐步遷移**。

#### 誰是最好的遷移候選者？

按效益高低排列：

| 優先順序 | 候選者類型 | 原因 |
|---|---|---|
| 1 | **開發頻率最高的區域** | 最頻繁變動的邏輯是最大衝突來源，分離它們才能讓團隊真正獨立 |
| 2 | **擴展性需求最高的元件** | 作為 Monolith 的一部分，這些元件的擴展性受限最嚴重 |
| 3 | **技術債最少、邏輯邊界清晰的元件** | 複雜度低、測試覆蓋好的元件，遷移風險最低 |

選擇「開發頻率最高的區域」優先遷移，是因為：**沒有人在動的程式碼根本不是問題的根源**。我們要解決的是阻礙團隊前進的那些高變動區塊。

#### 漸進式策略的關鍵優點

- **不需要硬性 deadline**：即使需要一年，也不會有時間壓力
- **持續產出可見的進步**：每次成功遷移一個元件，都是實質的價值交付
- **單次超時以天/週計算**：而不是月或年——不會對業務造成致命影響

---

### 遷移前的準備工作

在選定要遷移的元件後，正式開始遷移前有三個必備步驟：

1. **確保有良好的測試覆蓋率（Test Coverage）**
   - 這是最關鍵的一步
   - 有了充分的測試覆蓋，我們才能有信心地重構而不破壞功能
   - 如果測試不足，遷移過程中任何 bug 都可能讓整個專案延誤

2. **為要遷移的元件定義清楚且經過深思熟慮的 API**
   - 這是 Service 與外部世界的契約，一旦定義後應盡量穩定
   - API 設計的好壞直接決定了 Service 間的耦合程度

3. **將元件與應用程式的其餘部分隔離（Remove Interdependencies）**
   - 消除任何不必要的跨模組依賴
   - 這是 Cohesion 原則的實際應用

### Strangler Fig Pattern（絞殺者無花果模式）

Martin Fowler 提出的這個模式，名稱源自一種植物：它在一棵老樹上萌芽，沿著樹幹攀爬生長，最終完全覆蓋並取代老樹。

#### 執行步驟

```
1. 在 Monolith 前引入一個 Proxy（通常是一個 API Gateway）
   → 稱為 Strangler Facade，一開始只是將請求原封不動地傳遞給 Monolith

2. 開發並部署新的 Microservice
   → 這個 Service 實作了原本某個元件的功能

3. 將該 API 的流量從 Monolith 逐步導向新的 Microservice
   → 這個切換對使用者完全透明

4. 觀察新 Microservice 的效能與功能穩定性
   → 確認一切正常後

5. 移除 Monolith 中相同功能的舊程式碼
   → 完成該元件的遷移

6. 重複以上步驟，直到 Monolith 完全消失或只留作支援舊客戶的 Legacy System
```

#### 這個模式的核心價值

> **「漸進式替換，而非一次顛覆。」**

Strangler Fig Pattern 讓整個遷移過程中，**使用者體驗完全不受影響**，同時大幅降低了單次變更的風險。每一次只專注遷移一個功能，失敗的範圍可控。

---

### 最重要的 Tips：保持技術棧不變

遷移到新架構時，工程師往往會興奮地想要順便引入新語言、新框架。但這裡有一個硬道理：

> **每一次額外的改變，都是一個新的潛在 bug 來源。遷移本身已經夠複雜了——不要再增加風險因素。**

所以在遷移期間：
- **程式碼語言盡量保持原樣**
- **Framework 盡量保持原樣**
- **只有在遷移完成後、新 Service 穩定了，才考慮重構到新技術棧**

---

## 💡 重點摘要

- **Big Bang 策略表面高效，實際上是失敗率最高的遷移方式——它讓組織在最需要新功能的時刻完全停擺。**
- **Incremental 策略的核心是持續識別並遷移「最有價值的候選者」：高變動頻率、高擴展需求、低技術債。**
- **遷移前的三大準備：充分測試覆蓋率 → 定義清楚 API → 隔離元件依賴。**
- **Strangler Fig Pattern 是業界標準的遷移執行模式：Proxy → 新 Service → 流量切換 → 移除舊程式碼，反覆執行。**
- **遷移期間堅持「技術棧不變」的紀律，能有效控制風險——新技術留到遷移完成後再引進。**
