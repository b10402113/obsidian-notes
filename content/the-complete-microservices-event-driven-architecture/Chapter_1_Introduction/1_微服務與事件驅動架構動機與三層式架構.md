# 微服務與事件驅動架構動機與三層式架構

## 📝 課程概述

本單元是整個課程的起點。我們先建立對 Microservices 與 Event-Driven Architecture 的學習動機，接著從一個具體的商業場景出发，逐步推導出「為什麼需要這些現代架構樣式」。透過理解傳統三層式架構（Three-Tier Architecture）的優勢與極限，為後續深入探討 Microservices 打下扎實的基礎。

---

## 核心觀念與實作解析

### 微服務架構的產業背景

Microservices Architecture 是目前業界最流行、最現代的系統架構樣式。它的核心精神是：**將整個系統拆分成多個各自獨立運作的 Service，每個 Service 有明確的職責範圍，並由專屬的團隊負責維護。**

為什麼這麼多頂尖科技公司都在用它？當 Microservices 實作得當，組織能夠容納數千甚至數萬名工程師同時協作，而且每個小團隊幾乎可以**獨立運作**。這使得大型組織能構建服務數十億使用者的系統，同時保持營運成本低廉、創新速度高。

> 這也是 System Design 面試中 Microservices 相關概念頻繁出現的原因——它已經成為大型系統設計的標配語言。

### 千萬別把 Microservices 當銀彈

聽到這裡，你可能會迫不及待想把手邊的 codebase 重構成微服務。但老師特別提醒：**Microservices 並不是銀彈（silver bullet）**。

業界有許多組織在遷移到 Microservices 後反而陷入泥沼，最後甚至退回傳統架構。問題在於：

- 如果公司還沒準備好（例如團隊規模不足、缺乏 DevOps 文化），Microservices 會帶來大量**不必要的複雜度與 overhead**
- 如果實作方式不當，原本簡單的問題會變得更難處理

因此本課程的核心目標不只是教會你「怎麼用」Microservices，更重要的是讓你理解**什麼時機適合用**、**如何正確地實作**，以及**如何避開常見的陷阱與反模式（anti-patterns）**。

### 三層式架構（Three-Tier Architecture）詳解

在談論 Microservices 之前，我們先回顧目前最主流的架構方式——**三層式架構**，它通常被稱為 **Monolithic Architecture（單體式架構）**。

整個系統分為三個邏輯與物理層次：

| 層次 | 名稱 | 職責 |
|------|------|------|
| 第一層 | Presentation Tier | 前端程式碼，運行在使用者的瀏覽器、手機或平板上 |
| 第二層 | Logic Tier（Application / Business Tier） | 承載所有業務邏輯，處理使用者與系統之間的所有互動 |
| 第三層 | Data Tier | 持久化儲存，包含資料庫與檔案系統 |

Monolithic Architecture 的特別之處在於：**所有業務邏輯都集中在 Logic Tier 的同一個 Codebase 中，並以單一 Process 的形式部署與運行。**

### Monolith 的適用場景

Monolith 並不是「過時的壞東西」。它之所以到現在仍是使用最廣泛的架構，是因為它自帶許多優點：

- **設計簡單**：三層式架構幾乎適用於任何 Web 系統，無論是新聞網站、股票交易系統、約會平台還是網路銀行，架構設計幾乎可以一模一樣。
- **適合小團隊快速啟動**：用少數 Full-Stack Developer 搭配主流 Web Framework 與標準資料庫技術，就能快速建構出完整系統。

Jeff Bezos 說得好：「團隊的理想規模是兩份披薩就能餵飽的大小。」如果你的公司符合這個模型，**完全沒有必要使用更複雜的架構**。

### Monolith 在成長過程中的崩解

然而，隨著公司越來越成功、團隊不斷擴大，Monolith 會開始暴露出兩大類問題：

#### 組織可擴展性（Organizational Scalability）問題

當越來越多工程師同時工作在**同一個 Codebase** 上時：

- **Code Merge Conflict 急劇增加**：每個人都在踩彼此的腳，連最簡單的功能都需要大量協調。
- **會議成本上升**：需要協調的人越多，會議越長、生產力越低。
- **Codebase 膨脹**：Feature 越多，Codebase 越大、越複雜。IDE 載入變慢、建置變慢、測試變慢、部署風險變高。
- **發布頻率下降**：每次 Release 包含更多 Feature，BUG 與系統故障的機率隨之上升。
- **新進工程師上手時間拉長**：龐大的 Codebase 需要更長的熟悉期。

> 最終，每多增加一位工程師，生產力的**邊際效益遞減**，直到某個臨界點後，**增加人手反而讓所有人的生產力下降**。

#### 技術可擴展性（Technical Scalability）問題

除了組織問題，Monolith 在技術層面也開始力不從心：

- **昂貴的硬體需求**：每個 Application Instance 必須承載整個業務邏輯，需要更強大、更昂貴的伺服器，無法使用便宜的 Commodity Hardware。
- **技術鎖定（Technology Lock-in）**：當初選定的技術棧可能已過時多年，但整個系統綁在一起，要更換 Library 已經很困難，更別說考慮新的程式語言或 Framework——那幾乎是個不可能的任務。
- **穩定性下降**：任何記憶體泄漏、效能瓶頸或 BUG 都可能影響**整個系統**，通常只能靠 Rollback 補救。

### Monolith 的極限：模組化只是緩解，不是解決

有些人可能會想：「那我把 Monolith 內部做好分層、模組化，甚至拆成 Library 不就好了？」

老師的答案是：**這只能緩解症狀，無法根本解決問題。**

原因是：
- 不同 Module 之間仍然是**緊密耦合（tightly coupled）**的
- 仍然受限於相同的技術棧與程式語言
- 仍然需要以**單一 Runtime Unit** 的形式部署

這些限制加在一起，就是我們尋找替代方案的起點。

## 💡 重點摘要

- **Microservices 是業界最流行的架構樣式，讓組織能容納數千至數萬名工程師同時獨立運作，是大型科技公司成功的重要支柱。**
- **Microservices 不是銀彈——在準備不足或實作不當的情況下，會引入大量不必要的複雜度，而非解決問題。**
- **Monolith 的三層式架構對小型 Startup 與小團隊仍是最佳選擇，直到組織規模與 Codebase 複雜度突破某個臨界點。**
- **Monolith 的主要瓶頸有兩類：組織可擴展性（Codebase 衝突、協調成本、發布風險）與技術可擴展性（硬體成本、技術鎖定、穩定性下降）。**
- **Event-Driven Architecture 能與 Microservices 完美搭配，透過非同步事件溝通實現更高度的解耦合與可擴展性。**
