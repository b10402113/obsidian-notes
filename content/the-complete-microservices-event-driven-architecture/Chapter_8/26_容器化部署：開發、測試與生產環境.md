# 容器化部署：開發、測試與生產環境

## 📝 課程概述

本單元介紹 **Container（容器）**作為 Microservice 包裝與交付的標準工具。我們將從「開發與生產環境不一致（Parity Problem）」這個根本問題出發，比較 VM 隔離方案的高額損耗，再介紹 Container 如何以「只隔離需要隔離的部分，其餘全部共享」的優雅方式解決這個問題，並探討 Container 在多雲環境下帶來的 Portability（可移植性）優勢，以及部署到生產環境時所面臨的新挑戰。

---

## 核心觀念與實作解析

### 問題的根源：開發與生產環境不一致

開發者在本機設定 Microservice 環境時，經常遇到這樣的落差：

- **本地端**：Standalone In-memory Database、不同版本的作業系統、不同位置的設定檔、不同版本的 Dependencies
- **生產端**：Distributed Database、不同 OS、不同設定檔路徑、不同 Dependencies 版本

結果是：**在本機測試一切正常，部署到生產後就出問題**。

---

### VM 方案及其高額代價

要消除 Parity 問題，最直覺的做法是：在開發者的機器上運行一個與生產環境相同的 VM。

但 VM 方案帶來了巨大的額外損耗：

```
[Host OS] → [Hypervisor] → [Guest OS Kernel] → [Guest OS 應用程式] → [檔案系統、網路、記憶體、安全]
                                       ↑
                               每個 VM 都要運行完整 OS
```

- **Hypervisor** 本身消耗 CPU 與記憶體
- 每個 Guest OS 都有自己的 **Kernel**
- **一個 Microservice Instance 就需要一個完整 VM**，多個 Microservice 就需要多個 VM
- 每個 VM 的啟動時間以**分鐘**計算

> 這讓本地開發與整合測試變得極慢且極沒效率。

---

### Container 的解決思路：「只隔離需要隔離的部分」

Container 的核心哲學與 VM 截然不同：

| VM | Container |
|---|---|
| 每個 VM 有自己的完整 OS 與 Kernel | 共享 Host OS 的 Kernel |
| 完整虛擬化硬體 | 只虛擬化必要的隔離資源 |
| 啟動時間：分鐘級 | 啟動時間：毫秒級 |
| 資源損耗高 | 資源損耗極低 |

Container Image 包含：
- Microservice Binary
- 啟動命令
- 所有必要的 Dependencies

每個 Container Instance 擁有：
- **獨立的檔案系統**
- **獨立的網路介面**
- **獨立的 Runtime**

但共享：OS Kernel、Device Drivers、以及其他不需要隔離的組件。

> 在同一台開發機器上運行數十個 Container，幾乎沒有任何額外負擔——非常適合本地開發與整合測試。

---

### Container 在 CI/CD Pipeline 中的優勢

Container Image **與 OS 及硬體完全解耦**：
- 在 macOS 上 Build 的 Image，可以在 Linux CI Server 上運行
- 可以在任何支援 Container Runtime 的 OS 與硬體上運行

> **一次 Build，到處 Run（Build Once, Run Anywhere）。**

這使得 CI/CD Pipeline 可以完全更換 OS 與硬體，而不需要重新調整 Microservice 的部署設定。

---

### Container 為生產環境帶來的價值

#### 消除 Cloud Vendor Lock-in

VM Image 格式通常是**雲端供應商專屬的**：
- 若要從 AWS 遷移到 GCP，需要重新建立所有 VM Image
- 在 Multi-cloud（跨雲）或 Hybrid-cloud（混合雲）環境下，VM 的管理複雜度急劇上升

Container 的 Portability 徹底解決這個問題：
- 只需要在目標環境安裝 **Container Runtime**
- 同一個 Image 可以直接部署到任何雲端供應商或私有資料中心

#### 更好的硬體利用率

```
[VM 方案]
多台小型 VM → 每台 VM 的 OS 各自損耗 CPU 與記憶體

[Container 方案]
一台大型 VM 或 Dedicated Host → 多個 Container 共用同一 OS Kernel
→ 可運行更多 Microservice Instance
→ 基礎設施成本大幅下降
```

Container 的啟動時間是**毫秒級**（相對於 VM 的分鐘級），讓 Auto-scaling 的響應速度大幅提升。

---

### Container 部署到生產環境時的新挑戰

將 Container 應用到生產環境時，架構會變得複雜——因為現在有**兩層抽象**需要整合：

**第一層：雲端基礎設施抽象**
- Cloud VM 或 Dedicated Host（需要租用與 Auto-scaling）
- Cloud Managed Services（Database、Message Broker、Distributed Logging）

**第二層：Container 抽象**
- 數十個 Container Image，各自代表不同的 Microservice
- 每個 Microservice 需要一組 Container Instance 部署到基礎設施上

**需要手動解決的問題：**
1. **Service Discovery**：如何讓各 Microservice Container 互相發現並連接？
2. **網路連線**：如何管理 Container 間的網路路由？
3. **可用性與擴展**：如何動態增減 Container 數量？如何替換崩潰的 Container？
4. **部署新版本**：當 Microservice 發布新版本時，如何滾動更新所有 Container？

> 手動管理數百甚至數千個跨雲端區域或跨雲端供應商的 Container Instance，**幾乎是不可能的任務**。

這個問題的解決方案，就是下一堂課要介紹的 **Container Orchestration**。

---

## 💡 重點摘要

- **Container 解決了開發與生產環境不一致的 Parity Problem，同時大幅降低了 VM 的資源損耗與啟動時間。**
- **Container 的核心優勢：共享 OS Kernel——只隔離需要隔離的資源，其餘全部共享，實現「毫秒級啟動」與「極低資源損耗」。**
- **Container Image 與 OS/硬體完全解耦，帶來了真正的 Multi-cloud Portability，消除 Cloud Vendor Lock-in。**
- **在生產環境中，多個 Container 共享同一 OS Kernel 可大幅提升硬體利用率，降低基礎設施成本。**
- **Container 部署到生產環境後，兩層抽象（基礎設施 + Container）的整合帶來了 Service Discovery、網路路由、Auto-scaling 與滾動更新等新挑戰，需要 Container Orchestration 工具來統一管理。**
