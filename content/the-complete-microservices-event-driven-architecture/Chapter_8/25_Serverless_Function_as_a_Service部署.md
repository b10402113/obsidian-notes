# Serverless：Function as a Service 部署

## 📝 課程概述

本單元介紹 **Function as a Service（FaaS）**——一種比傳統 VM 部署更加事件驅動的雲端部署模式。我們將從兩個真實場景（票務系統、廣告報表系統）出發，理解「閒置時間佔比極高」的 Microservice 為何在 VM 上部署是浪費資源；接著介紹 FaaS 的運作原理、計費模型，以及它的適用邊界與主要限制。

---

## 核心觀念與實作解析

### 兩個揭示問題的實際場景

#### 場景一：現場活動票務 Microservice

- 這支 Microservice 只處理**偶發性的現場活動售票**（每月最多一次）
- 開放售票後的 30 分鐘至 1 小時內，會湧入 99% 的流量
- 其餘時間**幾乎零流量**，但 VM 租金 24 小時都在計算

> **問題：為了偶爾的高峰流量，我們必須 24/7 租用 VM 並維護 Load Balancer 與 Auto-scaling Policy，成本效益極低。**

#### 場景二：廣告成效報表 Microservice

- 每月或每季由管理員或 Cron Job 觸發一次報表生成
- 即便有 1000 個客戶，每月也只有約 1000 次請求
- 同時還要維護一整套 Http Request Handler、Cron Job Script、部署 Pipeline

> **問題：這麼少的流量，為什麼還要維護一整個專屬的 Microservice？**

這兩個場景的共同問題：**流量模式高度間歇性（Seasonal / Burst）**的 Microservice，在傳統 VM 部署下，閒置時間的成本與維護負擔完全不划算。

---

### Function as a Service（FaaS）的運作原理

FaaS 是一種雲端服務，讓你的系統不只在軟體層面是事件驅動的，**在基礎設施層面同樣是事件驅動的**。

使用 FaaS 時，你只需要提供兩樣東西：
1. 這個 Function 要**處理什麼類型的請求或事件**
2. 事件觸發時要**執行什麼邏輯**

> **其餘全部由雲端供應商負責——Build、Package、Deploy 到實體硬體、執行、回應。**

- 請求抵達前：Function **完全不存在**，不佔用任何資源
- 請求抵達時：雲端供應商即時喚醒 Function、執行、回傳結果
- 請求結束後：Function 再次休眠，**不收費**

---

### FaaS 的橫向自動擴展能力

當流量出現季節性高峰時（例如票務系統開賣的那一刻）：

- 雲端供應商**自動處理橫向擴展**
- 同一 Function 的多個 Instance 同時運行以消化流量
- **不需要我們維護 Load Balancer 配置或 Auto-scaling Policy**

> 對工程師來說，Scaling 是完全透明的。

---

### FaaS 的計費模型

FaaS 的費用由兩個因素決定：

- **請求次數**：Function 被調用的次數
- **執行時間 × 所需記憶體**：每次執行的資源消耗

> **最關鍵的特性：沒有事件發生時，費用為零。**

---

### FaaS 的適用場景

**最有價值的場景：**

- 流量**極度季節性或間歇性**的 Microservice（如票務、促銷活動）
- **很少被觸發**的後台任務（如每月/每季報表、資料歸檔）
- 想**節省開發與維護成本**（不需要自己維護部署 Pipeline）

---

### FaaS 的主要代價

#### 代價一：隨流量成長，成本可能反超 VM

隨著業務複雜度與流量增加：
- Function 執行頻率提高 → 每次請求都收費
- 業務邏輯複雜化 → 每次執行時間變長、記憶體需求增加
- **計費可能逐漸超過 Dedicated VM 的固定租金**

> FaaS 在低流量時極度划算，但**不是所有場景的長期最優解**。

#### 代價二：效能不可預測

FaaS 的 Function 每次啟動時，**Cold Start**（冷啟動）需要時間。對延遲敏感的場景，這是不可接受的缺點。

> 高頻交易系統、遊戲伺服器、影像串流——這些場景**不適合 FaaS**。

#### 代價三：安全模型較弱

- Function 運行在**多租戶環境**中
- **原始程式碼完全暴露給雲端供應商**（因為必須上傳程式碼才能執行）

> 對安全要求極高的產業，FaaS 的安全模型可能無法通過合規審查。

---

### FaaS 與其他部署方式的總體比較

| 維度 | Cloud VM | Dedicated Host | FaaS |
|---|---|---|---|
| **定價** | 按時計費 | 按時計費 | 按請求/執行時間收費 |
| **閒置成本** | 有（即使零流量也收費） | 有（最貴） | **零**（無請求時不收費） |
| **效能穩定性** | 中等（Noisy Neighbor） | 最佳 | 最低（多租戶 + Cold Start） |
| **安全性** | 中等 | 最佳 | 較弱 |
| **維護負擔** | 高（需自管部署與 Scaling） | 高 | **低**（雲端供應商處理） |
| **適用場景** | 常態流量服務 | 高效能/合規需求 | 間歇性、低流量任務 |

---

## 💡 重點摘要

- **FaaS 的核心價值：沒有事件時不佔用任何資源，讓高度間歇性工作負擔的 Microservice 成本趨近於零。**
- **FaaS 不只在軟體層面事件驅動——基礎設施層面也是事件驅動的，工程師只提供程式碼與事件類型，其餘由雲端供應商處理。**
- **橫向自動擴展對 FaaS 是透明的，無需維護 Load Balancer 或 Auto-scaling Policy。**
- **FaaS 的缺點：流量與複雜度成長後成本可能反超 VM、效能不可預測（冷啟動延遲）、安全模型較弱。**
- **FaaS 是工具，不是萬能解：適合低流量/季節性任務，但對常態流量、高效能需求或高合規要求的 Microservice，傳統 VM 部署仍是更優選擇。**
