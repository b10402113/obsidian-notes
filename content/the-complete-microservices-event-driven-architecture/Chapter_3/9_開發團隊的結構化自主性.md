# 開發團隊的結構化自主性

## 📝 課程概述

本單元探討微服務遷移過程中一個最常見的迷思：**「每個團隊應該擁有完全的自由，決定自己的技術棧、工具與流程」**。我們將看到這種「無限自由」反而會帶來災難，並學習一個稱為「結構化自主性（Structured Autonomy）」的平衡框架——讓團隊保有自主權，同時為整個組織保留必要的結構與標準。

## 核心觀念與實作解析

### 一個最大的迷思

很多公司在遷移到微服務時，抱持這樣的想法：

> 「微服務最大的好處就是每個團隊可以完全自主地選擇技術棧、資料庫、API 框架與工具。」

這句話聽起來很美好，但現實中，這是**公司遷移到微服務後最常見的陷阱之一**。

---

### 為何「完全自由」反而是災難？

#### 1. 前期成本（Upfront Cost）暴增

在 Monolith 時代，整個組織共同承擔了大量前期投入：

- 統一規劃程式碼結構、Framework 選擇、模組組織方式
- 統一建立測試存放位置與執行方式
- 統一客製化建置工具、CI/CD pipeline
- 統一投資監控與告警工具
- 統一設計資料庫 Schema，使其具備擴展性、容錯性與高可用性

現在想像一下，若每個團隊在遷移到微服務後，都要**為自己的程式碼庫與生態重複這一切**。不只初期如此，維護這些基礎設施是**持續性的工作**——若有專屬 DevOps/SRE 團隊，要維護數百個彼此截然不同的微服務基礎設施，遲早會被淹沒。

#### 2. 學習曲線與協作成本

任何新進工程師加入某個團隊時，必須：
- 閱讀該團隊的文件
- 學習並記憶所有最佳實踐
- 每天工作中遵循這些規範

但若需要跨團隊協作（如修改另一個團隊的程式碼，或借鑒某個團隊如何設置整合測試），**每次跨團隊協作都必須重新學習一套完全不同的技術棧**——這在大組織中完全無法擴展。

#### 3. API 不一致（Non-uniform API）

所有微服務最終需要整合為一個系統來共同達成目標。若每個團隊可以自由定義自己的 API 風格與最佳實踐：

- 前端工程師必須學習每個團隊的 API 命名規範與風格，前端程式碼會變得難以維護。
- 若某個微服務需要與多個不同團隊的微服務溝通，**每次呼叫都必須重新閱讀另一團隊的文件**，確認其 API 是否符合己方預期。

---

### Structured Autonomy（結構化自主性）

正確的答案不是「完全自由」，而是**在自主性與結構之間找到平衡**：

> 每個團隊仍然是自主的，但**只在一定邊界範圍內**自主。

這些邊界分為三個層級（Tiers）：

---

### 第一層級：公司級統一標準（最嚴格）

這些領域**不應由各團隊自行決定**，必須全公司統一：

| 統一領域 | 原因 |
|---------|------|
| **監控與告警基礎設施** | 前期投入高，效益可分攤到全組織 |
| **CI/CD 技術與腳本** | 確保建置流程一致，降低維護成本 |
| **公開 API 與內部 API 設計指南** | 外部客戶不在乎內部有多少微服務；統一標準讓跨團隊整合更容易 |
| **安全與資料合規政策** | 若某個微服務被駭，整個系統都暴露風險；若某微服務違反隱私政策，整個組織須負責 |

---

### 第二層級：有限自由的技術選擇

這些領域允許團隊**在一定範圍內**做選擇：

- **程式語言與資料庫技術**：不同的 runtime 與儲存方案適用於不同場景，應允許差異化——但**可選範圍必須受到限制**，否則會出現每個團隊都選用最新、最流行語言的混亂局面。
- 多數大型科技公司通常只在數百個微服務中使用**不超過 5 種程式語言**。

---

### 第三層級：完全自主

這些領域讓每個團隊**完全掌控**：

- 發布流程、節奏與頻率
- 團隊自訂的本地開發與測試腳本與工具
- 團隊自有文件與新進開發者 onboarding 流程

---

### 影響三層級邊界的因素

這些界線**並非鐵板一塊**，會因以下因素而有所調整：

1. **DevOps / SRE 團隊的影響力**：DevOps/SRE 團隊越強勢，公司越傾向於統一標準（以便這些團隊更輕鬆地管理整個系統）。
2. **工程師的資歷（Seniority）**：資深工程師越多，對自主搭建基礎設施的自由度需求越大。
3. **公司文化**：有些公司採用嚴格的單一語言政策（如只允許 C#、Java 或 Python），好處是雇用工程師後可以幾乎無成本地在不同團隊之間調動。

---

## 💡 重點摘要

- **「每個團隊完全自主選擇技術」是微服務遷移中最常見的陷阱——它會帶來重複基礎設施成本、極高的學習曲線與非一致的 API，反而比 Monolith 更糟。**
- **Structured Autonomy 的核心精神是：團隊在一定邊界內自主，而非無邊界自由。**
- **第一層級（公司級統一）包括監控告警、CI/CD、API 設計指南、安全合規政策——這些領域統一投入的效益可分攤全組織，代價值得。**
- **程式語言與資料庫技術屬於第二層級，應允許差異但必須限制選項，防止技術叢林的產生。**
- **發布流程與頻率屬於第三層級，讓每個團隊完全自主，以支援真正的獨立運作。**
