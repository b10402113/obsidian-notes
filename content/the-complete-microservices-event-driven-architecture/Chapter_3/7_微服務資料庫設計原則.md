# 微服務資料庫設計原則

## 📝 課程概述

本單元探討微服務架構中最核心的資料管理原則之一——**Database per Microservice**（每個微服務獨立資料庫）。我們將從一個保險公司的實際遷移案例出发，理解為何共享資料庫反而會讓團隊重新陷入高度耦合的泥沼，並進一步認識這項原則帶來的代價與取捨。

## 核心觀念與實作解析

### 從 Monolith 到 Microservice：共享資料庫的陷阱

讓我們以一家保險公司為例。假設該公司將傳統三層式架構中的 Monolith 應用程式遷移至微服務，並為每個團隊配置了獨立團隊負責各自服務。按理說這應該大幅提升團隊的自治能力——但現實中，**共享資料庫反而讓各團隊再次被緊密耦合在一起**。

以下場景說明了這個問題：

- **報表服務需要跨服務取資料**：報表服務為了節省效能，希望直接存取 Policy Purchasing、Claims、Customers 三個資料庫，而不走 API 呼叫。這樣做看似合理，卻埋下了耦合的種子。
- **Policy Purchasing 團隊更換資料庫**：該團隊決定將緩慢的舊關聯式資料庫替換為讀取優化的 NoSQL 資料庫。但因為報表服務的程式碼與舊資料庫技術緊密耦合，**兩邊的程式碼必須同時修改、同時部署**。
- **Claims 團隊變更 Schema**：該團隊重新命名欄位、移除閒置欄位、加入新欄位。報表服務依賴舊 Schema，兩邊團隊必須開會協調 Schema 設計，再同步修改、測試、發布。
- **Customer Service 加入細粒度安全策略**：安全政策屬於 Customer Service 的領域，但報表服務需要知道這些實作細節，必須由 Customer Service 團隊完整溝通並同步變更。

> **核心問題**：即使團隊邊界已經拆分，只要共享資料庫，**程式碼庫之間的耦合依然存在，而且這種耦合會擴散到團隊之間的溝通協作**——這正是我們遷移到微服務時最想消除的東西。

---

### Database per Microservice 原則的定義

這個原則的定義很清晰：

> **每個微服務團隊完全擁有自己資料，不將資料直接暴露給任何其他服務——即使這會增加額外的網路延遲。**

具體而言：
- 當某個服務需要另一個服務的資料時，**必須透過該服務的 API 來取得**，不能直接查詢資料庫。
- 每個服務必須在 API 層級**抽象化**實際使用的資料庫技術與 Schema。
- 團隊可以自由更換底層資料庫技術或變更 Schema，只要 API 介面保持向後相容，消費者完全不受影響。
- 若 API 本身也需要變更，可以**同時維護兩個版本的 API**，讓其他團隊有數週甚至數月的時間遷移，**完全不需要跨團隊協調**。

### 代價與取捨

實作這項原則並非沒有成本。老師在這裡特別強調了三大代價：

#### 1. 額外的網路延遲

直接查詢資料庫 vs. 發送網路請求到另一個服務再解析響應，後者必然增加效能開銷。當這個開銷成為瓶頸時，可以允許在本機快取或存放屬於其他微服務的資料，但有兩個關鍵前提：

- **Source of Truth（真相來源）**必須保持單一——只有一個微服務可以對該資料執行寫入操作。
- 其他服務持有的是**讀取快取**，因此**只能保證最終一致性（Eventual Consistency）**，而非嚴格一致性。

#### 2. 無法跨資料庫 JOIN

在 Monolith 中，JOIN 操作可以輕鬆跨多個資料表完成。當資料分散到不同資料庫（且有些根本不是關聯式資料庫）時，JOIN 無法直接執行。要達到相同功能，必須：

1. 從兩個不同資料庫拉取資料
2. 將資料轉換為相同格式
3. 在應用程式層面以程式化的方式執行 JOIN

#### 3. 喪失交易保證（Transaction Guarantees）

回到單一資料庫時，我們可以在同一筆交易中原子性地修改多個資料表的資料。跨多個服務執行分散式交易極度困難，**理論上可行，實務上幾乎從不使用**。

> 不用擔心——課程後續章節會介紹專門用來解決這三個挑戰的設計模式（Event Sourcing、CQRS 等）。

## 💡 重點摘要

- **共享資料庫會讓原本獨立的微服務團隊重新被程式碼庫耦合與溝通協作所束縛，抵消了遷移的初衷。**
- **Database per Microservice 要求每個服務只透過 API 暴露資料，並在 API 層抽象化底層資料庫技術，讓團隊可以自由演進而不影響消費者。**
- **跨服務快取資料時，Source of Truth 只能有一個，其他服務只能讀取，並接受最終一致性。**
- **分散式架構必然面臨 JOIN 複雜度上升與交易邊界模糊的代價，需要後續模式來補救。**
- **資料一致性的取捨是設計決策：無需嚴格一致性的資料（如商品評分）可以放心複製，需要嚴格一致性的資料（如庫存、帳戶餘額）則必須保留在單一 Source of Truth。**
