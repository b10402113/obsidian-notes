# DRY 原則在微服務與共享函式庫中的實踐

## 📝 課程概述

DRY（Don't Repeat Yourself）是軟體工程中最基礎的原則之一——重複的邏輯或資料應集中管理。但這堂課要告訴我們一個重要的反直覺事實：**在微服務架構中，DRY 原則並非處處適用，特別是在共享函式庫（Shared Libraries）的場景下**。我們將理解其中的原因，並學習何時應該共享、何時應該刻意複製。

## 核心觀念與實作解析

### DRY 原則的回顧

先快速確認 DRY 的核心精神：

- **單一模組內**：重複的商業邏輯應抽取為共享的 method、class 或 variable。
- **跨應用程式**：被多個專案使用的複雜邏輯，封裝為 shared library 供其他專案 import——修改時只需改一處，無需四處尋找重複。

這套邏輯在 Monolith 應用程式中運作得很好。然而在微服務架構中，事情變得複雜了。

---

### 共享函式庫在微服務中為何會出問題？

#### 1. 緊密耦合（Tight Coupling）

微服務的核心原則之一是**鬆散耦合**。但當多個微服務依賴同一個 shared library 時，這個函式庫的變更反而成為團隊之間耦合的新源頭：

- 該函式庫有 API 變更 → 必須通知所有使用它的團隊 → 各團隊必須同步修改程式碼、重新測試、重新部署。
- 即使 API 完全沒變，每次 shared library 更新，所有使用它的微服務都必須**重建、重新測試、重新部署**。
- 共享函式庫中的 bug 或安全漏洞可能**一次性影響所有使用它的服務**，完全破壞了微服務作為獨立運行單元的初衷。

#### 2. 依賴地獄（Dependency Hell）

這是一個常見但容易被忽視的問題。假設有以下情境：

- Microservice A 使用了 Library A，而 Library A 內部依賴 Library B。
- Microservice A 的另一部分程式碼也直接使用了 Library B。
- 某天，Library A 發布了一個重要更新，Microservice A 急需使用，但新版 Library A 依賴的是一個**比我們目前使用版本更新的 Library B**。

在某些語言與執行環境中，**無法同時依賴同一個函式庫的兩個版本**。這時就會被迫做一個不必要的程式碼修改，只是為了能升級 Library A——代價昂貴且容易出錯。

即使語言支援同時載入兩個版本，也意味著**重複載入幾乎相同的程式碼**，破壞了 DRY 的初衷，同時增加了 binary 體積與建構測試時間。

---

### 共享函式庫的替代方案

老師提出了幾個針對不同情境的替代方案：

#### 情境一：多個微服務共享複雜商業邏輯

**這往往預示著邊界劃分有問題**。如果遵循「業務能力分解」或「子網域分解」，這種情況應該極少發生。

處理方式：
- **調整邊界**：只由其中一個微服務保有該商業邏輯，另一個不保有。
- **獨立成新服務**：若該商業邏輯足夠複雜且重要，直接拆分為獨立的微服務。

#### 情境二：兩個微服務之間需要共享通訊用的資料模型

這種情況下，**共享程式碼反而是好做法**。原因很關鍵：

> 當兩個微服務需要相互通訊時，我們**希望**兩個團隊、兩個程式碼庫處於 co-dependent 狀態——若其中一個服務的 API 或資料模型變更到無法溝通時，**我們希望在第一時間就讓測試失敗**，而不是讓雙方測試都通過，卻在正式上線時才發現無法通訊。

另一種做法是使用**基於 Schema 的程式碼生成工具**（如 Protocol Buffers、OpenAPI Generator），讓資料模型定義統一存放，建置流程自動生成雙方程式碼——只要生成是建置流程的一部分，這種方式既安全又避免重複。

#### 情境三：工具函式（Utility Methods）等經常變動的程式碼

**直接複製到各微服務中是更好的選擇**，原因：

- 每個微服務可以有針對自身使用場景優化的專屬實作。
- 不會因為某個通用工具的變更而被迫同步更新所有服務。
- 日後若要將某個微服務整體遷移到另一種程式語言，遷移成本更低。

#### 情境四：極端情況——真的不想複製

這時有兩個選項：

1. **Sidecar Pattern（Sidecar 模式）**：將共享功能封裝為獨立程序，但部署在與微服務相同的實體主機上，而非獨立服務。微服務透過標準網路協定與同機部署的 Sidecar 程序溝通，延遲比跨主機呼叫低，但仍比直接嵌入程式碼高。
2. **共享函式庫（作為最後手段）**：當程式碼足夠通用且極度穩定（如日誌、重試、語法解析），鮮少變動時，可以使用 shared library——但前提是這些函式庫**彼此之間盡可能完全獨立**，以避免依賴地獄。

---

### 微服務中的資料複製：一個可接受的必要之惡

表面上看，在多個微服務中複製同一份資料是浪費儲存空間。但有時候這是**提升效能所必需的代價**：

- 某個微服務可以快取另一個微服務的資料，省去網路呼叫的延遲。
- **只有一個微服務**是該資料明確的 Owner 與 Source of Truth——只有它可以執行 GET、WRITE、UPDATE、DELETE，其他持有副本的服務**只能讀取**。

**一致性取捨原則**：
- **可接受最終一致性**：線上商城的商品評分、最近幾筆評論——些微延遲無傷大雅。
- **必須嚴格一致性**：產品的即時庫存、使用者帳戶餘額——這些資料絕對不能複製到其他服務以外的地方。

> **對內部程式碼而言**：每個微服務的內部程式碼中，DRY 原則仍然適用。同一個 microservice 內的程式碼複製是不可接受的，應始終抽取為共用 method、class 或 module。

---

## 💡 重點摘要

- **共享函式庫在微服務中會帶來緊密耦合：當函式庫變更時，所有使用它的服務必須同步修改、測試、部署，並面臨依賴地獄的風險。**
- **若多個微服務共享複雜商業邏輯，這通常是邊界劃分錯誤的警訊，應重新評估或將邏輯獨立為新服務。**
- **微服務之間的通訊用資料模型適合共享（希望測試能及時捕捉斷裂），但工具函式等經常變動的程式碼適合刻意複製。**
- **資料複製時，必須嚴守 Source of Truth 只能有一個的原則，並接受最終一致性而非嚴格一致性。**
- **每個微服務內部仍然應該遵循 DRY 原則——複製只適用於跨服務場景的刻意取捨。**
