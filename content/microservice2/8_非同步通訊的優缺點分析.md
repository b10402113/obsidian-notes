# 非同步通訊的優缺點分析

## 📝 課程概述

本單元深入分析上一課中介紹的第二種非同步通訊方式（Event Bus + Service 自有資料庫）的優點與缺點。我們將看到這種方式在**零依賴性**與**極致效能**上的顯著優勢，同時也誠實面對**資料重複**與**理解成本**兩大缺點。講者特別為「資料重複」這一疑慮做了成本分析，說明在現代雲端儲存環境下這幾乎不構成問題。

---

## 核心觀念與實作解析

### 優點一：Service D 零依賴（Zero Dependencies）

這是最重要的優點。當我們說「零依賴」時，意思是：**Service D 在處理傳入請求時，不需要聯繫任何其他 Service**。

想像一個極端的破壞性測試場景：

> Service A、B、C 因為某種災難性故障全部消失了。此時如果有一個請求打進來，問 Service D「User 1 訂過什麼商品？」

Service D **依然可以完整回答這個問題**，因為它資料庫裡的 Users 和 Products 資料都是事先同步好的。

> 這與同步通訊形成鮮明對比：在同步通訊下，只要任何一個下游 Service 故障，Service D 就跟著失敗。

---

### 優點二：極致查詢效能

由於 Service D 所需的資料全部都在自己的本地資料庫中，**查詢延遲極低且穩定**。它不需要跨網路發送多次 HTTP 請求，不需要等待下游 Service 的回覆，直接本地查詢就能得到答案。

> 在高流量的生產環境中，這種效能優勢會被進一步放大——你可以單獨擴展查詢量大的 Service，而不影響其他 Service。

---

### 缺點一：資料重複（Data Duplication）

在這種架構下，同一份資料會出現在多個地方：
- Service B 的資料庫有完整的商品資訊
- Service D 的資料庫也有商品的基本資訊（ID、Title、Image）

嚴格來說，這並不是 100% 的完整複製——Service D 只儲存了回答特定問題所需的最小資料集合（例如：不需要儲存商品價格、庫存、評論等資訊）。但確實存在重複。

#### 儲存成本到底貴不貴？

講師實際做了計算：以 AWS RDS、Google Cloud SQL 或 Azure MySQL 為例，**1 GB 資料儲存一個月的成本約為 $0.015 ~ $0.0175**。

以一個真實的 Amazon 商品 API 回應為例，一筆商品資料大約是 **1,200 bytes**（含 JSON 結構）。那麼儲存 **1 億筆商品資料**的成本是多少？

```
1 億筆 × 1,200 bytes = 120 GB
120 GB × $0.015/GB/月 = $1.8/月
```

**儲存 1 億筆商品資料，一個月只需要不到兩美元。**

> 結論：**儲存成本在現代雲端環境下幾乎可以忽略不計。以「多佔用了一些儲存空間」來否定這種架構，是一個錯誤的出發點。**

---

### 缺點二：真正需要關注的問題——Stale Data（資料過期）

如果因為系統故障或網路問題，Event Bus 漏掉了一些事件，Service D 的資料庫就會與事實不符——這就是 **Stale Data**。這是一個值得在未來課程中深入探討的課題，它比「多佔空間」要嚴肅得多。

---

### 缺點三：理解成本高（Hard to Understand）

這種事件驅動的架構確實增加了系統的複雜度：

- 開發者需要理解 **emit** 與 **listen** 兩種行為
- 需要追蹤事件在系統中的流向
- 當除錯時，不能只follow 一條請求鏈，而是要理解整個 Event Bus 的事件傳播邏輯

這無疑是額外的認知負擔。講師也坦言這不是一個容易理解的模式。

---

### 課程的選擇

經過權衡，講師明確表示：**在這個課程中，我們將採用這種基於 Event Bus 的非同步通訊方式**。

原因很簡單：

> 在追求**使用者體驗（效能與可用性）**的前提下，這種方式的優勢遠遠超過其缺點。零依賴帶來的韌性（Resilience）與極速查詢，是同步通訊和第一種非同步方式都無法比擬的。

未來的實作中，每個 Service 將遵循以下模式：

1. 接收傳入的請求
2. 在本地執行業務邏輯
3. 若有資料變更，**主動發出事件到 Event Bus**
4. 其他感興趣的 Service 監聽事件，據此更新自己的資料庫

---

## 💡 重點摘要

- **Event Bus + Service 自有資料庫的方式，讓 Service D 真正實現零依賴——即使其他所有 Service 崩潰，Service D 依然能正確回覆查詢。**
- **Service D 的查詢直接讀取本地資料庫，無需呼叫其他 Service，效能極佳且穩定。**
- **資料重複的成本在現代雲端環境下可以忽略不計（1 億筆商品資料每月不到 $2），這不應該是反對這種架構的理由。**
- **真正需要關注的是 Stale Data（資料過期）問題，這源於 Event Bus 漏發事件，而非儲存成本。**
- **這種模式的缺點是理解與維護成本較高，但换来的是系統韌性與效能的顯著提升。**
